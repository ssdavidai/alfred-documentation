## Overview

Alfred supports 19 distinct record types organized into two categories: **Entity Types** (14) for operational records and **Learning Types** (5) for extracted knowledge.

## Entity Types

Operational records representing people, projects, tasks, and other tangible entities.

<AccordionGroup>

<Accordion title="person - People and Contacts" icon="user">
**Directory**: `vault/person/`

**Purpose**: Represents individuals, contacts, team members, or stakeholders.

**Status Values**: `active`, `inactive`

**Required Fields**:
- `type`: `person`
- `created`: Creation date

**Optional Fields**:
- `name`: Person's full name
- `status`: One of `active`, `inactive`
- `org`: Organization (wikilink to org record)
- `email`: Contact email address
- `tags`: Array of tags
- `related`: Related records

**Example**:
```yaml
---
type: person
created: 2026-02-26
name: Alice Smith
status: active
org: "[[org/Acme Corp]]"
email: alice@acme.com
tags: [engineering, backend]
---
```
</Accordion>

## Learning Types

Meta-records that capture latent knowledge extracted from operational records by the Distiller tool.

<AccordionGroup>

<Accordion title="assumption - Implicit Assumptions" icon="thought-bubble">
**Directory**: `vault/assumption/`

**Purpose**: Captures implicit assumptions found in operational records.

**Status Values**: `active`, `challenged`, `invalidated`, `confirmed`

**Required Fields**:
- `type`: `assumption`
- `created`: Creation date

**Optional Fields**:
- `name`: Assumption title
- `status`: One of `active`, `challenged`, `invalidated`, `confirmed`
- `based_on`: Source records (wikilinks)
- `challenged_by`: Records that challenge this assumption (wikilinks)
- `confirmed_by`: Records that confirm this assumption (wikilinks)
- `invalidated_by`: Records that invalidate this assumption (wikilinks)
- `tags`: Array of tags

**Example**:
```yaml
---
type: assumption
created: 2026-02-26
name: Users prefer mobile-first design
status: active
based_on:
  - "[[project/Q1 Launch]]"
  - "[[conversation/Planning Session]]"
---

# Users prefer mobile-first design

Based on project discussions, we're assuming users primarily access the platform via mobile devices, though this hasn't been validated with data.
```
</Accordion>

**Status Values**: `draft`, `final`, `superseded`, `reversed`

**Required Fields**:
- `type`: `decision`

**Optional Fields**:

- `status`: One of `draft`, `final`, `superseded`, `reversed`
- `based_on`: Source records (wikilinks)

<Note>
Learning types are automatically extracted by the Distiller tool. You typically don't create them manually via the API.
</Note>

## Type Naming Conventions

- Type identifiers use lowercase, singular form (e.g., `person`, not `Person` or `people`)
- Directory names match type names (e.g., `person` -> `vault/person/`, `assumption` -> `vault/assumption/`)
- Record filenames use title case with spaces (e.g., `Alice Smith.md`)
- Two types (`conversation`, `input`) use `subject` as their name field instead of `name`

<Tip>
Use the `GET /api/v1/vault/schema` endpoint to retrieve the complete, up-to-date schema programmatically. The schema is the source of truth for all validation rules.
</Tip>
