Your data security is our top priority. Alfred implements defense-in-depth protection across encryption, network isolation, authentication, and access control. Here's how we keep your vault safe.

## Encryption at Rest

All your data is encrypted at rest using LUKS2 (Linux Unified Key Setup), the industry-standard full-disk encryption. Each instance has its own dedicated encrypted volume using AES-256-XTS encryption — the same standard used by government and financial institutions.

API keys and sensitive credentials stored in the SaaS database are encrypted with AES-256-GCM before being written to disk. Encryption keys are unique per instance and never shared. The encryption key material itself is generated using cryptographic randomness and is never exposed in logs or configuration files.

In the event of physical hardware failure or decommissioning, encrypted volumes are securely destroyed by Hetzner. Data cannot be recovered without the encryption keys, which are held only on the encrypted volume itself.

<Note>
When you delete your Alfred instance, all encrypted data is permanently destroyed. There are no unencrypted backups or shadow copies.
</Note>

## Encryption in Transit

All communication to and from your Alfred instance is encrypted end-to-end.

- **User-facing traffic** (your web browser to your instance) uses HTTPS with TLS 1.3 encryption, terminating at Cloudflare's edge network.
- **SaaS-to-instance communication** (dashboard API calls) flows through Tailscale's WireGuard mesh network, providing end-to-end encryption with Curve25519 key exchange.
- **Admin access** (SSH to your instance) uses Ed25519 public-key cryptography — never passwords, always encrypted.
- **Backups** are encrypted with restic before uploading to object storage.

There are no plaintext connections anywhere in the system. Even internal Docker network communication between services is isolated and encrypted at the transport layer.

## Network Isolation

Your Alfred instance runs on a dedicated Hetzner Cloud virtual machine with strict network boundaries.

**No application ports are exposed to the public internet.** Your vault, OpenClaw gateway, and worker services all bind to `localhost` (127.0.0.1) only. Users cannot connect directly to any service — all access goes through:

- **Cloudflare Tunnel** for web traffic (user-facing UI)
- **Tailscale WireGuard mesh** for API access (SaaS dashboard)
- **SSH key authentication** for administrative access

Your instance sits behind multiple firewalls: Hetzner Cloud Firewall at the hypervisor level, UFW (Uncomplicated Firewall) at the OS level, and Tailscale ACLs at the network mesh level. Tenant instances are completely isolated from one another — the network is configured to deny all traffic between different customer instances.

<Tip>
Even if someone discovered your instance's IP address, they cannot reach any services because all ports are firewalled or require authentication. Your data is inaccessible without credentials.
</Tip>

## Authentication

All API access to Alfred is authenticated with cryptographically secure API keys.

**API keys** start with the `alf_` prefix followed by 32 random hexadecimal characters. They are shown exactly once at creation — we don't store them in plain text. Keys are verified using constant-time comparison to prevent timing-based attacks. You can create up to 10 keys per account, and revoke any key instantly.

API keys are encrypted with AES-256-GCM before storage in the database, meaning even if our database were compromised, your keys would remain protected.

**Web authentication** uses industry-standard OAuth or email/password with bcrypt hashing. Sessions are protected with secure HTTP-only cookies.

**Administrative access** uses Ed25519 SSH keys, generated uniquely per instance. No password-based access is permitted.

For detailed information about creating and managing API keys, see the [API Authentication](/authentication) guide.

## Access Control

Each AI worker within Alfred (Curator, Janitor, Distiller) operates under scoped permissions. This ensures that if one component is compromised, the attacker cannot exceed that component's intended capabilities.

- **Curator** can create and edit vault records but cannot delete them.
- **Janitor** can edit and delete records (for cleanup and fixing structural issues) but cannot create new record types.
- **Distiller** can only create learning records (assumptions, decisions, constraints) and cannot edit, delete, or move existing records.
- **All mutations are logged** — every create, edit, move, and delete operation is recorded with a timestamp and scope identifier.

Users of your instance have full read and write access to the vault. Permissions are enforced at the operation level before any command executes. Attempting to exceed your scope results in an error, never a silent failure.

## Infrastructure Security

Alfred instances run on hardened Linux servers with automatic security updates and multiple defensive layers.

**OS hardening** includes:
- SSH key-only authentication (no passwords allowed)
- SSH configuration restricting login attempts, disabling tunneling, and setting idle timeouts
- Fail2ban intrusion prevention (blocks accounts after 3 failed login attempts for 1 hour)
- Unattended security updates enabled (patches install automatically)
- UFW firewall with default-deny incoming policy

**Container hardening** includes:
- Privileged operations disabled (`no-new-privileges`)
- Capability restrictions (`cap_drop: ALL` for most containers)
- Memory and process limits per container to prevent resource exhaustion
- Health checks that automatically restart failed services
- Services running as non-root users where possible

All custom container images are scanned for vulnerabilities and use the latest upstream versions of dependencies. The init script that bootstraps your instance is run once and then deleted — it is never stored or executed again.

## Backups

Automated daily encrypted backups ensure you can recover your data in any scenario.

Backups are taken at 3:00 AM UTC (with randomized jitter) using restic, which encrypts your vault with a unique per-instance encryption key before uploading to Hetzner Object Storage. We retain:
- 7 daily snapshots
- 4 weekly snapshots
- 6 monthly snapshots

If your instance's hardware fails, we can restore from backup to new hardware without you losing any data. If your encrypted volume becomes corrupted, we can restore from backup. Backups are verified on first provision — we confirm a successful backup exists before marking your instance as ready.

Backup restoration is included in your instance provisioning. If you ever need to recover from an older snapshot, contact support.

<Tip>
Your backups are encrypted with the same AES-256 encryption as your live data. We cannot access your backups, and your data is protected even in our object storage.
</Tip>

Alfred is built with security-by-default. We don't offer optional security features — all of these protections are active by default for every instance, every tenant.
