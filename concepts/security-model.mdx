## Overview

Alfred implements defense-in-depth security across four network layers, LUKS2 disk encryption, per-tenant VM isolation, container hardening, scoped authorization, and encrypted secret storage. This document covers the entire security posture of the platform.

<Warning>
All tenant services bind to `127.0.0.1` (localhost). User-facing access is exclusively through **Cloudflare Tunnel** (outbound-only). Admin access is exclusively through the **Tailscale mesh**. No application ports are exposed to the public internet.
</Warning>

---

## Trust Boundaries

The platform has four trust zones with well-defined boundaries:

```
+-- UNTRUSTED -----------------------------------------------+
|  Internet users, Cloudflare edge, public DNS               |
+------------------------------------------------------------+
        | HTTPS (Cloudflare TLS)           | HTTPS (Caddy TLS)
        v                                  v
+-- SEMI-TRUSTED -------------------------------------------+
|  Cloudflare Tunnel                SaaS App                 |
|  ({tenant}.alfred.black)          (app.alfred.black)       |
|  Cloudflare Access (optional)     Wasp auth + sessions     |
+------------------------------------------------------------+
        | HTTP (localhost)                 | HTTPS (Tailscale)
        v                                  v
+-- TRUSTED (per-tenant) -----------------------------------+
|  OpenClaw Gateway (:18789)       Tenant API (:3100)        |
|  Gateway token auth              Bearer token (AES-256)    |
+------------------------------------------------------------+
        | WebSocket (Docker network)       | Filesystem
        v                                  v
+-- DATA (encrypted at rest) -------------------------------+
|  /mnt/encrypted/ (LUKS2-AES-XTS)                          |
|  vault/  openclaw/  alfred/  temporal/                     |
+------------------------------------------------------------+
```

### Data Flow: User Request

1. User visits `https://acme.alfred.black` (Cloudflare edge terminates TLS)
2. Cloudflare routes through an outbound-only tunnel to the tenant VM
3. `cloudflared` daemon forwards to `localhost:18789` (OpenClaw)
4. OpenClaw serves the Control UI (gateway token auth)

### Data Flow: SaaS Dashboard

1. User authenticates at `app.alfred.black` (Wasp auth: email/password or OAuth)
2. Dashboard requests proxy through `tenantProxy.ts` on the server
3. Proxy connects to `https://alfred-{name}:3100/api/v1/*` over the Tailscale mesh
4. Tenant API authenticates via Bearer token (AES-256-GCM encrypted in PostgreSQL)
5. Tenant API executes scoped `alfred vault` CLI commands locally

### Data Flow: Provisioning

1. SaaS creates a PgBoss job with instance configuration
2. Worker spawns `node alfred-ctrl/dist/index.mjs provision {name}`
3. alfred-ctrl: generates SSH key, creates Hetzner server, uploads secrets via SSH, starts containers, creates Cloudflare Tunnel
4. Secrets are **never** in cloud-init `user_data` (readable via Hetzner metadata API); always uploaded post-boot via SSH

## Network Security

### Four-Layer Network Model

Alfred implements four overlapping network security layers. Any single layer provides meaningful protection; together they form defense-in-depth.

<AccordionGroup>
<Accordion title="Layer 1: Hetzner Cloud Firewall" icon="cloud">

Applied at the hypervisor level before traffic reaches the VM:

| Rule | Protocol | Port | Source | Purpose |
|------|----------|------|--------|---------|
| SSH | TCP | 22 | `ADMIN_SSH_CIDRS` | Admin access only |
| Tailscale | UDP | 41641 | 0.0.0.0/0 | WireGuard mesh |
| ICMP | ICMP | -- | 0.0.0.0/0 | Ping/diagnostics |

`ADMIN_SSH_CIDRS` is configurable via environment variable. In production, restrict to the control plane IP (e.g., `138.199.236.244/32`).

</Accordion>

<Accordion title="Layer 2: Host UFW Firewall" icon="server">

Mirrors Hetzner rules at the OS level. Default deny incoming. Defense-in-depth against firewall misconfiguration at either layer.

```bash
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp        # SSH
ufw allow 41641/udp     # Tailscale WireGuard
```

</Accordion>

<Accordion title="Layer 3: Tailscale Mesh (WireGuard)" icon="network-wired">

All inter-VM communication uses Tailscale's encrypted WireGuard tunnels. The ACL policy enforces tenant isolation:

| Source | Destination | Action | Purpose |
|--------|-------------|--------|---------|
| `tag:admin` | `tag:tenant:*` | Allow | Control plane to all tenants |
| `tag:admin` | `tag:admin:*` | Allow | Admin to admin |
| `tag:tenant` | `tag:tenant:*` | **Deny** | Tenant isolation (implicit) |
| `tag:tenant` | `tag:admin:*` | **Deny** | No tenant-to-admin (implicit) |
| `autogroup:member` | `*:*` | Allow | Personal/legacy devices |

New instances join with `tag:tenant` via ephemeral pre-authorized auth keys. The control plane uses `tag:admin`. Tenant VMs **cannot** communicate with each other.

</Accordion>

<Accordion title="Layer 4: Cloudflare Tunnel" icon="globe">

Each tenant VM runs `cloudflared` as a systemd service. The tunnel is outbound-only -- no inbound firewall ports are opened for user traffic.

```yaml
# /etc/cloudflared/config.yml
tunnel: <tunnel-id>
credentials-file: /etc/cloudflared/credentials.json
ingress:
  - hostname: acme.alfred.black
    service: http://localhost:18789
  - service: http_status:404
```

Cloudflare's edge provides TLS termination, DDoS protection, and optional WAF. SSL mode is set to **Full** (Cloudflare validates origin certificate).

</Accordion>
</AccordionGroup>

### SSH Hardening

Applied via `/etc/ssh/sshd_config.d/99-alfred-hardening.conf`:

| Setting | Value | Purpose |
|---------|-------|---------|
| PermitRootLogin | no | No root SSH access |
| PasswordAuthentication | no | Key-only authentication |
| MaxAuthTries | 3 | Brute-force mitigation |
| MaxSessions | 3 | Session limit |
| X11Forwarding | no | No X11 tunneling |
| AllowTcpForwarding | no | No port forwarding |
| AllowAgentForwarding | no | No agent forwarding |
| ClientAliveInterval | 300 | 5-minute idle timeout |
| ClientAliveCountMax | 2 | 10-minute max idle |
| LoginGraceTime | 30 | 30-second login window |

Fail2ban: 3 retries, 1-hour ban. Ed25519 keys only (generated per-instance).

### Service Binding

No service is directly exposed to the network:

| Service | Bind Address | User-Facing Via | Admin Via |
|---------|-------------|-----------------|-----------|
| OpenClaw (:18789) | 127.0.0.1 | Cloudflare Tunnel | -- |
| Temporal gRPC (:7233) | 127.0.0.1 | -- | Docker internal |
| Temporal UI (:8233) | 127.0.0.1 | -- | Tailscale Serve (HTTPS) |
| Tenant API (:3100) | 127.0.0.1 | -- | Tailscale Serve (HTTPS) |

## Tenant VM Containerization

### Container Security Controls

Every container runs with defense-in-depth hardening:

| Control | temporal | openclaw | alfred (worker) | init |
|---------|----------|----------|-----------------|------|
| `no-new-privileges` | Yes | Yes | Yes | -- |
| `cap_drop: ALL` | -- | Yes | Yes | -- |
| `cap_add` | None | `DAC_OVERRIDE` | None | None |
| `mem_limit` | 2 GB | 2 GB | 2 GB | -- |
| `pids_limit` | 256 | 256 | 256 | -- |
| Health check | Yes | Yes | No | -- |
| Restart policy | unless-stopped | unless-stopped | unless-stopped | None |
| Port binding | 127.0.0.1 | 127.0.0.1 | None | None |
| Run user | uid 1000 (temporal) | uid 1000 (node) | Default | Default |

<Note>
**Why `DAC_OVERRIDE` on OpenClaw?** The container mounts volumes owned by `deploy` (uid 1000 on host). The `node` user inside (also uid 1000) needs to read/write files that may have been created by other containers or the host. `DAC_OVERRIDE` permits bypassing owner/group permission checks within the mounted volumes only.
</Note>

### Container Images

| Container | Image | Source |
|-----------|-------|--------|
| init | `ssdavidai00/alfred-init:latest` | Custom -- scaffolds vault, OpenClaw config |
| temporal | `temporalio/temporal:latest` | Official Temporal dev server |
| openclaw | `ssdavidai00/alfred-openclaw:latest` | Custom -- OpenClaw gateway + skills |
| alfred | `ssdavidai00/alfred-worker:latest` | Custom -- Python Alfred tools |

### Startup Order

```
init (one-shot) ──completed──> temporal ──healthy──> openclaw ──healthy──> alfred
```

The `init` container scaffolds directories and exits. Temporal must pass its gRPC health check before OpenClaw starts. OpenClaw must respond on `/health` before Alfred starts.

### Volume Mounts

All persistent data lives on the LUKS2-encrypted volume:

```
/mnt/encrypted/
├── vault/              # Obsidian vault (Markdown + YAML frontmatter)
│   ├── inbox/          # Raw inputs for Curator processing
│   ├── person/         # Entity type directories (19 types)
│   ├── decision/       # Learning type directories (5 types)
│   ├── _templates/     # Per-type Markdown templates
│   └── _bases/         # Dataview base views
├── openclaw/           # OpenClaw state (openclaw.json, device pairing)
│   └── workspace/vault # Symlink to vault for wikilink resolution
├── alfred/             # Worker state, config, audit log, PID file
│   └── .gateway-token  # OpenClaw gateway auth token
└── temporal/           # Temporal SQLite database
```

Nothing sensitive is stored on the unencrypted root filesystem except the LUKS keyfile at `/opt/alfred/luks.key` (mode 0600, root-only).

### Docker Network

Containers communicate via Docker's default bridge network. The `alfred` worker connects outbound to OpenClaw via Docker DNS (`ws://openclaw:18789`). The worker has **no** exposed ports -- it reads/writes the vault filesystem directly.

## Encryption & Secrets

### Encryption at Rest

<CardGroup cols={2}>
<Card title="LUKS2 Volume" icon="hard-drive">
AES-256-XTS with 4 KB random keyfile. Auto-unlock via systemd service before Docker starts. All vault data, OpenClaw state, and worker state are on this volume.
</Card>
<Card title="PostgreSQL (SaaS)" icon="database">
Tenant API keys encrypted with AES-256-GCM. Unique random IV per operation. Auth tag verification on decryption. Key stored in `COLUMN_ENCRYPTION_KEY` env var.
</Card>
</CardGroup>

### Encryption in Transit

| Path | Encryption | Authentication |
|------|-----------|----------------|
| User to Cloudflare | TLS 1.3 (Cloudflare edge) | None or CF Access |
| Cloudflare to VM | Cloudflare Tunnel (TLS/QUIC) | Tunnel credentials |
| SaaS to Tenant API | Tailscale WireGuard (Curve25519) | Bearer token |
| Control Plane to VM | SSH (Ed25519) | Public key |
| alfred-ctrl to Hetzner | HTTPS | Bearer token |
| alfred-ctrl to Cloudflare | HTTPS | API key + email |
| alfred-ctrl to Tailscale | HTTPS | API key (basic auth) |

### Secret Lifecycle

Secrets follow a strict lifecycle during provisioning:

<Steps>
<Step title="Generate">
Ed25519 SSH keypair, LUKS keyfile, gateway token, AAS API key, restic password, Cloudflare tunnel secret -- all generated with cryptographic randomness.
</Step>
<Step title="Upload via SSH">
After the server boots and SSH is available, secrets are uploaded over the encrypted SSH channel. **Never** placed in cloud-init `user_data` (readable via Hetzner metadata API at `169.254.169.254`).
</Step>
<Step title="Store on encrypted volume">
On the VM, secrets land in `/opt/alfred/compose/.env` (mode 0600) or `/etc/cloudflared/credentials.json` (mode 0600). Application data lives on the LUKS2-encrypted volume.
</Step>
<Step title="Backup to control plane">
LUKS keyfile and restic credentials are backed up to the control plane at `data/ssh_keys/{id}/` (mode 0600) for disaster recovery.
</Step>
<Step title="Cleanup">
Ephemeral secrets (Tailscale auth key, bootstrap script) are deleted after use. Bootstrap script set to mode 0700 during execution, then removed.
</Step>
</Steps>

### Secret Storage Locations

| Secret | Location | Protection | Risk Level |
|--------|----------|-----------|------------|
| SSH private keys | `data/ssh_keys/{id}/id_ed25519` | File mode 0600 | High |
| LUKS keyfile backup | `data/ssh_keys/{id}/luks.key` | File mode 0600, **plaintext** | High |
| Restic credentials | `data/ssh_keys/{id}/restic.env` | File mode 0600, **plaintext** | Medium |
| Tenant API keys | PostgreSQL `instances.apiKey` | AES-256-GCM encrypted | Low |
| Gateway tokens | SQLite `instances.gateway_token` | **Plaintext** | High |
| CF tunnel credentials | `/etc/cloudflared/credentials.json` | 0600, on LUKS volume | Medium |
| Hetzner/CF/TS API tokens | `.env` files | File permissions | High |

<Warning>
The SQLite database on the control plane stores `gateway_token` and `api_key` in plaintext. A control plane compromise exposes all tenant tokens. Encrypting these fields (matching the SaaS AES-256-GCM pattern) is recommended before GA.
</Warning>

## Authentication & Authorization

### Authentication Methods

| Layer | Method | Details |
|-------|--------|---------|
| SaaS app | Email/password or OAuth | Wasp framework, bcrypt hashing, JWT sessions |
| Tenant API | Bearer token | `alf_<32 hex>`, timing-safe comparison, AES-encrypted in DB |
| OpenClaw gateway | URL token | `?token={gateway_token}`, persistent per-instance |
| SSH | Ed25519 public key | Per-instance keypair, no password auth |
| Tailscale | Pre-authorized auth key | Ephemeral, single-use, 1-hour expiry |

### Vault Scope Enforcement

Each Alfred tool operates under a restricted scope that limits vault operations:

| Operation | Curator | Janitor | Distiller | API (default) |
|-----------|---------|---------|-----------|---------------|
| read | Yes | Yes | Yes | Yes |
| search | Yes | Yes | Yes | Yes |
| list | Yes | Yes | Yes | Yes |
| context | Yes | Yes | Yes | Yes |
| **create** | All types | **No** | Learn types only | All types |
| **edit** | All types | All types | **No** | All types |
| **move** | Inbox only | **No** | **No** | All types |
| **delete** | **No** | All types | **No** | All types |

Scope is enforced in `vault/scope.py` via `check_scope()` before every operation. The scope name is injected as `ALFRED_VAULT_SCOPE` environment variable.

- **Curator** can create and edit records but cannot delete. Can only move files out of `inbox/`.
- **Janitor** can edit and delete (for cleanup) but cannot create or move.
- **Distiller** can only create learning types (assumption, decision, constraint, contradiction, synthesis). Cannot edit, move, or delete.

### SaaS-to-Tenant Proxy

The proxy (`tenantProxy.ts`) enforces multiple checks:

1. User must be authenticated (Wasp session cookie)
2. User must own an instance with `status = "running"`
3. Instance must have `tailscaleHostname` and `apiKey` provisioned
4. API key decrypted from PostgreSQL before forwarding
5. 15-second request timeout via AbortController
6. Error responses sanitized (no internal details leaked)

## Input Validation & Output Sanitization

### Input Validation

<AccordionGroup>
<Accordion title="Instance Name" icon="input-text">
```typescript
if (!/^[a-zA-Z0-9_-]+$/.test(config.customer_name)) {
  throw new Error("Invalid customer name");
}
```
Prevents template injection in Nunjucks, path traversal in filesystem operations, and command injection in shell commands.
</Accordion>

<Accordion title="Vault Record Paths" icon="folder-tree">
```typescript
const normalized = args.path.replace(/\\/g, "/");
if (normalized.includes("..") || normalized.startsWith("/") || normalized.includes("\0")) {
  throw new HttpError(400, "Invalid record path");
}
```
Prevents directory traversal in the SaaS proxy's vault record endpoint.
</Accordion>

<Accordion title="Vault Operations (Python)" icon="file-lines">
- `Path.resolve()` + prefix check: all paths must stay within the vault directory
- Type validation against `KNOWN_TYPES` (19 record types)
- Status validation against per-type allowed values
- Required fields checked before record creation
- YAML frontmatter parsed with structured error handling
</Accordion>

<Accordion title="SQL Queries" icon="database">
All database code uses parameterized queries with `?` placeholder binding. No string concatenation in SQL statements anywhere in the codebase.
</Accordion>

<Accordion title="Command Execution" icon="terminal">
All shell commands use `child_process.execFile` with **array arguments** (no shell interpretation), preventing injection even with user-controlled data:
```javascript
execFile('/usr/bin/alfred', ['vault', 'create', 'person', '--data', JSON.stringify(data)])
```
</Accordion>
</AccordionGroup>

### Output Sanitization

**Tenant proxy error responses** are sanitized before returning to the user:

| Status Code | User Sees | Internal Detail |
|-------------|-----------|-----------------|
| 404 | "Resource not found" | Hidden |
| 401 | "Authentication failed" | Hidden |
| 500+ | "Internal tenant error" | Hidden |
| Timeout | "Tenant API request timed out" | Hidden |
| Connection failure | "Failed to reach tenant instance" | Hidden |

**Health check JSON** is constructed using `jq -n` with `--argjson`/`--arg` flags instead of string interpolation, preventing JSON injection via crafted container names.

## Audit Logging & Monitoring

### Vault Mutation Log

Every vault write operation is recorded in two locations:

1. **Session log**: Temporary JSONL file per agent session -- records `op`, `path`, `timestamp`, `field_changes`. Used by tools to verify agent work.
2. **Audit log**: Append-only JSONL at `data/vault_audit.log`. Persistent record of all creates, edits, moves, deletes. Survives container restarts.

```json
{
  "timestamp": "2026-02-26T10:30:00Z",
  "operation": "create",
  "type": "person",
  "title": "Alice Smith",
  "scope": "curator",
  "session": "abc123",
  "path": "vault/person/Alice Smith.md"
}
```

### Health Monitoring

Health checks run periodically over SSH, executing `/opt/alfred/healthcheck.sh`:

| Check | Threshold | Result |
|-------|-----------|--------|
| Container states | Any non-init container not running | Degraded |
| Disk utilization | > 90% | Degraded |
| Memory utilization | > 95% | Degraded |
| cloudflared service | Not "active" | Degraded |
| Script failure | Non-zero exit | Down |
| SSH failure | Connection refused/timeout | Unreachable |

<Note>
The `init` container always shows `exited` with exit code 0. This is expected -- it runs once at startup to scaffold the vault structure. The health checker treats this as healthy.
</Note>

### Alert Webhooks

Status changes trigger Slack or Discord notifications with customer name, old/new status, and instance IP. Configured via `ALERT_WEBHOOK_URL` and `ALERT_WEBHOOK_TYPE`.

### Instance Events

alfred-ctrl records lifecycle events in SQLite: `provisioned`, `health_ok`, `health_degraded`, `health_down`, `health_unreachable`, `destroyed`, `updated`, `rolled_back`.

## Backup & Disaster Recovery

### Automated Backups

Daily restic backups at 3:00 AM (with 30-minute random jitter):

<Steps>
<Step title="Stop containers">
Alfred, OpenClaw, and Temporal containers are stopped for data consistency.
</Step>
<Step title="Backup encrypted volume">
`restic backup /mnt/encrypted --exclude=/mnt/encrypted/temporal`
</Step>
<Step title="Restart containers">
All containers restarted. Typical downtime: ~5 minutes.
</Step>
<Step title="Prune old snapshots">
Retention: 7 daily, 4 weekly, 6 monthly.
</Step>
</Steps>

**Storage**: Hetzner Object Storage (S3-compatible), per-instance prefix path.
**Encryption**: Restic encrypts all data with a per-instance random password (32-byte hex).

<Tip>
Restic backup failure during provisioning is a **hard error** -- the instance will not be marked as running until a successful backup completes. This ensures every instance has at least one known-good backup from day one.
</Tip>

### Recovery Procedures

**VM lost, volume intact:**
1. Create new server in the same region
2. Attach existing encrypted volume
3. Unlock with backed-up LUKS keyfile
4. Restore docker-compose and .env, start containers

**Volume also lost:**
1. Create new server + volume, provision fresh
2. Restore from restic: `restic restore latest --target /mnt/encrypted/`
3. Start containers -- vault data is restored

## Supply Chain Security

### Package Installation

| Package | Method | Verification |
|---------|--------|-------------|
| Tailscale | APT with GPG key from `pkgs.tailscale.com` | GPG signature |
| cloudflared | APT with GPG key from `pkg.cloudflare.com` | GPG signature |
| Docker | APT from Docker's official repository | GPG signature |
| System packages | Ubuntu APT | Standard APT signatures |

<Note>
Previously, Tailscale was installed via `curl | sh` (supply chain risk). This was remediated by switching to APT with GPG key verification.
</Note>

### Container Image Risks

All custom images use mutable `:latest` tags. The auto-update timer pulls every 15 minutes without signature verification:

```
docker compose pull --quiet && docker compose up -d --remove-orphans
```

**Risks**: Image mutation, malicious push to Docker Hub, no health gate between pull and restart, no automatic rollback on failure.

**Recommended mitigations** (not yet implemented):
- Pin images by digest (`@sha256:...`)
- Sign images with cosign and verify before pull
- Add health check gate: pull, start canary, verify health, promote
- Enable Docker Content Trust

### OpenClaw Skill Supply Chain

OpenClaw supports installing third-party skills from ClawHub. Known threats:

- **CVE-2026-25253** (CVSS 8.8): OpenClaw Control UI RCE via untrusted `gatewayUrl`
- **ClawHavoc campaign** (Jan 2026): 1,184+ malicious skills on ClawHub

Alfred mitigates by binding OpenClaw to localhost and fronting it with Cloudflare Tunnel (no direct internet exposure).

## Attack Surface Analysis

### External Attack Vectors

| Vector | Exposure | Auth | Impact | Mitigation |
|--------|----------|------|--------|------------|
| SSH brute-force | Port 22 (admin CIDRs) | Ed25519 key | Server access | fail2ban, key-only, MaxAuthTries=3 |
| CF Tunnel hijack | Public HTTPS | Tunnel creds | Traffic interception | Creds on encrypted volume, outbound-only |
| DNS poisoning | `*.alfred.black` | Cloudflare proxied | User redirection | Cloudflare DNS, proxied records |
| SaaS exploitation | `app.alfred.black` | Wasp auth | User data, provisioning | Input validation, parameterized SQL |
| OpenClaw skill RCE | Via CF Tunnel | Gateway token | Container access | Localhost binding, container caps |
| Tailscale compromise | WireGuard mesh | Ephemeral key | Mesh access | Ephemeral keys, ACL isolation |

### Internal Attack Vectors (post-compromise)

| Vector | Prerequisite | Impact | Mitigation |
|--------|-------------|--------|------------|
| Container escape | Container access | Host access | `no-new-privileges`, `cap_drop: ALL`, limits |
| LUKS key theft | Root on VM | All tenant data | Keyfile 0600, root-only |
| Lateral movement | Compromised tenant | Other tenants | Tailscale ACL denies tenant-to-tenant |
| Control plane compromise | SSH to control plane | **All tenants** | Restricted SSH CIDRs, key auth |
| Docker socket abuse | `deploy` user | Host root equivalent | docker group (inherent risk, accepted) |

### Tenant Isolation Model

Each tenant is isolated at multiple levels:

| Layer | Isolation Method |
|-------|-----------------|
| **Compute** | Dedicated Hetzner VM per tenant |
| **Storage** | Dedicated LUKS2-encrypted volume per tenant |
| **Network** | Tailscale ACL prevents tenant-to-tenant traffic |
| **DNS** | Unique Cloudflare Tunnel per tenant |
| **Auth** | Unique API keys, gateway tokens, SSH keys |
| **Firewall** | Shared Hetzner firewall policy |

**Shared resources** (single point of compromise):
- Hetzner API token (manages all VMs)
- Tailscale tailnet (isolation via ACLs only)
- Cloudflare account (per-tenant tunnels and DNS)
- Docker Hub images (pulled by all tenants)
- S3 bucket (per-tenant prefix path)

## Out of Scope & Accepted Risks

### Explicitly Out of Scope

| Area | Reason |
|------|--------|
| DDoS protection | Delegated to Cloudflare |
| WAF rules | Delegated to Cloudflare; no app-level WAF |
| Container image scanning | No Trivy/Grype in CI |
| Runtime threat detection | No Falco/Sysdig; container hardening only |
| Key Management Service | No Vault/KMS; secrets in env vars and files |
| SOC 2 / ISO 27001 | No formal compliance program |
| Penetration testing | No formal pentest conducted |
| Centralized logging | Per-VM in systemd journal; no SIEM |
| IDS/IPS | fail2ban only; no OSSEC/Wazuh |

### Accepted Risks

| Risk | Rationale |
|------|-----------|
| `deploy` user in docker group | Required for container management. Docker group = root equivalent. Mitigated by SSH key auth + fail2ban. |
| Shared Hetzner API token | Single token for all infrastructure. Compromise = total fleet control. Accepted for operational simplicity. |
| Shared Tailscale tailnet | All tenants on same tailnet, isolated by ACLs. ACL misconfiguration = lateral movement. ACLs are version-controlled. |
| Auto-update without verification | 15-min pull cycle, no signatures. Accepted for rapid patching. Pin digests before GA. |
| LUKS auto-unlock | Keyfile on root disk enables unattended boot. Physical access (Hetzner DC) could extract it. Hetzner DCs are ISO 27001. |

## Incident Response

If a tenant instance is compromised:

<Steps>
<Step title="Isolate">
Remove from Tailscale network (`tailscale logout` on the VM). Delete the Cloudflare Tunnel DNS record to cut user access.
</Step>
<Step title="Snapshot">
Create a Hetzner snapshot for forensic analysis before any changes.
</Step>
<Step title="Rotate">
Rotate: API key, SSH keys, Tailscale auth key, gateway token. Revoke the Cloudflare tunnel.
</Step>
<Step title="Audit">
Review `vault_audit.log` for unauthorized mutations. Check systemd journal for suspicious activity.
</Step>
<Step title="Restore">
Provision a new instance and restore from the last known-good restic backup.
</Step>
<Step title="Investigate">
Analyze the snapshot to identify the attack vector. Update hardening controls.
</Step>
</Steps>

<Warning>
Do **not** destroy the compromised instance until forensic analysis is complete. Create a snapshot first.
</Warning>
