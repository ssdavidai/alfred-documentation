## Overview

Alfred's vault uses a structured Markdown format with YAML frontmatter. Every record is a `.md` file with metadata that defines its type, relationships, and properties.

## Record Format

All vault records follow this pattern:

```markdown
---
type: person
created: 2026-02-26
org: "[[org/Acme Corp]]"
email: alice@example.com
---

# Alice Smith

Notes about Alice go here. You can use [[wikilinks]] to reference other records.
```

<Note>
The vault is the source of truth for all data. State files are just bookkeeping and can be deleted to force re-processing.
</Note>

## Record Types

Alfred supports 19 distinct record types, organized into two categories:

### Entity Types (14)

Core operational records representing people, projects, tasks, and other tangible entities:

| Type | Purpose | Directory |
|------|---------|-----------|
| `person` | People and contacts | `vault/person/` |
| `org` | Organizations and companies | `vault/org/` |
| `project` | Projects and initiatives | `vault/project/` |
| `task` | Tasks and action items | `vault/task/` |
| `event` | Events and occurrences | `vault/event/` |
| `note` | General notes and observations | `vault/note/` |
| `location` | Physical or virtual locations | `vault/location/` |
| `process` | Processes and workflows | `vault/process/` |
| `account` | Accounts and credentials | `vault/account/` |
| `asset` | Assets and resources | `vault/asset/` |
| `conversation` | Conversations and discussions | `vault/conversation/` |
| `input` | Incoming items to be processed | `vault/input/` |
| `run` | Execution runs and operations | `vault/run/` |
| `session` | Working sessions | `vault/session/` |

### Learning Types (5)

Meta-records that capture latent knowledge extracted from operational records:

| Type | Purpose | Directory |
|------|---------|-----------|
| `assumption` | Implicit assumptions | `vault/assumption/` |
| `decision` | Decisions and rationale | `vault/decision/` |
| `constraint` | Constraints and limitations | `vault/constraint/` |
| `contradiction` | Contradictions and conflicts | `vault/contradiction/` |
| `synthesis` | Syntheses and connections | `vault/synthesis/` |

<Tip>
Learning types are automatically extracted by the Distiller tool from your operational records. They are stored in their own top-level directories alongside entity types (e.g., `vault/assumption/`, `vault/decision/`).
</Tip>

## Relationships

Relationships between records use Obsidian wikilink syntax:

```markdown
[[type/Record Name]]
```

Examples:
- `[[person/Alice Smith]]` - links to a person
- `[[project/Q1 Launch]]` - links to a project
- `[[org/Acme Corp]]` - links to an organization

<Warning>
The type prefix (`person/`, `project/`, etc.) is required. Bare wikilinks like `[[Alice Smith]]` are not valid in Alfred's schema.
</Warning>

## Status Fields

Most record types have a `status` field with type-specific valid values:

| Type | Valid Status Values |
|------|---------------------|
| `account` | `active`, `suspended`, `closed`, `pending` |
| `asset` | `active`, `retired`, `maintenance`, `disposed` |
| `assumption` | `active`, `challenged`, `invalidated`, `confirmed` |
| `constraint` | `active`, `expired`, `waived`, `superseded` |
| `contradiction` | `unresolved`, `resolved`, `accepted` |
| `conversation` | `active`, `waiting`, `resolved`, `closed`, `archived` |
| `decision` | `draft`, `final`, `superseded`, `reversed` |
| `event` | _(no status constraint)_ |
| `input` | `unprocessed`, `processed`, `deferred` |
| `location` | `active`, `inactive` |
| `note` | `draft`, `active`, `review`, `final` |
| `org` | `active`, `inactive` |
| `person` | `active`, `inactive` |
| `process` | `active`, `proposed`, `design`, `deprecated` |
| `project` | `active`, `paused`, `completed`, `abandoned`, `proposed` |
| `run` | `active`, `completed`, `blocked`, `cancelled` |
| `session` | `active`, `completed` |
| `synthesis` | `draft`, `active`, `superseded` |
| `task` | `todo`, `active`, `blocked`, `done`, `cancelled` |

## Required Fields

All record types share the same two globally required fields:

<AccordionGroup>
<Accordion title="Required Fields (all types)">
- `type`: The record type identifier (e.g., `person`, `task`, `project`)
- `created`: The creation date of the record
</Accordion>

<Accordion title="Optional Common Fields">
- `name`: Display name for the record (most types use this as the title field)
- `subject`: Display name for `conversation` and `input` types (used instead of `name`)
- `status`: Current status (valid values depend on type; see status table above)
- `owner`: Who is responsible (wikilink to person)
- `tags`: Array of tags
- `related`: Array of related record wikilinks
</Accordion>
</AccordionGroup>

<Note>
There are no per-type required fields beyond `type` and `created`. All other fields are optional.
</Note>

## Name Field

Most types use `name` as their title field. Two types use `subject` instead:

| Type | Name Field |
|------|-----------|
| `conversation` | `subject` |
| `input` | `subject` |
| All other types | `name` |

## List Fields

The following fields are treated as multi-value lists (arrays) across all record types:

`tags`, `aliases`, `related`, `relationships`, `participants`, `outputs`, `depends_on`, `blocked_by`, `based_on`, `supports`, `challenged_by`, `approved_by`, `confirmed_by`, `invalidated_by`, `cluster_sources`, `governed_by`, `references`, `project`

## Schema Endpoint

You can retrieve the complete schema programmatically:

```bash
curl -H "Authorization: Bearer alf_YOUR_API_KEY" \
  https://app.alfred.black/api/v1/vault/schema
```

This returns a JSON object with:
- All 19 record types
- Valid status values per type
- Directory mappings
- List field definitions
- Name field overrides

<Note>
The schema is defined in the Alfred source at `src/alfred/vault/schema.py` and is the authoritative reference for all vault operations.
</Note>
