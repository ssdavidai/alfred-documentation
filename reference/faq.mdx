## About Alfred

<AccordionGroup>

<Accordion title="What is Alfred Black, exactly?">
A managed butler service. Sir gets a dedicated server, a vault that grows from Sir's email and the apps Sir connects, a small staff of specialist workers attending to it, and an agent — Alfred — reachable through the dashboard, email, SMS, voice, Slack, and Telegram. Sir does not install software or maintain configuration. See the [Introduction](/) for the full picture.
</Accordion>

<Accordion title="Where is Sir's data stored?">
On Sir's own dedicated Hetzner Cloud server, encrypted at rest with LUKS2. Every subscriber gets their own machine — no shared databases or commingled tables. Services bind to localhost; external access exists only via Sir's private Tailscale mesh. See the [Infrastructure layer](/architecture/infrastructure) for details.
</Accordion>

<Accordion title="Who can see Sir's vault?">
Only Sir, and the specialist workers running on Sir's own server. The SaaS layer at `alfred.black` proxies API calls into Sir's tenant via Tailscale but does not store vault contents. Composio holds OAuth credentials for connected apps in its own vault — Sir's tenant never sees the secrets. Cross-tenant access exists only on **Alfred Prime** instances and only between explicitly listed peers.
</Accordion>

<Accordion title="Can Sir export the vault?">
Yes. The vault is a tree of plain Markdown files with YAML frontmatter — Obsidian-compatible. Sir can pair a device (`/guides/device-pairing`) and sync the vault directly, or read it through the API at `GET /api/v1/vault/records/...`. Nothing is locked into a proprietary format.
</Accordion>

<Accordion title="What happens if Sir cancels?">
Sir's instance is deprovisioned on Sir's request. Sir is given an opportunity to export the vault first. After cancellation the Hetzner server is destroyed and disks are wiped per Hetzner's standard procedure.
</Accordion>

</AccordionGroup>

## Apps and integrations

<AccordionGroup>

<Accordion title="How does Sir add a new integration?">
Open **Apps** in the dashboard sidebar. The catalog lists 1,000+ Composio-backed integrations. Click an app, authorise via the OAuth popup, and Alfred wires up the stream, the schedule, and the action toolkit automatically. Sir can also disable individual streams or actions per app.
</Accordion>

<Accordion title="Why does the dashboard say 'Configuring…' for a moment after connecting an app?">
The first Composio app Sir connects (and disconnect of the last one) writes `composio_execute` into the OpenClaw gateway's allowlist, which triggers a ~40-second restart of the gateway process. The dashboard hides this behind the **Configuring…** banner and resumes once the gateway responds healthy twice in a row.
</Accordion>

<Accordion title="Is Sir's OAuth refresh handled?">
Composio holds the credentials and refreshes them on Sir's behalf. Sir authorises once. If a token genuinely revokes (Sir changes Google password, revokes from the Google security page), the relevant stream will start failing — Alfred surfaces this and asks Sir to re-authorise.
</Accordion>

<Accordion title="Can Alfred act on Sir's behalf, or only read?">
Both. Every connected app gives Alfred **data in** (streams polling new email, calendar events, GitHub notifications) and **actions out** (send the email, create the calendar event, post to Slack). Actions are executed via the Composio gateway under Sir's account.
</Accordion>

</AccordionGroup>

## Channels

<AccordionGroup>

<Accordion title="How does Sir send Alfred an email?">
Sir's address is `alfred.<username>@mail.alfred.black`. The first time Sir replies to the First Brief from Sir's Google address, that address becomes an authorised sender — emails from it are handled by the agent (full reply, attachments work both ways). Mail from unfamiliar senders is captured into a stream as a record, not a conversation. Sir can manage the list of authorised senders conversationally ("authorise X as a trusted sender") or via `GET/POST/DELETE /api/v1/auth/senders`.
</Accordion>

<Accordion title="Can Alfred call Sir, or does Sir call Alfred?">
Sir calls Alfred. Sir gets a dedicated phone number on signup; Alfred answers it. The voice agent is OpenAI Realtime with a context primer fetched at call start — open matters, last seven days of session summaries, recent messages — so Alfred knows what's been going on. Hangup transcripts are written back to the vault for cross-channel memory.
</Accordion>

<Accordion title="Does SMS share memory with voice and email?">
Yes. SMS replies, voice transcripts, email threads, and Slack DMs all write to the same vault. What Sir said by voice this morning informs the email reply this afternoon.
</Accordion>

</AccordionGroup>

## How Alfred behaves

<AccordionGroup>

<Accordion title="What if Alfred makes a mistake?">
Correct it the way Sir would correct a new butler — say so. Corrections recorded in chat ("that one wasn't an expense, it was a personal payment") are observed by the learning hook and folded into the next nightly reflection, which adjusts the relevant instinct. Persistently wrong instincts are deprecated automatically.
</Accordion>

<Accordion title="How do approvals work?">
Instincts that execute on Sir's behalf go through a trust gradient. Under 10 observations: every run requires Sir's approval. 10–50: configurable per instinct. 50+ with consistent approval history: Alfred acts autonomously and reports after the fact. Approval requests appear as tasks in the dashboard — Sir clicks Approve or Reject.
</Accordion>

<Accordion title="How long until Alfred starts handling things on its own?">
Depends on volume. Alfred needs at least three observations of the same pattern before forming an instinct. With regular daily use, the first routine matters tend to be handled within one to two weeks.
</Accordion>

<Accordion title="Can Sir disable Alfred's learning?">
Yes. `POST /api/v1/learning/disable` (or the dashboard toggle) pauses all learning processes. Existing observations and instincts are preserved. Re-enable at any time.
</Accordion>

<Accordion title="What does Sir see on the homepage?">
The **Vault Nebula** — a data-driven visualisation of Sir's knowledge graph as drifting clouds of related records. People, organisations, projects, decisions; clustered by meaning, labelled, and connected.
</Accordion>

</AccordionGroup>

## Models and configuration

<AccordionGroup>

<Accordion title="What language model does Alfred use?">
The agent Sir speaks to is configurable per tenant — current production runs on `xai/grok-4` or `google/gemini-2.5-pro` for the main agent, and `xai/grok-4-1-fast-reasoning` for the background workers. All LLM calls route through the OpenClaw gateway, so changing the model is a config edit, not a code change. Sir can manage this from **Settings → Services**.
</Accordion>

<Accordion title="Can Sir bring Sir's own API key?">
Yes. Sir's API keys for OpenAI, Anthropic, Google, xAI, OpenRouter, and others can be set per tenant via `PATCH /api/v1/admin/credentials` or the dashboard. The OpenClaw gateway picks the appropriate key for whatever model the agent is configured to use.
</Accordion>

<Accordion title="Can Sir run a local model?">
Embeddings already run locally — every tenant ships an Ollama sidecar serving `nomic-embed-text` for the Surveyor. Local inference for the main agent isn't supported today; the OpenClaw gateway expects an OpenAI-compatible HTTP API and the configured providers are all hosted.
</Accordion>

</AccordionGroup>

## Operations and troubleshooting

<AccordionGroup>

<Accordion title="How long until Sir's inbox uploads are attended to?">
Usually 10–60 seconds. The Curator watches the inbox and processes new items automatically. If something seems delayed, check `GET /api/v1/workers/status` from the dashboard or the API.
</Accordion>

<Accordion title="Sir's vault has broken links — what then?">
The Janitor will catch them on its next sweep. To force a check immediately: `POST /api/v1/workers/janitor/scan` then `POST /api/v1/workers/janitor/fix`. Both are also available from the dashboard's worker controls.
</Accordion>

<Accordion title="Sir wants to surface deeper insights from existing records">
Ask the Distiller: `POST /api/v1/workers/distiller/scan` then `POST /api/v1/workers/distiller/run`. The Distiller produces learning records — assumptions, decisions, constraints, contradictions — from existing vault content.
</Accordion>

<Accordion title="Are Sir's backups handled?">
Yes. Sir's instance runs daily encrypted backups of the vault, the workspace, and the Temporal database. Sir does not need to manage backups manually.
</Accordion>

<Accordion title="What does Sir do if a workflow appears stuck?">
List active workflows via `GET /api/v1/workflows` (or from the dashboard's Workflows tab). Workflows can be terminated, signalled, or restarted from there. Note that some activities — such as the chore-deploy step that restarts the learn worker — appear to fail with a heartbeat timeout but the underlying work succeeds; the dashboard accounts for this.
</Accordion>

</AccordionGroup>
