Alfred has three AI-powered workers that maintain your vault. Each runs as a daemon with auto-restart, health checks, and manual control via API.

## The Three Workers

<CardGroup cols={3}>
  <Card title="Curator" icon="inbox">
    Processes raw inputs from inbox into structured records
  </Card>
  <Card title="Janitor" icon="broom">
    Scans for structural issues and fixes them

  <Card title="Distiller" icon="flask">
    Extracts latent knowledge from operational records
  </Card>
</CardGroup>

## Worker Status

### Check All Workers

```bash
curl -X GET https://app.alfred.black/api/v1/workers/status \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

The status endpoint always returns raw text from the `alfred status` command:

```json
{
  "raw": "curator: running (pid 1234)\njanitor: running (pid 1235)\ndistiller: running (pid 1236)"
}
```

### Check Individual Worker

```bash
curl -X GET https://app.alfred.black/api/v1/workers/janitor/status \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

## Starting and Stopping

### Start Workers

```bash
curl -X POST https://app.alfred.black/api/v1/workers/up \
  -H "Authorization: Bearer YOUR_API_KEY"
```

You can optionally start only specific workers:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/up \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"only": "curator,janitor"}'
```

**Response:**

```json
{
  "message": "Workers started"
}
```

### Stop Workers

```bash
curl -X POST https://app.alfred.black/api/v1/workers/down \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

```json
{
  "message": "Workers stopped"
}
```

### Restart Alfred Container

Restart the entire alfred container (stops and restarts all workers):

```bash
curl -X POST https://app.alfred.black/api/v1/workers/restart \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

```json
{
  "message": "Alfred container restarted"
}
```

<Warning>
Restarting the Alfred container stops all daemons (Curator, Janitor, Distiller). They will auto-restart but lose in-progress work.
</Warning>

## Curator Operations

Curator watches the inbox and processes raw inputs into structured vault records.

### Process Inbox

Trigger immediate processing:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/process \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

You can optionally pass parameters:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/process \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"limit": 5, "dry_run": true, "jobs": 2}'
```

### Ingest Command

Split bulk conversations into inbox files:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/ingest \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

You can optionally run a dry-run:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/ingest \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dry_run": true}'
```

## Janitor Operations

Janitor scans the vault for structural issues (broken links, invalid frontmatter, orphans) and fixes them.

### Scan for Issues

Run a vault scan without fixing:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/janitor/scan \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

### Fix Issues

Run Janitor to fix detected issues:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/janitor/fix \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

<Warning>
Janitor can modify or delete records. Review scan results before running fix.
</Warning>

### Janitor History

Get recent scan/fix runs:

```bash
curl -X GET "https://app.alfred.black/api/v1/workers/janitor/history?limit=5" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

### Ignore a File

Tell Janitor to ignore a specific file in future scans:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/janitor/ignore \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "file": "note/Random thought.md",
    "reason": "Intentionally orphaned"
  }'
```

**Response:**

```json
{
  "message": "File ignored"
}
```

## Distiller Operations

Distiller extracts latent knowledge (assumptions, decisions, constraints) from operational records.

### Scan for Candidates

Scan for records that may contain extractable knowledge:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/distiller/scan \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

## Error Handling

All errors follow a consistent format:

```json
{
  "error": {
    "code": "EXEC_ERROR",
    "message": "alfred failed (1): Error processing inbox",
    "details": {
      "stderr": "Error: no files to process"
    }
  }
}
```

Error codes: `UNAUTHORIZED` (401), `VALIDATION_ERROR` (400), `NOT_FOUND` (404), `EXEC_ERROR` (500), `INTERNAL_ERROR` (500).

Run all workers in sequence:

## Troubleshooting

### Worker Stuck

If workers seem stuck (status shows running but no recent activity), restart the Alfred container:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/restart \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### Worker Crashed

If the worker hit the retry limit (5 crashes), it won't auto-restart. Start workers manually:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/up \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### Slow Processing

Curator/Distiller can be slow for large inputs (60+ seconds). This is normal for complex extractions. Check history to see average duration:

```bash
curl -X GET "https://app.alfred.black/api/v1/workers/janitor/history?limit=20" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### No Issues Found

If Janitor scan returns no issues but you see problems manually:

- Janitor only detects specific issue types (broken links, invalid frontmatter, orphans)
- Custom validation rules require code changes
- Check Janitor logs for scan errors
