## What there is to monitor

Sir's tenant is a Docker stack of seven containers (`init`, `ctrl-api`, `temporal`, `openclaw`, `openclaw-workers`, `alfred`, `alfred-learn`, `ollama`), a Temporal workflow engine running ~12 distinct workflow types, the openclaw AI gateway with its agent sessions, and four background vault specialists. Most days you won't think about any of this ã Alfred just runs. But when something feels off, here's how to look.

---

## The one-shot health check

Start here:

```bash
curl -s "$ALFRED_API_BASE/api/v1/admin/dashboard" \
  -H "Authorization: Bearer $ALFRED_API_KEY" \
  | jq
```

Returns containers + service health + vault stats in one shot. If everything looks green, you're done. If something's wrong, drill in.

---

## Containers

| Endpoint | What it does |
|---|---|
| `GET /api/v1/admin/containers` | List every container with state (running, exited, restarting) and uptime |
| `GET /api/v1/admin/containers/:service/logs?tail=100` | Last N log lines from a container |
| `POST /api/v1/admin/containers/:service/restart` | Restart a container (requires `X-Confirm-Self-Restart: yes` header for `openclaw` or `ctrl-api`) |
| `POST /api/v1/admin/containers/:service/stop` | Stop a container |
| `POST /api/v1/admin/containers/:service/start` | Start a stopped container |

The dashboard's Logs page wraps the same endpoints. You can filter by service, set the tail count, and watch logs without leaving the browser.

The self-restart guardrail (the `X-Confirm-Self-Restart` header) prevents an agent from accidentally killing the container holding its own session. Without the header, restarting `openclaw` or `ctrl-api` returns 409.

## Workflows (Temporal)

Every scheduled job and every on-demand task runs as a Temporal workflow. To inspect:

```bash
# List recent workflows
curl -s "$ALFRED_API_BASE/api/v1/workflows" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq

# Inspect a specific workflow ã activities, results, failures
curl -s "$ALFRED_API_BASE/api/v1/workflows/<workflow-id>" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq
```

The dashboard's Workflows tab is the rich version ã drill into activity history, see retry attempts, view inputs and outputs.

The workflows you'll see most often:

| Workflow | When it runs |
|---|---|
| `EventProcessorWorkflow` | Every 2 min ã fetches stream events, drops to inbox, marks processed |
| `SessionTrackerWorkflow` | Every 5 min ã tracks live agent sessions |
| `LearningWorkflow` | Every 5 min ã extracts observations |
| `JudgmentWorkflow` | Every 2 min ã routes new inputs through instincts |
| `HourlyEnrichmentWorkflow` | Every hour ã one Clerk call enriches all pending records |
| `ReflectionWorkflow` | Daily 2am ã synthesises observations into instincts |
| `DailyDigestWorkflow` | Daily 6pm ã assembles the daily digest |
| `MediaIngestionWorkflow` | Per binary upload ã Whisper / OCR / extraction |
| `StreamPullerWorkflow` | Per stream, on its schedule ã fetches new events |
| `chore-*` | Per chore on its cron ã Sir's generated workflows |

## Schedules

Every periodic job is a Temporal schedule. List, inspect, and control them:

```bash
# List schedules
curl -s "$ALFRED_API_BASE/api/v1/schedules" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq

# Trigger out of cycle
curl -s -X POST "$ALFRED_API_BASE/api/v1/schedules/al-hourly-enrichment/trigger" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq

# Pause / unpause
curl -s -X POST "$ALFRED_API_BASE/api/v1/schedules/<id>/pause" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq
```

Schedules use predictable IDs: `al-*` for system schedules, `chore-*` for generated chores. The Workflows tab shows live schedule status alongside the workflows it's spawning.

## Workers (vault specialists)

The four vault specialists (Curator, Janitor, Distiller, Surveyor) report status independently:

```bash
curl -s "$ALFRED_API_BASE/api/v1/workers/status" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq
```

Restart a stuck worker:

```bash
curl -s -X POST "$ALFRED_API_BASE/api/v1/workers/restart" \
  -H "Authorization: Bearer $ALFRED_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"worker": "curator"}' | jq
```

The dashboard's Worker Activity Charts (under the Vault Nebula on the homepage) show the 24h rolling activity of each specialist.

## System info

CPU, memory, disk, uptime:

```bash
curl -s "$ALFRED_API_BASE/api/v1/admin/system/info" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq
```

Disk usage matters ã Sir's vault grows steadily, and the streams jsonl files retain 7 days by default. If you see disk pressure, prune old stream events: `POST /api/v1/streams/prune` with `{retention_days: 3}` to be more aggressive.

## Recent activity

The Activity tab in the Intelligence menu shows everything that has happened ã workflow runs, agent sessions, API calls, chore executions, container events. Filter by:

- Service (ctrl-api, openclaw, alfred-learn, alfred, temporal)
- Time window (last hour, last 24h, last 7 days)
- Workflow type
- Source (web, slack, voice, sms, email, schedule)

Equivalent API: `GET /api/v1/admin/activity`.

## What "broken" tends to look like

| Symptom | Most likely cause | First check |
|---|---|---|
| Dashboard shows the reconfiguring banner for >2 min | Composio gateway restart stuck | `GET /api/v1/openclaw/ready` |
| New stream events not appearing in vault | EventProcessor or HourlyEnrichment workflow stuck | Workflows tab ã check last EventProcessor run |
| Chore "ran" but produced nothing | Chore in dry-run quarantine (first 3 runs) | Chores tab ã quarantine status |
| Voice call connects but agent says nothing | Voice Bridge primer fetch failed | `GET /api/v1/phone/voice-context` |
| SMS reply takes >30s | Cold openclaw session bootstrap | Wait ã first SMS each session takes ~15-30s, subsequent are fast |
| Email send returns 4xx | AgentMail not configured / quota issue | `GET /api/v1/email/status` |
| Slack DMs go unanswered | Openclaw container exited | Containers list ã openclaw status |
| Connected app stops syncing | OAuth token revoked upstream | Apps page ã reconnect |

## Audit logs

Every API call hits the activity log. For longer-term audit (who did what, when), the system-openclaw-sessions stream captures every main-agent session with its summary, the channel it came from (slack, web, email, voice, sms), and the actions taken. Read it like any other stream:

```bash
curl -s "$ALFRED_API_BASE/api/v1/streams/events?stream_id=system-openclaw-sessions" \
  -H "Authorization: Bearer $ALFRED_API_KEY" | jq
```
