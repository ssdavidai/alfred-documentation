## What's in the vault

Sir's vault is a structured library of every fact, event, person, decision, and idea Alfred has ever recorded. Records are stored as Markdown files with YAML frontmatter, organised by type, on the LUKS-encrypted volume of your dedicated VPS. The dashboard renders them; the agent reads and writes them; you can also query them directly through the API.

There are around 20 record types. The most important:

| Type | What it represents |
|---|---|
| `matter` | An ongoing concern — a project, a relationship, a responsibility you're carrying |
| `task` | A discrete actionable item (errand) — pending, in-progress, done |
| `chore` | A recurring scheduled workflow definition |
| `note` | Free-form text — meeting notes, research, observations |
| `person` | Someone Sir interacts with |
| `org` | An organisation (employer, client, vendor) |
| `event` | Something that happened — emails, calendar events, voice calls, GitHub notifications, slack messages |
| `conversation` | Multi-speaker dialogue — meetings, voice calls, Omi recordings |
| `decision` | A choice that was made, with rationale |
| `assumption` | Something believed but not proven |
| `constraint` | A limit on what's possible (budget, timeline, capacity) |
| `synthesis` | A higher-order conclusion derived from multiple records |
| `contradiction` | Two records that disagree, surfaced for resolution |
| `reflection` | A nightly synthesis of recent activity |
| `observation` | A signal the learning system captured (instruction, correction, confirmation) |
| `instinct` | A learned routing rule |
| `project` | A specific initiative within a matter |
| `idea` | A captured thought |
| `location` | A place |
| `account` / `asset` | Financial records |

## Three ways to browse

<Tabs>
  <Tab title="Vault Nebula (homepage)">
    The Vault Nebula is the dashboard's entry point — a real-time visualisation of your knowledge graph. Each cloud is a Surveyor-built cluster of related records. Hover to see what's inside, click to drill into individual records.

    The Nebula is the right starting point when Sir wants to see "what is Alfred currently tracking" at a glance, without knowing which specific records to query.
  </Tab>

  <Tab title="Intelligence menu (typed views)">
    The five tabs of the Intelligence menu — Inbox, Matters, Errands, Chores, Activity — give you typed views with proper filtering and sort. This is where Sir lives day-to-day:

    - Matters tab → all `matter/` records, status filter, full body in the detail panel
    - Errands tab → all `task/` records, status + owner filter
    - Chores tab → all `chore/` records, with source + dependency audit
    - Inbox → events Alfred hasn't fully classified yet
    - Activity → workflow runs, agent sessions, API activity
  </Tab>

  <Tab title="Vault Browser (everything else)">
    For record types not covered by the Intelligence tabs (`person`, `org`, `event`, `decision`, `assumption`, `constraint`, `synthesis`, `contradiction`, `reflection`, `observation`, `instinct`, …) the Vault Browser is the universal interface. Filter by type, search by name or body, view the full record with its frontmatter and connected records.
  </Tab>
</Tabs>

## Searching

Two search surfaces:

- **Cmd+K (or Ctrl+K)** — global fuzzy search across all records, by name and body. Best for quick lookups: "find that note about the kitchen renovation."
- **Vault search via API** — `GET /api/v1/vault/search?q=...` for full-text search. Optional `&type=matter` filters to a record type. The agent uses this constantly through `self({endpoint: "/api/v1/vault/search", query: {q: "..."}})`.

For glob-style searches by file path, the API also accepts `?glob=` (e.g. `?glob=matter/*acme*`). For grep-style content search, `?grep=` runs against record bodies.

## Reading a single record

Click any record in the dashboard or call `GET /api/v1/vault/records/{path}` (where `path` is vault-relative, e.g. `matter/kitchen-renovation.md`). The detail view shows:

- The full body — Markdown rendered, with internal `[[Wiki Links]]` resolving to other vault records
- The frontmatter — type, status, dates, custom fields
- The relationships — every other record that links to this one (back-references)
- A timeline — recent edits, related events, related sessions

For matters specifically, the body preview goes up to 3000 characters by default, so the story is visible without expanding.

## The Surveyor's clusters

The Surveyor (one of your specialists) periodically:

1. Embeds every vault record using a local Ollama-hosted `nomic-embed-text` model
2. Clusters by semantic similarity using Milvus
3. Labels each cluster with a generated heading (via `x-ai/grok-4.1-fast`)
4. Discovers cross-cluster relationships and writes them as vault `synthesis/` records

Embeddings happen locally — no embedding cost goes to OpenRouter. The Surveyor's index lives at `/mnt/encrypted/alfred/milvus_lite.db`.

When Sir adds a flood of new records (a big upload, a new Connected App with backfill), the Surveyor re-runs and the Nebula updates within a few minutes.

---

## Direct file access (advanced)

The vault is just Markdown files on disk at `/mnt/encrypted/vault/<type>/<slug>.md`. If Sir SSHes into the tenant (with the deploy key) you can `ls` and `cat` records directly. **Don't write to the filesystem this way** — bypass the ctrl-api and you bypass the Janitor's structural checks, the entity resolution, and the back-reference index. Use the API or the dashboard instead. Reading is fine.
