The inbox workflow is Alfred's primary ingestion pipeline. Drop raw content into the inbox, and Curator automatically processes it into structured vault records.

## Overview

The inbox workflow has three stages:

<Steps>
  <Step title="Upload">
    Upload files or text snippets to the inbox via API
  </Step>
  <Step title="Processing">
    Curator watches the inbox and processes new content automatically
  </Step>
  <Step title="Verification">
    Check the vault for newly created structured records
  </Step>
</Steps>

## Supported Formats

Curator accepts the following file types:

- **Text**: `.txt`, `.md`, `.mdx`
- **Documents**: `.pdf`, `.docx`, `.odt`
- **Web**: `.html`, `.mhtml`
- **Code**: `.py`, `.js`, `.ts`, `.json`, `.yaml`
- **Data**: `.csv`, `.tsv`

All filenames are sanitized automatically (special characters like `/\:*?"<>|` are replaced with underscores).

## Single File Upload

Upload a single file to the inbox by sending JSON with the filename and content:

```bash
curl -X POST https://app.alfred.black/api/v1/vault/inbox \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "filename": "meeting-notes.md",
    "content": "# Weekly Standup 2026-02-26\n\nAttendees: Alice, Bob\n\n## Action Items\n- Migrate database by Q2\n- Hire 2 engineers"
  }'
```

**Response:**

```json
{
  "message": "Uploaded meeting-notes.md to inbox",
  "filename": "meeting-notes.md"
}
```

<Tip>
The API automatically sanitizes filenames. `Meeting Notes (2026).md` becomes `Meeting_Notes__2026_.md`.
</Tip>

## Bulk Upload

Upload multiple files at once:

```bash
curl -X POST https://app.alfred.black/api/v1/vault/inbox/bulk \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "files": [
      {"filename": "notes1.md", "content": "First set of notes..."},
      {"filename": "notes2.md", "content": "Second set of notes..."},
      {"filename": "transcript.txt", "content": "Meeting transcript..."}
    ]
  }'
```

**Response:**

```json
{
  "message": "Uploaded 3 files to inbox",
  "filenames": ["notes1.md", "notes2.md", "transcript.txt"]
}
```

## Listing Inbox Contents

Check what's in the inbox:

```bash
curl -X GET https://app.alfred.black/api/v1/vault/inbox \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

```json
{
  "files": [
    "meeting-notes.md",
    "transcript.txt"
  ]
}
```

<Note>
The inbox listing returns filenames only (hidden files starting with `.` are excluded). It does not include file sizes or modification times.
</Note>

## Automatic Processing

Curator watches the inbox directory and processes new files automatically (default interval: 60 seconds).

To check worker status:

```bash
curl -X GET https://app.alfred.black/api/v1/workers/status \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

The workers status endpoint always returns raw text from the `alfred status` command:

```json
{
  "raw": "curator: running (pid 1234)\njanitor: running (pid 1235)\ndistiller: running (pid 1236)"
}
```

## Manual Processing

Trigger immediate processing without waiting for the next sweep:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/process \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

You can optionally pass parameters in the body:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/process \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "limit": 5,
    "dry_run": true
  }'
```

<Warning>
Manual processing runs synchronously. For complex inputs, it may exceed the 15-second API timeout and return a `504` response. If this happens, the operation is still running — check worker status or inbox contents to confirm completion.
</Warning>

## Checking Results

After Curator processes inbox files, check the vault for new records:

```bash
curl -X GET https://app.alfred.black/api/v1/vault/list/person \
  -H "Authorization: Bearer YOUR_API_KEY"
```

**Response:**

If the underlying CLI returns valid JSON, it is passed through directly. Otherwise, the output is returned as `{ "raw": "<text>" }`.

## Example Workflows

### Meeting Notes

<Steps>
  <Step title="Record the meeting">
    Take notes in Markdown or export from Zoom/Meet
  </Step>
  <Step title="Upload to inbox">
    ```bash
    curl -X POST https://app.alfred.black/api/v1/vault/inbox \
      -H "Authorization: Bearer YOUR_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"filename": "weekly-standup-2026-02-26.md", "content": "..."}'
    ```
  </Step>
  <Step title="Wait for processing">
    Curator extracts attendees as `person` records, action items as `task` records, and decisions as `decision` records
  </Step>
  <Step title="Review results">
    ```bash
    curl -X GET https://app.alfred.black/api/v1/vault/list/task \
      -H "Authorization: Bearer YOUR_API_KEY"
    ```
  </Step>
</Steps>

### Conversation Dump

Upload a chat transcript:

```bash
curl -X POST https://app.alfred.black/api/v1/vault/inbox \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "filename": "db-migration-chat.md",
    "content": "Alice: We need to migrate the database by Q2.\nBob: Agreed. Let'\''s use Postgres.\nAlice: I'\''ll create the migration plan."
  }'
```

Curator extracts:
- **Decision**: "Migrate database to Postgres by Q2"
- **Task**: "Create database migration plan" (assigned to Alice)
- **Person**: "Alice", "Bob" (if not already in vault)

### Document Batch Processing

Upload multiple files from a project:

```bash
curl -X POST https://app.alfred.black/api/v1/vault/inbox/bulk \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "files": [
      {"filename": "requirements.md", "content": "..."},
      {"filename": "architecture.md", "content": "..."},
      {"filename": "timeline.md", "content": "..."}
    ]
  }'
```

Curator processes all documents and extracts relevant entities (projects, decisions, constraints, people).

## Error Handling

All errors follow a consistent format:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "filename and content are required"
  }
}
```

Error codes: `UNAUTHORIZED` (401), `VALIDATION_ERROR` (400), `NOT_FOUND` (404), `EXEC_ERROR` (500), `INTERNAL_ERROR` (500).

## Troubleshooting

### File Not Processing

Check worker status:

```bash
curl -X GET https://app.alfred.black/api/v1/workers/status \
  -H "Authorization: Bearer YOUR_API_KEY"
```

If workers are stopped, start them:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/up \
  -H "Authorization: Bearer YOUR_API_KEY"
```

### File Rejected

Common issues:
- **Unsupported format**: Only text-based formats are supported
- **Empty file**: Files must contain content
- **Encoding issues**: Ensure UTF-8 encoding

### Duplicate Records

Curator uses content hashing to avoid re-processing the same file. If you upload the same content twice, it will be skipped. To force re-processing, restart the workers:

```bash
curl -X POST https://app.alfred.black/api/v1/workers/restart \
  -H "Authorization: Bearer YOUR_API_KEY"
```
