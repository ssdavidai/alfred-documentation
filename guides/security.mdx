## The short version

| Layer | How it's protected |
|---|---|
| **Infrastructure** | Dedicated `cx53` Hetzner VPS in the EU per tenant. No shared servers, no multi-tenancy. |
| **Data at rest** | LUKS2 encryption on the data volume with a per-instance auto-generated passphrase |
| **Network ingress** | Hetzner firewall allows only SSH (22) + ICMP. All Alfred services bind to localhost. |
| **External access** | Tailscale tailnet (WireGuard) and Cloudflare Tunnel only. No public IP exposure of services. |
| **API auth** | Per-account Bearer keys (`alf_…`), SHA-256 hashed at rest, AES-256-GCM encrypted internal credentials |
| **Third-party credentials** | Composio holds OAuth tokens server-side. Sir's tenant only stores connection IDs. |
| **Per-tenant isolation** | `COMPOSIO_USER_ID` enforced server-side; explicit ownership verification on every connection lookup |
| **Audit** | Every API call logged. Every agent session captured to a queryable stream. |
| **Data residency** | EU only — Hetzner Germany or Finland depending on availability at provision time |

---

## Infrastructure isolation

Every paying subscriber gets a dedicated Hetzner Cloud `cx53` VPS. Not a container, not a shared instance — a dedicated VM. There is no shared compute, no shared storage, no shared database between tenants. The only shared infrastructure is:

- The SaaS host at `alfred.black` (which proxies but does not store vault data)
- The shared AgentMail pod (which holds inbound mail briefly before it's dispatched to the tenant — keys are inbox-scoped per tenant)
- The shared Twilio account at the SaaS layer (with per-tenant subnumbers)
- The shared Composio API key (with per-tenant `user_id` for isolation)

Each tenant's dedicated VM is provisioned in the EU — Hetzner Germany or Finland depending on availability at provision time.

## Encryption at rest

The data volume on every tenant is encrypted with **LUKS2** (Linux Unified Key Setup, version 2). The passphrase is generated per-instance at provisioning time and never reused.

- Vault records (`/mnt/encrypted/vault`)
- Stream events and configs (`/mnt/encrypted/alfred/streams`)
- OpenClaw config and workspace (`/mnt/encrypted/openclaw`)
- Workers config (`/mnt/encrypted/openclaw-workers`)
- Temporal SQLite database (`/mnt/encrypted/temporal`)
- AgentMail credentials (`/mnt/encrypted/alfred/.agentmail-credentials.json`)
- Composio user ID fallback (`/mnt/encrypted/alfred/.composio-user-id`)
- Authorised senders list (`/vault/.auth/authorized_senders.json`)
- Authorised phone numbers (`/mnt/encrypted/alfred/.authorized-phone-numbers.json`)

All of it lives on the encrypted volume. At rest, the volume content is unreadable without the LUKS passphrase. The passphrase is uploaded post-provision via SSH (never via Hetzner cloud-init `user_data`, which is readable through the metadata API), held in memory by `systemd` for the volume's lifetime, and backed up encrypted to S3 for disaster recovery.

## Per-tenant Vaultwarden

Every tenant runs its own **Vaultwarden** — a self-hosted, Bitwarden-compatible password manager — on the VPS. It's where Sir's ~45 tenant secrets live: third-party API keys, admin credentials, OAuth tokens for accounts Alfred needs to act on, and anything else that would otherwise be loose in scripts or `.env` files. The Vaultwarden database sits on `/mnt/encrypted/vaultwarden`, so the entire vault inherits LUKS2 protection at the volume layer **and** Bitwarden's per-record AES-256 encryption derived from Sir's master password on top.

The master password lives in the tenant `.env` as `BW_PASSWORD` (it's also a service credential used internally by the `vault-init` sidecar to unlock the CLI on boot). It's surfaced in the dashboard at **Settings → Claude Setup → Vault Login**, with click-to-reveal — Sir copies it to his personal password manager once and never again. Don't rotate it via Vaultwarden's "Change Master Password" dialog; that would desync the service-credential side and break boot.

When Sir asks Alfred to do something that needs a stored credential — "log into the broker and pull this month's statement", "rotate the Stripe webhook secret" — the main agent reads from Vaultwarden through the ctrl-api `/api/v1/vaultwarden/*` routes (list, search, get, create, update, delete on login items; folders are read/write; collections and orgs are read-only and managed in the web UI). Reads return passwords and TOTPs in the response envelope; writes go through the same authenticated bearer surface as the rest of the admin API.

Cross-tenant boundary: Vaultwarden runs **inside** the tenant's Docker stack, bound to localhost, encrypted at rest under the tenant's own LUKS volume. There is no shared Vaultwarden, no shared org, no shared collection. Peers cannot reach into each other's vault — vault data is per-tenant in the same hard sense as the rest of the dedicated-VPS model. See [the full Vaultwarden guide](/guides/vaultwarden) for usage details.

## Network model

Inbound to Sir's tenant:

| Port | Service | Who can reach it |
|---|---|---|
| **22** | SSH | Anyone — but only the per-instance ed25519 deploy key is authorised |
| **ICMP** | ping | Anyone — for liveness checks only |
| **3100** | ctrl-api | Localhost only. Reachable externally **only** through Tailscale or Cloudflare Tunnel. |
| **18789** | openclaw gateway | Localhost only. |
| **18790** | openclaw-workers gateway | Localhost only. |
| **7233** | Temporal gRPC | Localhost only. |
| **8233** | Temporal UI | Localhost only. |
| **11434** | Ollama (embeddings) | Localhost only. |

The Hetzner Cloud Firewall enforces this at the network edge — port 3100 is not reachable from the public internet, full stop. The only public-internet-accessible Alfred surface is `https://alfred.black`, which proxies requests through Tailscale (authenticated) to your tenant.

## Agent sandboxing

Every Alfred agent's tool surface is controlled at multiple layers:

1. **`gateway.tools.allow`** — the openclaw config explicitly lists every tool the main agent (and every other agent) can invoke. New tools must be added; nothing is implicit.
2. **MCP `self`** — the agent's primary ctrl-api access is one tool, not 41 prefixed tools. The endpoint catalogue is documented in the workspace `TOOLS.md`. The agent can only hit endpoints ctrl-api exposes; ctrl-api cannot reach beyond the tenant.
3. **Composio gateway** — third-party app actions go through `composio_execute`. Composio holds credentials; Sir's tenant only holds connection IDs.
4. **Ephemeral subagents** — the TaskRunner spawns short-lived agents per task with **scoped tool access**. A subagent that only needs Slack can't read your email.
5. **Workspace shell scope** — `bash`, `read`, `write`, `edit`, `find` are scoped to `/home/node/.openclaw/workspace`. The agent can't `cat /etc/passwd`.
6. **Self-restart guardrail** — restarting `openclaw` or `ctrl-api` requires `X-Confirm-Self-Restart: yes`, preventing the agent from accidentally killing its own session.
7. **Confirm-before-destruct** — the workspace skills explicitly require Sir's confirmation before delete, terminate, cancel, or live-chore-trigger operations.

## Per-tenant Composio isolation

Composio is the most security-critical shared dependency. Every tenant has a unique `COMPOSIO_USER_ID` (typically `alfred-<slug>-<instance-id>`), enforced server-side:

- ctrl-api refuses to start if `COMPOSIO_USER_ID` is unset or `"default"`
- Every Composio call passes `user_id` to filter results
- Every connection lookup by ID **verifies ownership** — Composio's server-side filter has been observed to silently return cross-tenant accounts, so ctrl-api re-checks ownership on every read
- Connection IDs that don't belong to the tenant return 404 (not 403, to prevent enumeration)

This invariant is non-negotiable. If you ever see another tenant's connected app on your Apps page, that's a bug — file a ticket immediately.

---

## Per-tenant AgentMail isolation

The shared AgentMail pod is the inbound delivery layer. Isolation runs through **inbox-scoped API keys**:

- One master AgentMail API key lives only on the SaaS host (in env + macOS Keychain). Never on tenants.
- Each tenant gets its own inbox-scoped key at provision time, derived from the master.
- The tenant's key can read/send for its inbox only — it can't enumerate, read, or modify any other tenant's inbox.
- Webhook delivery from AgentMail to the SaaS uses Svix signature verification with a shared secret.

Sir's tenant stores its inbox-scoped key at `/mnt/encrypted/alfred/.agentmail-credentials.json` and in the tenant `.env`. Both locations are on the LUKS-encrypted volume.

## Data residency

All Alfred Black tenants run on Hetzner Cloud in the EU — Germany (Falkenstein, Nuremberg, or Helsinki area) depending on which region had capacity at provision time. The SaaS host at `alfred.black` is also in Germany.

Sir's vault, his streams, his email metadata, his transcripts — all of it lives within the EU. AgentMail's inbox storage is also EU-hosted; Composio's storage region depends on the upstream third-party (Gmail = Google's region, Notion = Notion's region, etc.). The Twilio account that powers AgentPhone is US-located but operates as a passthrough — voice and SMS content flows through Twilio briefly before reaching Sir's tenant.

GDPR rights (access, deletion, portability) are honoured. Cancel within the 30-day grace period and your data is preserved; cancel outside it and the data is destroyed in full.

## Audit logging

Every authenticated API call hits the activity log on Sir's tenant. The log is queryable through `GET /api/v1/admin/activity` and rendered in the dashboard's Activity tab. For longer-term audit, the `system-openclaw-sessions` stream captures every main-agent session (every Slack DM, every web chat, every voice call, every email reply, every SMS) with its summary, channel, and tool calls.

This is a forensic surface as much as a memory primer. If Sir wants to know what Alfred did between 9pm and midnight last Thursday, it's all there.
