## What pairing does

Pairing a device gives it a token to talk to Sir's openclaw gateway. Once paired, the device can stream data in (audio chunks from an Omi, a mobile app's clipboard, a desktop's clipboard) and the agent can address it (push a notification, request a transcription).

Each device has:

- A **device ID** — generated at pair time, persistent
- A **role** — what the device is allowed to do (e.g. `omi`, `mobile`, `desktop`, `cli`)
- An optional **scope** — narrowing the role's permissions
- A **token** — the bearer credential the device uses

Tokens can be rotated without re-pairing. Devices can be revoked individually or in bulk.

## Pair an Omi wearable

The Omi is the canonical pairing case. The Omi is a physical wearable that records ambient audio and ships chunks to Alfred for transcription, speech detection, and (where the audio captures multi-topic dialogue) braindump extraction. Audio is held for 48 hours then expires.

<Steps>
  <Step title="Open Devices in the dashboard">
    Click **Devices** in the sidebar. You'll see currently paired devices with their roles and last-seen timestamps.
  </Step>

  <Step title="Click Add Device → Omi">
    The dashboard shows a one-time pairing code. Codes are short-lived — get the code into the Omi app within a few minutes.
  </Step>

  <Step title="Enter the code in the Omi app">
    On Sir's phone, in the Omi companion app, choose "Pair with Alfred Black" and type the code. The Omi posts the device-request to Sir's tenant via the openclaw gateway.
  </Step>

  <Step title="Approve the request">
    The dashboard's Devices page shows the pending request. Approve it to issue the token.

    Equivalent API call:

    ```bash
    curl https://alfred.black/api/v1/devices/<requestId>/approve \
      -X POST \
      -H "Authorization: Bearer alf_..."
    ```

    Add `--data '{"latest": true}'` to approve the most recent pending request without specifying its ID.
  </Step>

  <Step title="Verify it's streaming">
    Once approved, the Omi starts shipping audio. Within a couple of minutes you should see `omi-audio` events in the Streams tab and `conversation/` records in the vault as transcripts arrive (Groq Whisper turns each chunk around in roughly 3 seconds).
  </Step>
</Steps>

## Pair other devices

The same flow works for any openclaw client. Available roles depend on what your tenant has registered:

- `mobile` — a phone client app
- `desktop` — Sir's laptop
- `cli` — a command-line tool that needs to authenticate as a device rather than as Sir

Each pairing follows the same pattern: client requests pairing → request appears in Sir's dashboard → Sir approves → token issued.

The dashboard's Devices page lets you:

- **List** all paired devices and pending requests
- **Approve** a pending request
- **Reject** a pending request
- **Remove** a device (revokes its token, the device must re-pair)
- **Rotate** a device's token (issues a new one, invalidates the old)
- **Revoke** specific role access (e.g. drop `mobile` from a desktop client)
- **Clear** all devices, or only pending requests

---

## Token rotation

If a device's token is suspected of being compromised — phone lost, laptop stolen, log file leaked — rotate it from the dashboard or the API:

```bash
curl https://alfred.black/api/v1/devices/<deviceId>/rotate \
  -X POST \
  -H "Authorization: Bearer alf_..." \
  -H "Content-Type: application/json" \
  -d '{"role": "omi"}'
```

The old token is invalidated immediately. Provide the new token to the device through whatever pairing UI it offers.

---

## Revocation

Revoking a device cuts it off from Sir's tenant. Existing data the device has already sent (Omi recordings, mobile uploads) stays in the vault — revocation only affects future communication.

```bash
curl https://alfred.black/api/v1/devices/<deviceId> \
  -X DELETE \
  -H "Authorization: Bearer alf_..."
```

For role-specific revocation (drop one role but keep others):

```bash
curl https://alfred.black/api/v1/devices/<deviceId>/revoke \
  -X POST \
  -H "Authorization: Bearer alf_..." \
  -H "Content-Type: application/json" \
  -d '{"role": "mobile"}'
```

---

## When pairing fails

The most common failure mode is an expired pairing code — codes are short-lived for security. If pairing times out, generate a fresh code and retry.

If the device can reach `alfred.black` but the pairing request never appears in your dashboard, check:

- Sir is logged into the right account (the device pairs to the user, not to a specific browser session)
- The instance is `running` — pairing requires the openclaw gateway to be up
