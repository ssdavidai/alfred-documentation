## What "intuition" means

Intuition is Alfred's ability to act without asking. Day one Alfred asks about everything because he knows nothing. Day thirty Alfred handles routine decisions silently because he's seen the same pattern enough times to be confident. The mechanism that gets him from one to the other is the intuition system: observations, instincts, judgment, and the trust gradient.

This is not a feature you turn on. It's always running.

---

## The loop

```
Sir does something  →  Observation captured  →  Reflection synthesises observations into instincts
       ▲                                                            │
       │                                                            ▼
       └─── Judgment fires the instinct on the next matching input ←
```

Four moving parts:

| Component | What it does | When it runs |
|---|---|---|
| **Observation** | Captures a single signal — a routing decision, an instruction, a correction, a confirmation | Continuously, on every signal |
| **Reflection** | Reads accumulated observations and proposes new instincts (or strengthens existing ones) | Nightly at 2am |
| **Instinct** | A learned routing rule with input patterns, a destination, a confidence score, and a discretion threshold | Persistent in the vault |
| **Judgment** | On every new input, scores it against all instincts and decides whether to act autonomously or escalate | Every 2 minutes (and per-input on demand) |

---

## Three kinds of observation

The observation hook captures three categories of user signal — beyond the routing decisions Alfred makes on his own:

### 1. Instruction

Sir teaches Alfred a new rule. "From now on, anything from accounts@acme.com goes straight to the Acme matter." The observation captures the trigger condition and the expected action.

### 2. Correction

Sir corrects a previous decision. "That email about the Q3 contract should have gone to the legal matter, not the general inbox." The observation captures what was wrong, what should have happened, and the assistant's prior context.

### 3. Confirmation

Sir validates Alfred's behaviour. "Good — keep doing that for newsletters from substack senders." The observation strengthens the pattern that was validated.

Each entry includes `assistant_context` — what Alfred said or did just before — so the Reflection has the full picture when synthesising the rule.

## What an instinct looks like

Here's the structure of an instinct record (the YAML frontmatter — the body adds the routing logic in prose):

```yaml
type: instinct
name: route-noise-to-log
status: active
description: "Auto-route noise-tier senders to stream log"
input_patterns:
  sender_domains: [marketing.acme.com, newsletter.foo.com, ...]
  subject_keywords: []
  attachment_types: []
  input_types: [email]
routing_rule:
  destination_type: stream
  destination: "tier2 stream-log"
  process: archive
  default_assignee: ""
confidence_score: 0.9
observation_count: 47
discretion_threshold: 0.85
matching_weights:
  domain: 0.30
  keywords: 0.30
  input_type: 0.15
  attachment: 0.15
  tags: 0.10
```

When a new email arrives, the JudgmentWorkflow:

1. Loads every active instinct
2. Extracts metadata from the input (sender domain, subject keywords, attachments, type)
3. Scores each instinct against the input using the matching_weights
4. Picks the highest-scoring match
5. Compares the score to the instinct's `discretion_threshold`
6. If the score wins, Alfred routes autonomously. If it loses, Alfred escalates to Sir.

## The trust gradient

The discretion threshold is not fixed. It scales with evidence — the more observations an instinct has accumulated, the lower the threshold (the more autonomous Alfred becomes).

| Observations | Discretion threshold | Butler equivalent |
|---|---|---|
| **< 5** | 0.95 | "I've barely seen this before, sir. Your guidance?" |
| **5–9** | 0.90 | "I believe I know, but I'd rather confirm." |
| **10–19** | 0.85 | "I'm fairly certain this goes here." |
| **20–49** | 0.80 | "I've seen this many times. Handling it." |
| **50+** | 0.75 | "This is routine. Already done." |

A brand-new instinct (created from a single instruction) starts at 0.95 — Alfred almost always asks. As the same pattern accumulates more observations across multiple incidents, the threshold relaxes. By 50 observations, Alfred handles it silently.

The instinct can override the gradient by setting an explicit `discretion_threshold`. The onboarding instinct pack sets cautious defaults (0.85–0.95 depending on type) and lets the gradient relax them naturally over time.

---

## Approval flow for execution tasks

For instincts that don't just route inputs but **execute** something on Sir's behalf (send a message, schedule a meeting, update a Notion page), the trust gradient is reinforced by an explicit approval gate:

- New execution instincts ship with `requires_approval: true`
- The TaskRunner workflow checks `task.approved == true` before running
- Approve / Reject buttons in the dashboard set the flag directly on the task record
- Once the instinct accumulates enough observations and clean executions, Reflection can flip `requires_approval: false` and the trust gradient takes over

This is deliberate friction. New autonomous execution requires Sir to explicitly green-light the first instances; once enough have run cleanly, the gate relaxes.

## Resetting an instinct

If Alfred's learned the wrong thing — autonomously routing emails Sir wanted to triage himself — open the instinct record and either:

- **Pause it** (set `status: paused` via `PATCH /api/v1/vault/records/instinct/<slug>.md`)
- **Delete it** (Sir's vault, Sir's call) and let Reflection re-derive a better one from observations
- **Reduce its confidence** by lowering `confidence_score`, which de-prioritises it in Judgment scoring

A correction observation immediately fed in ("this should not have been auto-routed") will shape the next nightly Reflection toward fixing the rule.
