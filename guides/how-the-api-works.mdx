## The shape of a request

Every API call Sir makes from outside his tenant follows the same path:

```
Your script  →  https://alfred.black/api/...  →  SaaS proxy
                                                      │
                                                      │  Tailscale (WireGuard)
                                                      ▼
                                        Sir's tenant ctrl-api :3100
                                                      │
                                                      │  internal docker network
                                                      ▼
                                        openclaw / temporal / alfred / alfred-learn
```

Three hops, three different auth mechanisms, no public IP exposure of Sir's tenant.

---

## Step 1 — Bearer auth at the SaaS edge

Your request lands at `alfred.black/api/...` carrying `Authorization: Bearer alf_…`. The SaaS:

1. SHA-256 hashes the token
2. Looks up the matching `ApiKey` row by hash
3. Reads the row's `userId`
4. Looks up the user's `Instance` (their tenant)
5. Checks the instance is `running` and has both `tailscaleHostname` and `apiKey` set
6. Updates `lastUsedAt` fire-and-forget

If the key doesn't exist, the instance isn't ready, or the instance isn't running, the SaaS returns the relevant 401/503 with a generic message. Internal details about the tenant are never leaked in the error body.

---

## Step 2 — AES-256-GCM decrypt of the per-instance API key

Sir's tenant ctrl-api uses a different bearer token internally — the `AAS_API_KEY` baked into the tenant's `.env` at provision time. The SaaS holds this per-instance key encrypted with AES-256-GCM under `COLUMN_ENCRYPTION_KEY` and decrypts it on demand for each proxied request.

Sir never sees this internal token. Sir's `alf_…` key authenticates Sir to the SaaS; the SaaS has its own credentials to talk to each tenant.

---

## Step 3 — Tailscale transport

The SaaS host and every tenant share a Tailscale tailnet. The SaaS sends the upstream request to `https://<tenant-tailscale-hostname>:3100/api/v1/...` over WireGuard. Routes:

- `tailscaleHostname` if the SaaS container can reach the tenant's MagicDNS name directly
- `subdomainUrl` (a per-tenant Cloudflare Tunnel hostname) as a fallback when the SaaS container is on a Docker bridge network without Tailscale access

Either way, the tenant's port 3100 is **never** exposed to the public internet. The Hetzner firewall on every tenant allows only SSH (22) and ICMP inbound. All Alfred services bind to localhost; external access is exclusively through Tailscale or Cloudflare Tunnel.

---

## Step 4 — Tenant ctrl-api dispatches the request

ctrl-api validates the inbound `Authorization: Bearer <internal-token>` against its own `AAS_API_KEY` using a constant-time comparison. If valid, the route handler runs.

Most routes pass through to:

- The vault filesystem at `/mnt/encrypted/vault` for record CRUD
- Docker exec into containers for chore source, openclaw config, alfred CLI commands
- The Temporal client at `temporal:7233` for workflow operations
- The OpenClaw gateway at `openclaw:18789` for the AI gateway
- AgentMail's API for outbound email
- The SaaS internal Twilio endpoint for outbound SMS / voice

The response — JSON, almost always — flows back the same chain in reverse: ctrl-api → Tailscale → SaaS proxy → your script.

## Endpoints

The full endpoint catalogue is in the [API Reference](/api-reference/introduction). The table below lists the major route groups; every entry is reachable as `https://alfred.black/api/v1/<path>` from outside, or as `self({endpoint: "/api/v1/<path>"})` for the agent inside.

| Group | Purpose |
|---|---|
| `vault/*` | Read, write, search, list records by type, fetch the full graph or schema |
| `streams/*` | List streams, fetch events, ingest manually, manage configurations, prune old events |
| `learning/*` | Status, queue, observations, instincts, reflections, sessions |
| `workflows/*` | List Temporal workflows, inspect, signal, terminate, cancel |
| `schedules/*` | List, create, trigger, pause, unpause, rewrite cron |
| `chores/*` | List, fetch source + dependency audit, pause, resume, trigger, create, edit, delete |
| `workers/*` | Worker status, restart |
| `admin/*` | Dashboard, health, system info, containers (list, restart, logs), credentials, models |
| `integrations/*` | Composio: list, catalog, connect, auto-config, enable/disable streams + tools, regenerate skills |
| `email/*` | AgentMail outbound — send, reply, forward, fetch message, fetch thread, fetch attachment |
| `phone/*` | AgentPhone — config, voice-context, sms (outbound), call (outbound), authorized-numbers CRUD |
| `auth/senders` | AgentMail authorized-senders allowlist CRUD |
| `devices/*` | OpenClaw device pairing, approve, reject, remove, rotate token, revoke |
| `omi/*` | Omi wearable upload + state |
| `tools/*`, `models/*`, `agents/*` | Internal — agent/model/tool registry queries |

## Rate limits

There is no global rate limit at the SaaS edge today. The constraints you'll hit instead are:

- **Tenant request timeout** — 15 seconds per request at the SaaS proxy. Long-running operations (workflow polling, big vault searches) should be polled, not held open.
- **OpenClaw gateway restart** — connecting or disconnecting a Composio app triggers a ~40-second gateway restart. During that window, anything hitting the gateway returns 502. Use `GET /api/v1/openclaw/ready` to check readiness before retrying.
- **alfred-learn restart** — chore-driven restarts of alfred-learn are rate-limited to once every 30 seconds. Excess requests return 429 with a `retry_after_seconds` field.
- **Composio per-action limits** — Composio's upstream APIs (Gmail, Slack, …) have their own per-app rate limits. Errors returned from the third-party are surfaced verbatim.

## Error codes

| Status | Meaning |
|---|---|
| `200` | Success |
| `201` | Resource created |
| `204` | Success, no body (acks) |
| `400` | Validation error — body details what's wrong |
| `401` | Bad or missing API key, or missing `X-Confirm-Self-Restart` header on a self-restart route |
| `403` | Twilio webhook signature failed (only on inbound Twilio routes) |
| `404` | Resource not found |
| `409` | Conflict — phone integration not provisioned, etc. |
| `429` | Rate-limited |
| `500` | Internal error |
| `502` | Upstream service unavailable (openclaw gateway down, Twilio rejected, …) |
| `503` | Tenant not ready (still provisioning, suspended) |
| `504` | Tenant API timeout |

Error bodies are JSON: `{error: "<short message>"}`. The proxy sanitises 5xx from the tenant to "Internal tenant error" rather than leaking internal stack traces. Specific 4xx messages from the tenant are passed through (truncated to 200 chars).

---

## Response shapes

JSON throughout. The shape is documented per-endpoint in the [API Reference](/api-reference/introduction). Common patterns:

- List endpoints return an array directly, or `{records: [...], total: N}` for paginated views
- Single-record reads return the parsed frontmatter + body, e.g. `{name, type, status, body, ...}`
- Mutating endpoints return `{status: "..."}` or the updated resource
- Workflow-triggering endpoints return `{workflow_id, run_id}`
- Phone / email actions return provider details (`{sid, mode}` for calls, AgentMail's response body for emails)
