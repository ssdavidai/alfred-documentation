## Streams vs Connected Apps — what's the difference?

**Streams** are passive data feeds. Each stream pulls or receives data from one source on a schedule (or on a webhook), turns each new item into a structured vault record using a zero-LLM Python template, and queues the record for hourly enrichment.

**Connected Apps** ([see guide](/guides/connected-apps)) are Composio integrations that include both a stream (data in) AND tool actions (actions out). When Sir connects Gmail, the auto-config creates a Gmail stream AND registers Gmail tool actions (`GMAIL_SEND_EMAIL`, `GMAIL_REPLY_TO_THREAD`, …) the agent can invoke.

A pure stream is data-only. A Connected App is data plus actions. Most users live entirely in the Connected Apps world; pure streams come up when wiring custom webhooks or polling an HTTP endpoint Composio doesn't cover.

## How a stream becomes a vault record

```
External source → stream event (raw JSON) → zero-LLM template → vault record
                                                     │
                                                     └── hourly batch enrichment
                                                         (one LLM call per hour
                                                         covers all pending records)
```

When an event lands, `EventProcessorWorkflow` (every 2 minutes) hands it to `create_stream_vault_record`. That's a pure-Python activity that picks the right template by source type and writes a structured record:

| Source type | Template | Vault record type |
|---|---|---|
| `gmail` / `email` / `agentmail` | email template | `event/` |
| `googlecalendar` (with `start`/`end`) | calendar template | `event/` |
| `github*` | github template | `event/` |
| `slack*` | slack template | `event/` |
| `omi-audio` | omi template (full transcript) | `conversation/` |
| `voice-call` | voice template (transcript) | `conversation/` |
| `sms` | sms template | `event/` |
| `notion` | notion template | `event/` |
| anything else | generic template | `event/` |

Each record is created with `enrichment_status: pending`. Once an hour, `HourlyEnrichmentWorkflow` runs **one** Clerk LLM call that batches every pending record (~200 events × ~80 tokens) into a single prompt, attaches entities and tags, and updates the records. This is a 97% reduction in LLM calls compared to per-event processing — Sir's costs scale flat against volume.

<Note>
**Zero-LLM ingest**, then **batched enrichment**. The vault is fresh within minutes (the template runs immediately); the AI-derived metadata catches up within the hour. This is why Alfred can show a Slack thread in your Activity feed seconds after it lands, but the cross-record entity links appear only after the hourly pass.
</Note>

## Pull modes

A stream's `pull_mode` tells the puller how to get data each cycle:

| Mode | What it does | Used by |
|---|---|---|
| **`snapshot`** | Every pull fetches the full latest state. Dedup happens at the StreamEvent table via the `(streamId, sourceRef)` unique index. | Notion (`NOTION_FETCH_DATA`), the catch-all default |
| **`append`** | Every pull fetches items newer than `last_pull_ts`. The cursor is a timestamp updated after each successful pull. | Gmail (`GMAIL_FETCH_EMAILS`), Slack, GitHub notifications |
| **`sync`** | Backfill once with a date window, then use a provider sync token (e.g. Calendar's `nextSyncToken`) for true delta sync. | Google Calendar |

Backfill windows differ by source: Gmail backfills 30 days, Slack 7 days, GitHub 14 days, Calendar 30 days past + 90 days future.

If a sync token expires (Calendar returns 410 Gone), the puller automatically resets to backfill mode and re-establishes the cursor.

## Built-in stream sources

| Source | How it arrives | Default cadence |
|---|---|---|
| **Composio-backed** (Gmail, Calendar, Slack, GitHub, Notion, Drive, …) | The puller calls `composio_execute({action: "..."})` on the tenant's openclaw workers | 5 min for most, 10 min for Notion |
| **Webhook** | External system POSTs to `/webhooks/:webhookToken` on the SaaS — token is auto-generated per stream and shown in the dashboard | Real-time |
| **HTTP pull** (custom) | Configure a `pull_endpoint`, optional `auth_config`, optional `cursor_field`. Generic JSON parser by default; per-source parsers available. | Configurable cron |
| **Omi audio** | The Omi wearable POSTs raw audio chunks; transcribed via Groq Whisper, then ingested as `conversation/` records | Per-chunk, real-time |
| **OpenClaw sessions** | Every agent session writes its summary to `system-openclaw-sessions` for cross-channel memory | Real-time |
| **AgentMail (unauthorised senders)** | Sir's `alfred.<username>@mail.alfred.black` inbox; non-authorised senders dispatch here | Webhook |
| **AgentPhone (unauthorised callers)** | Inbound SMS / voice from numbers not on the authorised list | Webhook |

Webhook tokens, OAuth credentials, and per-source state all live on Sir's tenant — the SaaS host only proxies. Webhook bodies up to 50 MB are accepted (sufficient for most Slack-attached files and GitHub payloads).

## Stream lifecycle controls

Every stream — Composio-backed or custom — supports the same controls via the dashboard or directly through ctrl-api:

| Action | Endpoint |
|---|---|
| List streams | `GET /api/v1/streams` |
| Recent events | `GET /api/v1/streams/events?status=unprocessed` |
| Push manually | `POST /api/v1/streams/ingest` with `{stream_id, stream_type, raw, summary?}` |
| Create | `POST /api/v1/streams` with `{id, name, type, source, enabled, webhookToken?}` |
| Update | `PATCH /api/v1/streams/:id` (any subset of config fields) |
| Pause | `POST /api/v1/streams/:id/pause` |
| Resume | `POST /api/v1/streams/:id/resume` |
| Delete | `DELETE /api/v1/streams/:id` |
| Prune old events | `POST /api/v1/streams/prune` with `{retention_days?}` (default 7) |

System streams (those with `system: true`) cannot be paused or deleted — they exist to power core flows like the openclaw sessions feed and the AgentMail / AgentPhone receivers.

## What a stream event looks like

```json
{
  "id": "ev_2L0vY7…",
  "stream_id": "gmail-primary",
  "stream_type": "gmail",
  "received_at": "2026-04-25T14:22:18.421Z",
  "source_ref": "msg_18f6c1234abcd",
  "raw": {
    "from": "sarah@acme.com",
    "to": ["sir@example.com"],
    "subject": "Re: Q3 contract revision",
    "body": "..."
  },
  "summary": "Sarah at Acme replied about the Q3 contract"
}
```

`raw` is the unmodified payload from the source. `summary` is set by the parser when it can derive one cheaply; otherwise enrichment fills it in on the hourly pass. The `(stream_id, source_ref)` pair is the dedup key — replays of the same event don't create duplicate vault records.
