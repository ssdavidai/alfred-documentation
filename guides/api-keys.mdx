## What an API key is for

API keys let scripts, automations, and external systems call Sir's Alfred without going through the dashboard's session login. Every call you make with a key is authenticated as Sir, hits Sir's dedicated tenant via the SaaS proxy, and runs against Sir's vault.

Keys are issued per-account, not per-instance. They authenticate you to the SaaS host at `alfred.black`, which then routes the request through Tailscale to your tenant's ctrl-api with the tenant's own internal credentials. You never see the per-instance internal credentials.

---

## Key format

Every key looks like `alf_<32 random hex characters>`. Total length is 36 characters. Example: `alf_a1b2c3d4e5f60718293a4b5c6d7e8f90`.

The dashboard shows you the full key **exactly once** — at creation. After that you only see the prefix (`alf_a1b2c3d4`) and the metadata (name, last-used timestamp).

<Warning>
The full key is only shown at creation time. Copy it immediately into your password manager or your script's secret store. If you lose it, revoke it and create a new one — there's no recovery flow.
</Warning>

## Creating a key

<Steps>
  <Step title="Open Settings">
    Sign in at [alfred.black](https://alfred.black) and click **Settings** in the sidebar.
  </Step>

  <Step title="Scroll to API Keys">
    Find the API Keys section. You'll see a list of your existing keys (by name and prefix), with their last-used timestamps.
  </Step>

  <Step title="Click Create Key">
    Give the key a descriptive name — "GitHub Actions deploy script", "Personal automation laptop", "Read-only monitoring". The name is for your reference only; it doesn't affect what the key can do.
  </Step>

  <Step title="Copy the key immediately">
    The full key appears in a modal. Copy it. The modal will warn you this is the only time you'll see it.
  </Step>
</Steps>

You can have up to **10 keys** per account. If you need more, revoke an unused one first.

## Using a key

Every call to the Alfred API uses Bearer authentication on `https://alfred.black/api/...`:

```bash
curl https://alfred.black/api/v1/vault/context \
  -H "Authorization: Bearer alf_a1b2c3d4e5f60718293a4b5c6d7e8f90"
```

The same key works for every endpoint Sir's tenant exposes — vault read/write, streams, learning, workflows, schedules, chores, workers, admin, phone, email. The proxy on the SaaS side handles the routing automatically based on which user's key it is.

See [How the API Works](/guides/how-the-api-works) for the proxy architecture, and [Recipes](/guides/recipes) for copy-pastable examples.

## Rotation

API keys don't expire automatically. They're valid until you revoke them. To rotate:

<Steps>
  <Step title="Create a new key">
    Settings → API Keys → Create Key. Give it a fresh name (e.g. "Personal automation laptop — rotated 2026-04").
  </Step>
  <Step title="Update your scripts and integrations">
    Replace the old key everywhere it's referenced. Test that everything still works.
  </Step>
  <Step title="Revoke the old key">
    Click Revoke next to the old key in the dashboard. The revocation is immediate — any subsequent call with the old key returns 401.
  </Step>
</Steps>

Rotate proactively if a key has been exposed (committed to a public repo, shared in a screenshot, leaked in a log) or if a device that held the key was lost.

---

## Last-used tracking

Every successful API call updates the `lastUsedAt` timestamp on the key. Use this to spot keys you've forgotten about — a key whose last use was three months ago is probably safe to revoke.

The timestamp updates fire-and-forget so it doesn't slow down your request, and only on successful authentication (a 401 from a bad key doesn't bump anything).

---

## What a key can NOT do

The key authenticates you to Sir's own tenant. It doesn't grant cross-tenant access (that's a separate Alfred Prime capability, gated by `ALFRED_PRIME=true` and a peer config — not surfaced to API key holders). It doesn't grant access to the SaaS host's admin endpoints. It doesn't bypass the per-instance Tailscale firewall.

If you need a more constrained credential — for instance, a key that can only read the vault, not write — that's not yet a built-in feature. Today every key has the same permissions as Sir himself does in the dashboard. Use one key per integration and revoke aggressively.
