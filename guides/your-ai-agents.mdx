## The cast

Sir doesn't talk to one AI. He talks to a small ensemble, each with a defined role and a different runtime.

| Agent | Role | When it runs | Where |
|---|---|---|---|
| **Main agent** | The Alfred who replies to Sir on Slack, web chat, email, voice, and SMS | On every user-facing turn | `openclaw` container |
| **Curator** | Processes manual inbox uploads into structured vault records | On every inbox event | `alfred` container (vault worker) |
| **Janitor** | Continuously scans the vault for broken links, structural issues, and inconsistencies | Periodic sweeps | `alfred` container (vault worker) |
| **Distiller** | Reads between the lines — surfaces assumptions, decisions, constraints, and contradictions | On-demand and scheduled | `alfred` container (vault worker) |
| **Surveyor** | Embeds records, clusters them by meaning, labels the clusters, discovers relationships | On-demand and scheduled | `alfred` container (vault worker), uses local Ollama embeddings |
| **Clerk** | Stateless LLM worker for analytical tasks — classification, extraction, summarisation, the hourly enrichment pass | Called by other agents | `openclaw-workers` container |
| **Ephemeral subagents** | Short-lived agents spawned per-task with scoped tool access | Per task execution | `openclaw-workers` container, lifecycle managed by `TaskRunnerWorkflow` |

## The main agent — Alfred

This is who Sir actually talks to. The main agent runs on the model configured for `agents.list[main].model.primary` in `openclaw.json` (Sir's tenant chooses this — Grok 4, Gemini 2.5 Pro, Claude, anything OpenClaw can route to).

His persona is composed from four files in the workspace:

- `USER.md` — who Sir is (written during onboarding)
- `SOUL.md` — how Alfred should speak (tone, register, what to call Sir)
- `MEMORY.md` — long-running notes Alfred maintains about Sir's preferences and life
- The platform `TOOLS.md` — what tools are available

The main agent has access to:

- The `self` MCP tool — calls Sir's own ctrl-api for vault, streams, learning, workflows, schedules, workers, admin, phone, and email operations
- `composio_execute` — every Connected App action Sir has authorised
- `web_search` and `web_fetch` for the world outside the vault
- `image`, `image_generate`, `video_generate`, `music_generate`, `tts`, `pdf`, `canvas` for media generation
- `cron` and `sessions_*` for scheduling and session control
- Standard shell primitives (`bash`, `read`, `write`, `edit`, `grep`, `find`, `ls`) scoped to the workspace

For Alfred Prime tenants only (today, that's just David), the main agent also gets `tenant({tenant, endpoint, ...})` and `ask_alfred({tenant, prompt})` — cross-tenant tools that proxy to peer tenants over Tailscale.

## The four vault specialists

These are independent processes inside the `alfred` container, each with their own daemon and their own LLM persona. They share the vault but operate without Sir's prompting.

### Curator

When a file lands in the inbox, Curator runs its 4-stage pipeline:

1. **Analyse** the body — what kind of content is this?
2. **Resolve entities** — for each person, organisation, project, location mentioned, find the existing vault record or create a new one
3. **Interlink** — wire the new content's mentions to the resolved entities via vault frontmatter
4. **Enrich** — derive tags, summary, surface decisions and action items

For binary uploads, the MediaIngestionWorkflow runs first (Whisper for audio, OCR for images, extraction for PDFs), then the Curator picks up the extracted text.

### Janitor

The vault is a living thing — links break when records get renamed, frontmatter goes out of date, an entity might end up referenced two slightly different ways. The Janitor sweeps periodically and fixes:

- Broken `[[Wiki Link]]` references
- Frontmatter schema violations
- Orphaned records (no incoming links AND no recent activity)
- Duplicate entities (same person, two slightly different records)

### Distiller

The Distiller reads what's already in the vault and surfaces what's hiding there:

- **Assumptions** the agent or Sir made implicitly that should be made explicit
- **Decisions** taken across multiple records but never recorded as a `decision/` entry
- **Constraints** that limit what's possible (a budget cap, a timeline, a capacity limit)
- **Contradictions** — two records that disagree, surfaced for Sir's resolution

Each distillation produces vault records of those types, linked back to the source records they were derived from.

### Surveyor

The Surveyor embeds every vault record (using a local Ollama-hosted `nomic-embed-text` model), clusters by semantic similarity using Milvus, labels each cluster with a generated heading (via `x-ai/grok-4.1-fast`), and discovers cross-cluster relationships. The output is what powers the Vault Nebula on the dashboard homepage.

Embeddings are local — no per-token cost. The Milvus index lives at `/mnt/encrypted/alfred/milvus_lite.db` on the tenant.

## The Clerk

Stateless LLM worker that runs on the `openclaw-workers` container. The Clerk has no persona, no SOUL, no memory — it does the analytical heavy lifting other agents need:

- Classification (which record type does this go to?)
- Extraction (pull structured data out of free text)
- Summarisation (what's the gist of this thread?)
- The **hourly enrichment pass** — one batched LLM call per hour covers every pending stream record

Every other agent (Curator, Distiller, the main agent when it needs analysis) dispatches to the Clerk through the `clerk.py` activity layer, which routes through the OpenClaw gateway.

## Ephemeral subagents

When the TaskRunner workflow needs to execute a discrete task — sending an email, posting to Slack, opening a PR — it spawns a short-lived **ephemeral subagent** on the `openclaw-workers` container with **per-task scoped tool access**. The subagent:

1. Is registered as `exec-<task_id>` in the workers' `openclaw.json`
2. Comes online when the workers config reloads (~15s)
3. Receives a single `sessions_spawn` with the task instructions
4. Runs to completion or timeout
5. Is deleted immediately after — no state carries over

Each subagent's `tools.allow` list is built from the task's declared tool requirements, not the full main-agent surface. This is the principle of least authority: a subagent that only needs to send one Slack message can't accidentally read every email in your vault.

## Where each agent runs

```
openclaw container
├── main (your Alfred)
└── learn-clerk (the Clerk)         ← also reachable from openclaw-workers

openclaw-workers container
├── learn-clerk                     ← background analytical work
├── vault-curator
├── vault-janitor                   ← these five mirror the vault specialists
├── vault-distiller                 ← so background Temporal activities can
├── vault-surveyor                  ← invoke them without contending with main
└── exec-<task_id> (×N)             ← ephemeral subagents per task

alfred container
├── curator daemon                  ← consumes inbox events
├── janitor daemon                  ← scheduled sweeps
├── distiller daemon                ← scheduled and on-demand
└── surveyor daemon                 ← scheduled clustering
```

The split exists so that long-running background work (the Surveyor re-clustering 5,000 records, the hourly Clerk pass enriching 200 events) doesn't compete with Sir's interactive sessions for openclaw's attention.

## What the agent says you did

Every main-agent session writes its summary to the `system-openclaw-sessions` stream. That stream is the cross-channel memory primer — when Sir calls Alfred at 5pm, the Voice Bridge bundles the last 7 days of session summaries into the call's instructions so Alfred remembers the morning Slack chat. Same for SMS.

You can read this stream directly: `GET /api/v1/streams/events?stream_id=system-openclaw-sessions`.
