Alfred uses restic with S3-compatible storage for automated encrypted backups. Backups run daily and include the entire vault, configuration, and worker state.

## What Gets Backed Up

The backup includes:
- **Vault**: All records (`/mnt/encrypted/vault/`)
- **Configuration**: `.env`, `openclaw.json`
- **State**: Worker state files, mutation logs, audit logs
- **OpenClaw**: Device pairing data, tokens

**Excluded**:
- Temporal database (workflow history — not critical)
- Docker volumes (ephemeral)
- Logs older than 7 days

Total backup size: typically 50-500 MB depending on vault size.

## Automated Backups

Backups run automatically every day at 3:00 AM UTC via cron job.

**Duration**: 2-5 minutes depending on vault size and S3 latency.

**Impact**: Containers are stopped during backup for consistency, then restarted automatically. Downtime: ~30 seconds.

<Note>
The backup script stops all containers, runs `restic backup`, and restarts containers. This ensures a consistent snapshot.
</Note>

## Manual Backup

Trigger an immediate backup:

```bash
curl -X POST https://alfred-acme.tailnet.ts.net:3100/api/backup/create \
  -H "X-API-Key: YOUR_API_KEY"
```

**Response (202 Accepted):**

```json
{
  "ok": true,
  "message": "Backup started",
  "job_id": "backup_20260226_153000"
}
```

<Warning>
The backup runs asynchronously. The API returns immediately (202) without waiting for completion. Monitor logs to verify success.
</Warning>

Check backup status in logs:

```bash
curl -N https://alfred-acme.tailnet.ts.net:3100/api/logs/stream?service=backup&tail=50 \
  -H "X-API-Key: YOUR_API_KEY"
```

**Log output:**

```

data: {"timestamp":"2026-02-26T15:30:05Z","service":"backup","level":"info","message":"Running restic backup..."}
data: {"timestamp":"2026-02-26T15:32:30Z","service":"backup","level":"info","message":"Backup complete: 245 MB in 2m 30s"}
data: {"timestamp":"2026-02-26T15:32:35Z","service":"backup","level":"info","message":"Restarting containers..."}
data: {"timestamp":"2026-02-26T15:32:40Z","service":"backup","level":"info","message":"Backup finished successfully"}
```

## Retention Policy

Restic prunes old snapshots automatically:

| Keep | Period |
|------|--------|
| **7** | Daily (last 7 days) |
| **4** | Weekly (last 4 weeks) |
| **6** | Monthly (last 6 months) |

Older snapshots are deleted after each backup to save storage.

<Tip>
Retention policy is configured in the backup script (`/opt/alfred/backup.sh`). Modify it via SSH if you need longer retention.
</Tip>

## Restoring from Backup

<Warning>
Restoration requires SSH access. The API does not provide restore endpoints (destructive operation).
</Warning>

### Full Restore

<Steps>
  <Step title="SSH into the instance">
    ```bash
    ssh deploy@alfred-acme.tailnet.ts.net
    ```
  </Step>
  <Step title="Stop all containers">
    ```bash
    cd /opt/alfred
    docker compose down
    ```
  </Step>
  <Step title="List available snapshots">
    ```bash
    restic -r s3:s3.amazonaws.com/your-bucket snapshots
    ```
  </Step>
  <Step title="Restore the snapshot">
    ```bash
    # Replace SNAPSHOT_ID with the ID from step 3
    restic -r s3:s3.amazonaws.com/your-bucket restore SNAPSHOT_ID \
      --target /mnt/encrypted
    ```
  </Step>
  <Step title="Verify restoration">
    ```bash
    ls -lh /mnt/encrypted/vault
    cat /mnt/encrypted/.env
    ```
  </Step>
  <Step title="Restart containers">
    ```bash
    docker compose up -d
    ```
  </Step>
  <Step title="Check health">
    ```bash
    curl https://alfred-acme.tailnet.ts.net:3100/api/health \
      -H "X-API-Key: YOUR_API_KEY"
    ```
  </Step>
</Steps>

### Partial Restore

Restore only specific files:

```bash
# Restore just the vault
restic -r s3:s3.amazonaws.com/your-bucket restore SNAPSHOT_ID \
  --target /tmp/restore \
  --include /mnt/encrypted/vault

# Copy to live location
cp -r /tmp/restore/mnt/encrypted/vault/* /mnt/encrypted/vault/
```

### Restore to a New Instance

<Steps>
  <Step title="Provision a new instance">
    Use alfred-ctrl to create a new instance
  </Step>
  <Step title="SSH into the new instance">
    ```bash
    ssh deploy@new-instance.tailnet.ts.net
    ```
  </Step>
  <Step title="Stop containers">
    ```bash
    cd /opt/alfred
    docker compose down
    ```
  </Step>
  <Step title="Configure restic with old credentials">
    ```bash
    export AWS_ACCESS_KEY_ID="old-access-key"
    export AWS_SECRET_ACCESS_KEY="old-secret-key"
    export RESTIC_PASSWORD="old-restic-password"
    ```
  </Step>
  <Step title="Restore from old snapshot">
    ```bash
    restic -r s3:s3.amazonaws.com/old-bucket restore SNAPSHOT_ID \
      --target /mnt/encrypted
    ```
  </Step>
  <Step title="Update .env with new instance details">
    ```bash
    nano /mnt/encrypted/.env
    # Update TAILSCALE_HOSTNAME, AAS_API_KEY, etc.
    ```
  </Step>
  <Step title="Restart containers">
    ```bash
    docker compose up -d
    ```
  </Step>
</Steps>

## Disaster Recovery Scenarios

### Corrupted Vault

<Steps>
  <Step title="Identify the last good snapshot">
    ```bash
    curl -X GET https://alfred-acme.tailnet.ts.net:3100/api/backup/snapshots \
      -H "X-API-Key: YOUR_API_KEY"
    ```
  </Step>
  <Step title="SSH and restore">
    Follow the full restore process above
  </Step>
  <Step title="Verify vault integrity">
    ```bash
    curl -X POST https://alfred-acme.tailnet.ts.net:3100/api/workers/janitor/scan \
      -H "X-API-Key: YOUR_API_KEY"
    ```
  </Step>
</Steps>

### Lost API Key

If `AAS_API_KEY` is lost and you can't access the API:

<Steps>
  <Step title="SSH into the instance">
    ```bash
    ssh deploy@alfred-acme.tailnet.ts.net
    ```
  </Step>
  <Step title="Regenerate the API key">
    ```bash
    NEW_KEY=$(openssl rand -hex 32)
    echo "AAS_API_KEY=$NEW_KEY" >> /mnt/encrypted/.env
    ```
  </Step>
  <Step title="Restart AAS">
    ```bash
    cd /opt/alfred
    docker compose restart alfred-aas
    ```
  </Step>
  <Step title="Test the new key">
    ```bash
    curl https://alfred-acme.tailnet.ts.net:3100/api/health \
      -H "X-API-Key: $NEW_KEY"
    ```
  </Step>
</Steps>

### Server Failure

If the server is completely lost:

<Steps>
  <Step title="Provision a new instance">
    ```bash
    node dist/index.mjs provision alfred-recovery
    ```
  </Step>
  <Step title="SSH and configure restic">
    Use old backup credentials
  </Step>
  <Step title="Restore from the latest snapshot">
    Follow "Restore to a New Instance" workflow
  </Step>
  <Step title="Update Tailscale">
    Re-authenticate Tailscale with the same hostname
  </Step>
</Steps>

## Backup Verification

Test backups monthly:

<Steps>
  <Step title="List snapshots">
    ```bash
    curl -X GET https://alfred-acme.tailnet.ts.net:3100/api/backup/snapshots \
      -H "X-API-Key: YOUR_API_KEY"
    ```
  </Step>
  <Step title="Verify snapshot integrity">
    ```bash
    ssh deploy@alfred-acme.tailnet.ts.net
    restic -r s3:s3.amazonaws.com/your-bucket check
    ```
  </Step>
  <Step title="Test partial restore">
    ```bash
    restic -r s3:s3.amazonaws.com/your-bucket restore latest \
      --target /tmp/test-restore \
      --include /mnt/encrypted/vault/person
    ```
  </Step>
  <Step title="Verify contents">
    ```bash
    ls /tmp/test-restore/mnt/encrypted/vault/person
    ```
  </Step>
  <Step title="Cleanup">
    ```bash
    rm -rf /tmp/test-restore
    ```
  </Step>
</Steps>

## Troubleshooting

### Backup Failed

Check logs:

```bash
curl -N https://alfred-acme.tailnet.ts.net:3100/api/logs/stream?service=backup&tail=100 \
  -H "X-API-Key: YOUR_API_KEY"
```

Common errors:
- **S3 credentials invalid**: Update `HETZNER_S3_*` in `.env`
- **Restic password wrong**: Update `RESTIC_PASSWORD` in `.env`
- **Network timeout**: Retry or check S3 endpoint

### Snapshot List Empty

Verify restic repository:

```bash
ssh deploy@alfred-acme.tailnet.ts.net
restic -r s3:s3.amazonaws.com/your-bucket snapshots
```

If no repository exists, initialize it:

```bash
restic -r s3:s3.amazonaws.com/your-bucket init
```

### Restore Incomplete

Check restic logs:

```bash
restic -r s3:s3.amazonaws.com/your-bucket restore SNAPSHOT_ID \
  --target /tmp/restore \
  --verbose
```

If files are missing, the snapshot may be corrupted. Try an older snapshot.
