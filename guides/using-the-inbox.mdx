## What the inbox is for

The inbox is Sir's manual upload channel — anything that doesn't arrive via a stream or a Connected App goes here. Meeting notes, voice memos, scanned documents, screenshots, PDFs from a research session, a CSV from your accountant. Drop it in and the Curator (one of your five AI specialists) takes over.

Files land in `vault/inbox/` on the tenant. The Curator processes them through its 4-stage pipeline (analyse → entity resolution → interlink → enrich) and writes structured vault records. Binaries take a different path — they're routed to `MediaIngestionWorkflow`, which calls Whisper for audio, OCR for images, or document extraction for PDFs before the structured records get written.

---

## How to put something in the inbox

<Tabs>
  <Tab title="From the dashboard">
    Click **Inbox** in the sidebar. Drag and drop one or more files onto the drop zone, or click to open the file picker. Multi-file uploads are supported.

    The dashboard shows the upload progress and, once accepted, the file appears in the Inbox tab of the Intelligence menu with a status of `processing`. When the Curator finishes, the file disappears from Inbox and the resulting vault records appear in the relevant tabs (Matters, Errands, or as standalone notes).
  </Tab>

  <Tab title="From Alfred conversationally">
    Tell Alfred over Slack, web chat, voice, or email: "I'm sending you the meeting notes from this morning, please file them with the kitchen renovation matter." Attach the file. Alfred will upload it through the inbox API on your behalf, then make sure the resulting records get linked to the right matter.
  </Tab>

  <Tab title="Via API">
    The ctrl-api accepts uploads at `POST /api/v1/vault/inbox` with a JSON body of `{filename, content}` (content base64-encoded for binaries). For text content (markdown, plain text), pass it as-is. See the [Recipes](/guides/recipes) guide for a copy-pastable example.
  </Tab>
</Tabs>

## Scan vs upload — two routes, one inbox

The dashboard upload and the conversational upload both end up calling the same endpoint, which writes the file to `vault/inbox/` and immediately emits a stream event so the Curator picks it up.

There's also a **scan** endpoint — `POST /api/v1/streams/inbox/scan` — that walks the inbox directory and ingests anything not yet seen. It's idempotent (uses `inbox:<filename>` as the dedup key), so triggering it repeatedly is safe. The scan runs automatically on a periodic Temporal schedule, but you can fire it manually if you've dropped files into the inbox directory by some out-of-band route.

## What happens to binary files

When the inbox upload sees a media file (audio, image, video, document), it emits the event with `stream_type: "media"` instead of `stream_type: "inbox-upload"`. That routing change matters: media events go to `MediaIngestionWorkflow`, which:

| File type | Pipeline |
|---|---|
| **Audio** (`.mp3`, `.m4a`, `.wav`, `.ogg`, …) | Groq Whisper transcription → topic-split if it's a braindump → vault `conversation/` records |
| **Images** (`.png`, `.jpg`, `.heic`, …) | Vision model description → vault `event/` or `note/` records |
| **Documents** (`.pdf`, `.docx`, `.xlsx`, …) | Text extraction → Curator pipeline as if it were inline text |
| **Video** (`.mp4`, `.mov`, …) | Audio extracted → Whisper → as audio above |

Plain text and markdown files skip the media route entirely and go straight to the Curator, which reads the body and decides what kind of record(s) to write.

---

## Braindumps

The MediaIngestionWorkflow has a special path for **braindumps** — long-form audio (typically a recording of you walking and talking through several topics at once). The braindump detector checks length, speaker count, and topic drift. If it triggers, the workflow splits the transcript by topic and creates one vault record per topic, each linked to the original conversation record.

This is what lets you walk for thirty minutes thinking out loud about three different projects, hand the recording to Alfred, and end up with three separate matters updated rather than one giant note.

---

## What the Curator produces

For a single document or text upload, the Curator runs four passes:

1. **Analyse** — read the body, identify what kind of content this is (meeting notes, contract, research summary, …)
2. **Entity resolution** — for every person, organisation, project, or location mentioned, look up the existing vault record or create a new one
3. **Interlink** — wire the new content's mentions to the resolved entities via vault frontmatter
4. **Enrich** — derive tags, add a summary if missing, surface decisions and action items

The output is typically a primary record (usually a `note/` or `conversation/`) plus updates to any related `person/`, `org/`, `matter/`, or `event/` records. A single meeting note can produce ten or more vault writes — every participant, every action item, every decision becomes its own record connected to the meeting.

---

## Watching it happen

The Activity tab in the Intelligence menu shows every Curator pass as it runs. Each pass logs its input file, the records it touched, and any failures. If something looks wrong — Curator missed an entity, picked the wrong matter, mis-summarised — you can promote the inbox event back, tag what was wrong, and the Reflection workflow uses that signal as a future correction.

<Note>
**Read before writing.** If you upload a doc that's already in the vault (a re-export of the same PDF, a duplicate meeting note), the Curator's entity resolution will detect existing records and update them rather than creating duplicates. The dedup key is the source_ref, and for re-uploaded files that's the filename — rename a file before re-uploading if you want a fresh record.
</Note>
