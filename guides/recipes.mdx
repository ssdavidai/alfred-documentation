## Recipe 1: Meeting Notes Pipeline

Extract structured data from meeting notes in seconds. This recipe shows how to upload raw meeting notes and have Alfred automatically extract people, tasks, decisions, and action items.

<Steps>
<Step title="Upload Meeting Notes">
First, create a plain text file with meeting notes:

```
Meeting: Q1 Planning - Product Team
Date: 2024-02-28
Attendees: Alice Chen (PM), Bob Smith (Lead Dev), Carol Lee (Design), David Park (QA)

Agenda Items:
1. Review Q4 outcomes - Alice presented metrics showing 32% user growth
2. Q1 roadmap - focus on mobile app redesign and API performance
3. Technical debt - Bob raised concerns about database migrations
4. Timeline - Carol needs 3 weeks for design mockups, Bob estimates 6 weeks for dev

Decisions Made:
- Freeze new feature requests after March 15th
- Hire 2 additional QA engineers (David to provide job descriptions)
- Use PostgreSQL 15 for migration (previously debated MySQL)
- Ship MVP by May 1st

Action Items:
- Alice: Schedule stakeholder check-in by March 5th
- Bob: Create migration timeline and risk assessment
- Carol: Deliver design mockups by March 21st
- David: Submit QA test plan by March 1st
```

Upload these notes to your inbox:

```bash
curl -X POST "https://app.alfred.black/api/v1/inbox/upload" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "filename": "Q1_Planning_Meeting_2024-02-28.txt",
    "content": "Meeting: Q1 Planning - Product Team\nDate: 2024-02-28\nAttendees: Alice Chen (PM), Bob Smith (Lead Dev), Carol Lee (Design), David Park (QA)\n\nAgenda Items:\n1. Review Q4 outcomes - Alice presented metrics showing 32% user growth\n2. Q1 roadmap - focus on mobile app redesign and API performance\n3. Technical debt - Bob raised concerns about database migrations\n4. Timeline - Carol needs 3 weeks for design mockups, Bob estimates 6 weeks for dev\n\nDecisions Made:\n- Freeze new feature requests after March 15th\n- Hire 2 additional QA engineers (David to provide job descriptions)\n- Use PostgreSQL 15 for migration (previously debated MySQL)\n- Ship MVP by May 1st\n\nAction Items:\n- Alice: Schedule stakeholder check-in by March 5th\n- Bob: Create migration timeline and risk assessment\n- Carol: Deliver design mockups by March 21st\n- David: Submit QA test plan by March 1st",
    "context": "meeting-notes"
  }'
```

**Expected Response:**

```json
{
```

</Step>


</Tip>

## Recipe 2: Email Thread Digest

Extract decisions, action items, and participants from a long email thread. This is perfect for catching up on discussions you missed.

<Steps>
<Step title="Prepare Email Thread">

Save a representative email thread as plain text:

```
From: sarah@company.com
To: eng-team@company.com
Date: 2024-02-25 09:15 AM
Subject: Re: API Rate Limiting Implementation

Sarah: Team, we need to finalize the rate limiting strategy. Current API
is getting 5M requests/day, mostly from dashboard. Two proposals:

1. Token bucket with 1000 req/min per API key (Alice's proposal)
2. Sliding window counter, 500 req/min per key (Bob's proposal)

Let me know thoughts by EOD. We need to ship by March 15th.

---

From: bob@company.com
Date: 2024-02-25 02:30 PM
Subject: Re: API Rate Limiting Implementation

Sarah, I prefer sliding window. Token bucket can have clock skew issues
in distributed systems. Also prefer 500/min to be conservative—better to
upgrade limits later than deal with abuse. Can we do a quick load test
to see what 500/min looks like during peak hours?

---

From: alice@company.com
Date: 2024-02-25 03:45 PM
Subject: Re: API Rate Limiting Implementation

Bob makes a good point on clock skew. I'm fine with sliding window.
However 500/min might be aggressive for our power users. Let's do 750/min
as a compromise. Load test is smart—I can run that Tuesday if we freeze
the implementation details.

Carol, what's your timeline for updating the client SDKs to handle 429s?

---

From: carol@company.com
Date: 2024-02-25 04:15 PM
Subject: Re: API Rate Limiting Implementation

SDK updates: I can have the 429 handling in v2.1.0 by March 10th if you
freeze the spec by March 1st. That gives us 10 days for integration testing.

Sarah, once the rate limit is final, I'll send the spec to mobile team too.

---

From: sarah@company.com
Date: 2024-02-25 05:00 PM
Subject: Re: API Rate Limiting Implementation

DECISION: Sliding window counter, 750 req/min per API key.
Load test Tuesday. Spec freeze: March 1st. Ship: March 15th.

Action items:
- Bob: Run load test, validate 750/min doesn't tank performance (due Tuesday)
- Alice: Finalize implementation spec (due Feb 29)
- Carol: SDK v2.1.0 with 429 handling (due March 10)
- Me: Notify customers, update API docs (due March 5)

Expect an email Tuesday evening with load test results.
```

```bash
    "filename": "email_thread_api_rate_limiting.txt",
    "content": "[Full email thread content from above]",
    "context": "email-thread"
```

## Recipe 3: Daily Vault Maintenance

Scan your vault for structural issues, review them, and apply automated fixes.

<Steps>
<Step title="Scan for Issues">

Run a janitor scan to identify broken links, invalid frontmatter, orphaned records, and other issues:

```bash
curl -X POST "https://app.alfred.black/api/v1/janitor/scan" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{}'
```

**Expected Response:**

```json
{
  "scan_id": "scan_20240228_001",
  "started_at": "2024-02-28T09:00:00Z",
  "status": "running",
  "estimated_completion_time_seconds": 45,
  "message": "Scanning vault for structural issues..."
}
```

</Step>

Once scanning is complete, retrieve the full report:

<Step title="Review Issues">

The janitor report gives you a detailed breakdown. In your UI dashboard, you can:
- See high-severity issues highlighted in red
- Click through to the exact line in each file
- Review suggestions
- Approve/reject fixes before applying

For the most critical issue (broken link to Steve Johnson), you have options:
1. Create the missing person record
2. Update the link to point to an existing person
3. Remove the link

</Step>

## Recipe 4: Knowledge Extraction

After your vault has accumulated many operational records, use distiller to extract latent knowledge—the assumptions, decisions, constraints, and contradictions embedded in your records.

<Steps>
<Step title="Understand the Distiller Workflow">

The distiller analyzes your vault records to find:
- **Assumptions**: Beliefs underlying decisions (e.g., "users prefer mobile over desktop")
- **Decisions**: Past choices and their rationales
- **Constraints**: Limitations that shape decisions (e.g., "limited budget", "strict compliance requirements")
- **Contradictions**: Conflicting assumptions or decisions that need resolution
- **Syntheses**: Connections between concepts

Before running distiller, ensure you have at least 20-30 operational records (tasks, projects, decisions) in your vault.

## Recipe 6: Automated Inbox via Bash Script

Create a simple bash script that watches a local folder and automatically uploads new files to Alfred's inbox. This is perfect for continuous input from notes apps, email exports, or document processors.

<Steps>
<Step title="Create the Watcher Script">

Save this as `watch_and_upload.sh`:

```bash
#!/bin/bash

# Alfred Inbox Auto-Uploader
# Watches a local folder and uploads new files to Alfred

set -e

# Configuration
WATCH_DIR="${WATCH_DIR:-.}"
API_KEY="${ALFRED_API_KEY:?Error: ALFRED_API_KEY not set}"
API_BASE_URL="${ALFRED_API_URL:-https://app.alfred.black}"
STATE_FILE=".alfred_upload_state"
LOG_FILE="alfred_uploads.log"

# Initialize state file (tracks uploaded files)
if [ ! -f "$STATE_FILE" ]; then
  touch "$STATE_FILE"
fi

log() {
  echo "[$(date +'%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"
}

upload_file() {
  local file="$1"
  local filename=$(basename "$file")

  log "Uploading: $filename"

  # Read file content
  local content=$(cat "$file")

  # Determine context based on file extension
  local context="document"
  if [[ "$filename" == *".meeting"* ]] || [[ "$filename" == *"meeting"* ]]; then
    context="meeting-notes"
  elif [[ "$filename" == *".email"* ]] || [[ "$filename" == *"thread"* ]]; then
    context="email-thread"
  elif [[ "$filename" == *".md" ]] || [[ "$filename" == *".txt" ]]; then
    context="document"
  fi

  # Upload via API
  local response=$(curl -s -X POST "$API_BASE_URL/api/v1/inbox/upload" \
    -H "Authorization: Bearer $API_KEY" \
    -H "Content-Type: application/json" \
    -d @- <<EOF
{
  "filename": "$filename",
  "content": $(echo "$content" | jq -Rs .),
  "context": "$context"
}
EOF
  )

  # Check for success
  if echo "$response" | jq -e '.success' > /dev/null 2>&1; then
    local inbox_id=$(echo "$response" | jq -r '.inbox_id')
    log "SUCCESS: $filename (inbox_id: $inbox_id)"
    echo "$file" >> "$STATE_FILE"
  else
    log "FAILED: $filename"
    log "Response: $response"
    return 1
  fi
}

monitor_directory() {
  log "Starting to monitor: $WATCH_DIR"
  log "API Key: ${API_KEY:0:10}..."

  # Initial scan
  while IFS= read -r file; do
    # Skip if already uploaded
    if grep -q "^$file$" "$STATE_FILE"; then
      continue
    fi

    # Skip special files
    if [[ "$file" == .* ]] || [[ "$file" == *"~" ]] || [[ "$file" == ".alfred_upload_state" ]]; then
      continue
    fi

    upload_file "$file"
  done < <(find "$WATCH_DIR" -maxdepth 1 -type f)

  # Watch for new files using fswatch (install: brew install fswatch)
  # Fallback to simple polling if fswatch not available
  if command -v fswatch &> /dev/null; then
    log "Using fswatch for real-time monitoring"
    fswatch -r "$WATCH_DIR" | while read file; do
      if [ -f "$file" ] && ! grep -q "^$file$" "$STATE_FILE"; then
        upload_file "$file"
      fi
    done
  else
    log "fswatch not available, using polling (60s intervals)"
    while true; do
      while IFS= read -r file; do
        if [ -f "$file" ] && ! grep -q "^$file$" "$STATE_FILE"; then
          upload_file "$file"
        fi
      done < <(find "$WATCH_DIR" -maxdepth 1 -type f -newer "$STATE_FILE")
      sleep 60
    done
  fi
}

# Main
if [ ! -d "$WATCH_DIR" ]; then
  log "ERROR: Directory not found: $WATCH_DIR"
  exit 1
fi

monitor_directory
```

</Step>

<Accordion title="Troubleshooting">

**Problem: "API Key not set"**
```bash
# Solution: Source .env file
source .env && ./watch_and_upload.sh
```

**Problem: Files uploaded twice**
```bash
# Solution: Delete .alfred_upload_state file to reset tracking
rm .alfred_upload_state
```

**Problem: Permission denied**
```bash
# Solution: Make script executable
chmod +x watch_and_upload.sh
```

**Problem: fswatch not found**
```bash
# Solution: Install fswatch (optional, script falls back to polling)
brew install fswatch  # macOS
apt-get install fswatch  # Linux
```

</Accordion>

<Tip>
The script uses a `.alfred_upload_state` file to track which files have been uploaded, preventing duplicates. To force a re-upload, simply remove that file. Logs are written to `alfred_uploads.log` for debugging.
</Tip>
