## What your vault actually is

Your vault is a directory of plain markdown files on a disk Alfred owns. Every person you mention, every matter Alfred is tracking, every email that flows through your accounts, every decision recorded — each one is its own file. The files are organised into folders by type (`person/`, `matter/`, `event/`, `decision/`, and so on), and they reference one another with `[[wikilinks]]`.

That is the entire system. There is no database behind it.

```
/vault
├── person/
│   ├── jane-doe.md
│   └── michael-ng.md
├── matter/
│   └── kitchen-renovation.md
├── event/
│   └── 2026-04-12-a3f9c1b2e8d4.md
├── task/
│   └── confirm-rsvp-with-cabinet-supplier.md
├── decision/
│   └── proceed-with-carrara-marble.md
└── conversation/
    └── 2026-04-02-1676adbd5443.md
```

## Why files-on-disk

A vault on disk gives Sir three things a database cannot.

<CardGroup cols={3}>
  <Card title="Sir owns the bytes" icon="lock">
    Every record is a markdown file Sir can read, edit, back up, or take elsewhere. Nothing is locked behind a proprietary schema.
  </Card>
  <Card title="Diffable history" icon="git-branch">
    Every change Alfred makes is appended to a mutation log. Sir can see what changed, when, and by whom.
  </Card>
  <Card title="Obsidian-compatible" icon="book-open">
    The vault is a valid Obsidian vault. If Sir prefers to browse it that way, Alfred will defer — and reuse Obsidian's live link index when it is running.
  </Card>
</CardGroup>

## How records are organised

Every record is a markdown file with a YAML frontmatter block declaring its type and a body containing the prose.

```markdown
---
type: matter
name: Kitchen renovation
status: active
created: 2026-03-04
tags: [home, renovation]
---

# Kitchen renovation

The whole-floor refit Sir signed off on in February. Carrara marble counters,
custom cabinet supplier in Brescia, plumbing rough-in scheduled for week 18.
```

Files live in folders named for their type — `matter/kitchen-renovation.md` for the example above. Some types share a folder (sessions and inputs are placed flexibly), but for the most part you can predict where a record lives from its type alone. See [Record types](/vault/record-types) for the full list.

## How Sir interacts with the vault

There are four ways for Sir to touch the vault. All four route through the same code path — the `alfred vault` CLI on the tenant — so the same scope rules and audit trail apply regardless of which surface Sir uses.

<Tabs>
  <Tab title="The dashboard">
    Sir's primary surface. The dashboard reads the vault through the API and renders it as the Vault Nebula homepage, the matter and errand views, and the per-record detail pages. Everything is live — edits Sir makes propagate back to the markdown files immediately.
  </Tab>
  <Tab title="A conversation with Alfred">
    The most natural surface. Sir tells Alfred something — "Jane Doe is the project manager for the kitchen, please add her" — and Alfred performs the vault writes through its tools. Every operation is scope-checked and logged.
  </Tab>
  <Tab title="The API">
    `GET /api/v1/vault/context`, `POST /api/v1/vault/create`, `PATCH /api/v1/vault/edit`. Eighty-six endpoints in total, documented in the [API reference](/api-reference/introduction). Used by the dashboard, by Alfred's agents, and by any integration Sir cares to build.
  </Tab>
  <Tab title="Direct file access">
    Available, but discouraged. Sir can SSH into the tenant and edit the markdown files directly. The [Janitor](/architecture/agent) will eventually notice any structural issues and tidy them — but the writes will not appear in the mutation log, and any specialist mid-operation may overwrite them.
  </Tab>
</Tabs>

## How Alfred uses the vault

The vault is the single source of truth for Alfred's understanding of Sir's world. Every specialist reads from it, and most of them write to it under strict permission boundaries.

| Specialist | Reads | Writes |
|------------|-------|--------|
| **Curator** | All | Creates entity records from inbox uploads (cannot delete) |
| **Janitor** | All | Repairs structural issues; can edit and delete, never creates |
| **Distiller** | All | Creates only learning records (assumption, decision, constraint, contradiction, synthesis) |
| **Surveyor** | All | Edits only `alfred_tags` and `relationships` frontmatter fields |
| **Main agent** | All | Full read/write — bounded by the same vault CLI gate |

These permissions are enforced **in code** by `alfred/vault/scope.py` before any operation reaches the filesystem. The Curator cannot delete; the Janitor cannot create; the Distiller cannot edit anyone's body text. See [Agentic Layer](/architecture/agent) for the full enforcement model.

## Encryption and ownership

<Note>
The vault lives on an encrypted volume on the Hetzner VPS Alfred provisioned for Sir. The disk is encrypted at rest with a key Sir holds; daily backups go to Hetzner S3, also encrypted. Tailscale is the only inbound network path. No third party — including the LLM providers — ever sees the vault contents in bulk; only the records Alfred deliberately includes in a prompt.
</Note>

The vault is not shared. It is not pooled. It is not multi-tenant. Every subscriber has a dedicated VPS, a dedicated vault, and a dedicated set of credentials. If Sir leaves, Sir takes the vault with him — it is a folder of markdown files, and Alfred will hand it over.
