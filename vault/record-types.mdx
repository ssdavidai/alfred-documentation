Every record in your vault declares a `type` in its frontmatter. The type determines which folder the file lives in, which status values are valid, which fields the dashboard renders, and which specialist can touch it.

Alfred recognises **twenty-nine types** — the original nineteen from the underlying [`alfred`](https://github.com/ssdavidai/alfred) vault package, plus ten extensions added by the platform for the kinetic, intuition, and signal layers (matter, chore, ledger_entry, triage, observation, instinct, reflection, stream_event, signal, needs_attention).

The canonical list lives in two files:

- `c/alfred/src/alfred/vault/schema.py` — the original nineteen
- `packages/ctrl/src/api/routes/vault.ts` — the platform's full list of twenty-nine

This page groups them by purpose.

## People and places

Records about the people, organisations, and locations Alfred is tracking on Sir's behalf.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `person` | `person/` | `active`, `inactive` | Curator extracts from inbox uploads; main agent on request |
| `org` | `org/` | `active`, `inactive` | Curator extracts from inbox uploads; main agent on request |
| `location` | `location/` | `active`, `inactive` | Mostly manual; occasionally extracted from calendar events |

<Note>
Person records are flexible — frontmatter typically holds `name`, `aliases`, `org` (a wikilink), `role`, `email`, `phone`. Body text is the brief Alfred has built up about the person from everything Sir has shared.
</Note>

**Example: a person record**

```markdown
---
type: person
status: active
name: Jane Doe
aliases: [Jane, J.D.]
org: "[[org/brescia-cabinet-supplier]]"
role: Project lead
email: jane@brescia-supplier.it
phone: "+39 030 555 1234"
created: 2026-03-04
tags: [vendor, kitchen]
---

# Jane Doe

Project lead at the Brescia supplier. Sir met her on the site visit
in March. Manages the cabinet rough-in schedule.
```

## Work

Records about ongoing work — the projects, matters, tasks, and recurring routines Alfred is helping Sir manage.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `project` | `project/` | `active`, `paused`, `completed`, `abandoned`, `proposed` | Manual or extracted from inbox |
| `matter` | `matter/` | `active`, `resolved`, `abandoned` | Onboarding matter pack; promoted from triage |
| `task` | `task/` | `todo`, `active`, `blocked`, `done`, `cancelled` | Curator from inbox; main agent; onboarding errand pack |
| `chore` | `chore/` | `active`, `paused`, `completed` | Onboarding chore generation; `POST /api/v1/chores` |
| `process` | `process/` | `active`, `proposed`, `design`, `deprecated` | Manual definition |
| `run` | `run/` | `active`, `completed`, `blocked`, `cancelled` | Started when a process executes |
| `triage` | `triage/` | `unprocessed`, `reviewed`, `resolved`, `dismissed` | Inbox classifier when an item cannot be confidently routed |
| `ledger_entry` | `ledger_entry/` | `active` | TaskRunner when an autonomous task completes — an audit-trail record |

<Tip>
**Matter vs project vs task.** A *matter* is an ongoing concern with no fixed end-state ("the kitchen renovation", "Q2 fundraise"). A *project* is a defined initiative with a beginning and an end ("ship the v3 release"). A *task* is a single, discrete, completable thing ("call Erik to book the cleaning"). Alfred's onboarding pipeline writes errands as `task` records with `tags: [errand]` — they are tasks, just so you know.
</Tip>

**Example: a matter record (auto-generated during onboarding)**

```markdown
---
type: matter
name: Kitchen renovation
status: active
category: home
description: Whole-floor refit; Carrara counters; Brescia cabinet supplier.
created: 2026-03-04
created_by: onboarding_pipeline
generated: true
key_people:
  - Jane Doe
  - Michael Ng
tags: [onboarding, matter, auto-generated, home]
---

# Kitchen renovation

> Whole-floor refit Sir signed off on in February.

## Context
Cabinet rough-in scheduled for week 18. Plumbing inspection requested
by the Brescia supplier on the same site visit.

## Suggested next actions
- Confirm RSVP with the supplier for the 14 April site visit.
- Decide on the splash-back finish before week 17.
```

**Example: a chore record**

```markdown
---
type: chore
name: Call Erik — car cleaning
status: active
template: call_erik_car_cleaning
schedule: '0 7 27 4 *'
schedule_id: chore-call-erik-car-cleaning
params: '{"chore_slug":"call-erik-car-cleaning","channel":"slack"}'
quarantine: false
quarantine_remaining: 0
created_by: self
created: '2026-04-25T08:04:04.820Z'
tags: []
workflow_class_name: CallErikCarCleanWorkflow
user_facing_description: 'Monday 9am Budapest — call Erik to book full car cleaning'
---

# Call Erik — car cleaning

Monday 9am Budapest — call Erik to book full car cleaning.
```

The `template` and `workflow_class_name` fields point to a Python Temporal workflow generated by Opus during onboarding (or by the main agent on request) and deployed to `/alfred-data/user-chores/`. See [the Kinetic layer](/architecture/kinetic) for the full chore lifecycle.

## Knowledge

Records that capture what Alfred has learned, decided, assumed, or noticed — the epistemic types.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `note` | `note/` | `draft`, `active`, `review`, `final` | Curator from inbox uploads; manual |
| `decision` | `decision/` | `draft`, `final`, `superseded`, `reversed` | Curator; Distiller; main agent |
| `assumption` | `assumption/` | `active`, `challenged`, `invalidated`, `confirmed` | Distiller only |
| `constraint` | `constraint/` | `active`, `expired`, `waived`, `superseded` | Distiller only |
| `contradiction` | `contradiction/` | `unresolved`, `resolved`, `accepted` | Distiller only |
| `synthesis` | `synthesis/` | `draft`, `active`, `superseded` | Distiller and Surveyor |

<Note>
The five **learning types** — `assumption`, `decision`, `constraint`, `contradiction`, `synthesis` — are the only types the [Distiller](/architecture/agent) is permitted to create. They each carry a `confidence` field, `source` references, and a chain of supporting evidence (`based_on`, `supports`, `challenged_by`, `confirmed_by`, `invalidated_by`).
</Note>

**Example: an assumption surfaced by the Distiller**

```markdown
---
type: assumption
status: active
confidence: medium
source: "[[conversation/2026-04-02-1676adbd5443]]"
source_date: 2026-04-02
project: ["[[matter/kitchen-renovation]]"]
based_on:
  - "[[event/2026-04-12-a3f9c1b2e8d4]]"
challenged_by: []
created: 2026-04-12
tags: [kitchen]
---

# The Brescia supplier handles export logistics in-house

## Claim
The Brescia supplier ships the cabinets directly to Sir's address in
Budapest without involving a third-party freight forwarder.

## Basis
Mentioned by Jane Doe during the 2 April call. Not yet confirmed in
writing — flagged for the Janitor to chase a contract clause.
```

## Stream-derived

Records created automatically from data flowing in through Streams and Connected Apps. These are the bulk of the vault — every email, calendar event, payment, GitHub notification, Slack message, and Omi audio chunk becomes one of these.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `stream_event` | `stream_event/` | (no constraint) | `EventProcessorWorkflow` for every stream event (zero-LLM); unified target for all sources since Phase 6.6 |
| `event` | `event/` | (no constraint) | Steward audit records only — `steward-action-*`, `signal-action-*`, `auto-task-created-*`, `daily-digest-*`. **Not** for new stream events. |
| `conversation` | `conversation/` | `active`, `waiting`, `resolved`, `closed`, `archived` | **Deprecated for new writes.** Pre-T6.6.1 OMI audio, voice call, and OpenClaw chat records still live here; new equivalents go to `stream_event/`. |
| `input` | `input/` | `unprocessed`, `processed`, `deferred` | Anything dropped into `inbox/` |
| `session` | `session/` | `active`, `completed` | `SessionTrackerWorkflow` from Alfred-Sir interactions |

The `stream_event` type is the workhorse. It carries `source`, `source_ref`, `stream_type`, `source_type`, and `enrichment_status` fields, and it is created by `stream_vault.py`'s per-source templates. The hourly enrichment pass adds entities, tags, and links in a single batched LLM call. T6.6.1 unified what used to be split between `event/` and `conversation/` into this single folder; the `source_type` frontmatter scalar (gmail, slack, omi, openclaw-chat, vexa, sure, gcal, plane, vault_edit, generic) lets the signal extractor classify in O(1) without filename heuristics. The migrator `packages/learn/scripts/migrate_to_stream_event.py` consolidated existing records during T6.6.1 and rewrote in-vault wikilinks accordingly.

**Example: a stream_event record from Gmail**

```markdown
---
type: stream_event
created: 2026-04-12T17:03:01+00:00
status: active
name: "Brescia Supplier: site visit confirmation — 14 April"
source: "993465f7-a2d0-41be-a83c-6b5d5d0e4c53"
source_ref: "gmail:19c24755a763b506"
stream_type: gmail
source_type: gmail
tags: [email]
enrichment_status: pending
---

# Brescia Supplier: site visit confirmation — 14 April

**From**: Jane Doe <jane@brescia-supplier.it>
**To**: David <david@example.com>

Confirming our site visit on 14 April at 10:00 — please let us know
the access code for the building.
```

The same `stream_event` type covers every stream source: Gmail, Slack, GitHub, Google Calendar, Notion, Polar payments, Omi audio, voice calls, SMS, OpenClaw chat. Per-source templates in `packages/learn/src/activities/stream_vault.py` shape the body and tags appropriately.

<Note>
The `event/` directory is **still alive** but only for steward audit records — `steward-action-*`, `signal-action-*`, `auto-task-created-*`, `daily-digest-*`. Those are not stream events; they are audit records emitted by `steward.apply_state_change`, `signal_actions._emit_signal_action_audit`, and the task-creation audit path. Genuine stream events written today always land under `stream_event/`.
</Note>

## Signal layer

Records produced by the Phase 6 signal extractor and steward — the distilled "what should change in the world" derived from raw stream events.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `signal` | `signal/` | `active`, `applied`, `superseded` | Signal extractor distils a stream event into an effect/target/confidence triple. Persists forever. |
| `needs_attention` | `needs_attention/` | `open`, `resolved`, `dismissed` | Signals that did not clear the autonomy threshold and surface to Sir for explicit review. |

A `signal` is the smallest reusable unit of "something happened that probably means X" — slug `<ts>-<short_hash>.md`, frontmatter carries `effect`, `target`, `confidence`, and a `source_ref` back to the originating stream event. A `needs_attention` record is what the steward writes when a signal looks real but discretion says "don't act on this autonomously" — slug `<ts>.md`, body explains what was noticed and what the steward considered. See [the signal layer](/architecture/signal-layer) for the full extract → judge → apply pipeline.

## Capabilities

Records about the accounts, services, and resources Alfred can act on or knows Sir owns.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `account` | `account/` | `active`, `suspended`, `closed`, `pending` | Mostly manual; sometimes from billing emails |
| `asset` | `asset/` | `active`, `retired`, `maintenance`, `disposed` | Manual or extracted from purchase confirmations |

`account` covers financial accounts, SaaS subscriptions, and platform logins. `asset` covers software licences, hardware, domains, vehicles — anything Sir might want Alfred to remember the existence of, with a vendor and a renewal date.

## Intuition

Records that power Alfred's learning loop — the Intuition layer's record types.

| Type | Folder | Status values | When created |
|------|--------|---------------|--------------|
| `observation` | `observation/` | `unprocessed`, `processed`, `invalid` | The observer hook captures every routing decision and user instruction |
| `instinct` | `intuition/instincts/` | `active`, `proposed`, `deprecated`, `merged` | Onboarding instinct pack; `ReflectionWorkflow` |
| `reflection` | `reflection/` | (no constraint) | Daily `ReflectionWorkflow` — reviews observations, proposes instinct changes |

<Tip>
**Observation, instinct, reflection** form a feedback loop. The observer captures what happened (an observation). The reflector reviews many observations and proposes changes (a reflection). The change either updates an existing instinct or creates a new one. Over time Alfred's autonomous behaviour comes to match how Sir actually works. See [the guide on intuition](/guides/intuition).
</Tip>

## A note on terminology

Alfred uses some terms that overlap with general usage in deliberate ways. Where there is risk of confusion:

- **Errand** is not a record type. Errands are written as `task` records with `tags: [errand]` (and frontmatter fields like `urgency`, `owner`, `related_matter`).
- **Skill** is not a record type. Skills live on disk under the agent's workspace, not in the vault.
- **Memory** is not a record type. The agent's working memory is `MEMORY.md` in the agent workspace; long-term memory is the vault itself.

If a term you read in the dashboard does not appear in this list, it is almost always a tagged or specialised view of one of the types above.
