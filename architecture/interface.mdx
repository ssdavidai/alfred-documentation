<Note>
The interface layer is everything Sir touches. The other layers do the work; this layer is where Sir gives instructions, checks on things, and lives alongside his butler.
</Note>

## A single front door, many entrances

Every entrance — dashboard, Slack DM, email reply, SMS, phone call, Telegram message — feeds the same agent and the same vault. The persona doesn't change. The capabilities don't change. Whether Sir asks "what's on this afternoon" by typing into the dashboard or by speaking it into a phone, Alfred answers the same way and from the same context.

## The dashboard

`https://alfred.black`. Sir's window into everything Alfred is doing. Routed in `packages/saas/app/main.wasp` and built with React under `packages/saas/app/src/dashboard/`.

The sidebar (`DashboardLayout.tsx`):

| Section | Path | What's there |
|---|---|---|
| **Alfred** | `/dashboard` | The Vault Nebula homepage. |
| **Command Center** | `/dashboard/command-center` | Health, vault stats, container status, knowledge graph, recent activity. |
| **Vault** | `/dashboard/vault` | Browse every record by type or path. Open a record to read, edit, link. |
| **Intelligence** | `/dashboard/tasks` | Five tabs: Inbox, Matters, Errands, Chores, Activity. |
| **Streams** | `/dashboard/streams` | Configured streams with status, event counts, pause/resume/edit. |
| **Tools** | `/dashboard/tools` | Per-app inventory of every Composio action available to the agent. |
| **Apps** | `/dashboard/integrations` | Connect/disconnect Composio apps; per-app stream + tool controls. |
| **Settings** | `/dashboard/settings` | Models, credentials, identity, billing. |

Plus per-tenant pages reached from elsewhere: Phone (`/dashboard/phone`), Devices, Logs, Workspace, Terminal, API Docs.

### The Vault Nebula

The homepage. Source: `packages/saas/app/src/components/nebula/VaultNebula.tsx`. Pure CSS — no WebGL, no Canvas. Layered radial gradients with an SVG blur filter, generated from real cluster data the Surveyor produced.

Each cloud is a cluster of related records. Cloud colour maps to record type — `matter` and `project` in gold, `decision` and `constraint` in deep teal, `contradiction` and `triage` in red, `synthesis` in violet, people and notes in warm cream. Density maps to count. Sir's vault visibly grows over time — new clusters appear, existing ones swell, the colour balance tells him at a glance what kind of work is currently weighing on the household.

Every other dashboard page renders the nebula as a faint background at 20% opacity, so Sir always sees his world behind whatever he's reading.

### The Command Center

Health, container status, disk, memory, recent workflow runs, recent specialist activity. Pulls from `GET /api/v1/admin/dashboard` plus the per-section endpoints (`/admin/health`, `/admin/containers`, `/admin/system/info`, `/admin/activity`).

When the openclaw gateway is restarting (e.g. right after Sir connects a new Composio app, which takes ~40 seconds), the dashboard hides the gateway window behind a "Configuring {toolkit}…" banner driven by `OpenclawStatusContext` polling `/api/v1/openclaw/ready` every two seconds.

### The Terminal page

`/dashboard/terminal` opens a live terminal inside the openclaw container, with a real PTY allocation, proper `SIGWINCH` for resizes, and the OpenClaw TUI/CLI directly available. Same access as `docker exec -it compose-openclaw-1 bash` over SSH, but in the browser.

## The MCP `self` tool — the agent's API

Every endpoint on the dashboard is also reachable by Alfred himself, through the MCP `self` tool defined in `packages/openclaw/mcp/ctrl-server.mjs`:

```js
self({ endpoint: "/api/v1/vault/context" })
self({ endpoint: "/api/v1/vault/records", method: "POST", body: { type: "task", name: "...", content: "..." } })
self({ endpoint: "/api/v1/email/send", method: "POST", body: { to, subject, text } })
self({ endpoint: "/api/v1/phone/sms", method: "POST", body: { to, body } })
self({ endpoint: "/api/v1/admin/dashboard" })
```

Same auth (the tenant's `AAS_API_KEY`), same JSON shapes, same routes. When Sir asks Alfred "is everything OK?", Alfred calls `self({ endpoint: "/api/v1/admin/dashboard" })` and reads what Sir would see on the Command Center. When Sir says "send a follow-up to Robert", Alfred reaches `/api/v1/email/send`. The agent and the dashboard live on the same surface.

The endpoint catalogue is in `packages/openclaw/workspace-template/docs/TOOLS.md`. Skill files (`workspace-template/skills/alfred-vault-operations`, `alfred-chore-management`, `alfred-learning-introspection`, `alfred-ops-health`, `alfred-email-channel`, `alfred-voice`) tell the agent the patterns for using each section.

## Federated tools (Alfred Prime only)

When a tenant is configured with `ALFRED_PRIME=true` and a `CROSS_TENANT_PEERS` JSON list, two more MCP tools appear:

- **`tenant({ tenant: "miguel", endpoint: "...", method?, body?, query? })`** — direct CRUD on a peer's ctrl-api over Tailscale. Same endpoint catalogue as `self`. Uses Prime's tokens.
- **`ask_alfred({ tenant: "miguel", prompt: "..." })`** — hand a prompt to a peer's Alfred and return the peer's reasoned reply. Uses the peer's tokens. Hits the peer's `POST /api/v1/cross-tenant/ask` endpoint.

Today only one tenant is Prime. On a non-Prime tenant the two tools simply aren't in the agent's tool list — there's no surface for the agent to invent a cross-tenant call.

The peer list is delivered through the `CROSS_TENANT_PEERS` env var as JSON: `[{ "id": "miguel", "tailscaleHost": "...", "tailscaleIp": "...", "apiKey": "...", "label": "Miguel" }]`. Routing prefers the Tailscale hostname (TLS) over the raw IP.

## Channels

Sir can reach Alfred from anywhere. Five channels are live; all converge on the same agent.

| Channel | Inbound | Outbound | Where Sir lives |
|---|---|---|---|
| Dashboard chat | Built into `/dashboard` | Same page | Sir's browser |
| Slack DM | OpenClaw's Slack adapter (Composio app) | Slack chat.postMessage via `/api/v1/notifications` | Sir's Slack workspace |
| Telegram | OpenClaw's Telegram adapter | sendMessage via `/api/v1/notifications` | Sir's phone |
| Email | AgentMail Svix webhook → tenant `/channels/email/inbound` | `/api/v1/email/send` / `/reply` / `/forward` | `alfred.<username>@mail.alfred.black` |
| SMS / voice | Twilio webhooks → tenant `/phone/sms/inbound` or Voice Bridge | `/api/v1/phone/sms` / `/phone/call` | Sir's Twilio subnumber |

The channel that matters most for the architecture is the dispatch split: **authorized senders get a conversational reply; everyone else becomes a stream event**. See [Channels](/architecture/channels) for the full picture.

### Cross-channel memory

A conversation Sir starts in Slack at 09:00 is in the agent's context when Sir calls the phone at 14:00. The Voice Bridge fetches `GET /api/v1/phone/voice-context` at call start and primes the OpenAI Realtime session with `MEMORY.md`, the last seven days of session summaries, and Sir's open matters and tasks. Email replies and SMS turns are written back to the streams pipeline so the next channel knows what happened on the previous one.

## Device pairing

Before any external client can reach the openclaw gateway, it has to be paired:

<Steps>
  <Step title="Device requests pairing">
    A client (the dashboard, a CLI, a custom app) initiates a request through the openclaw gateway.
  </Step>
  <Step title="Sir reviews">
    The request appears in the dashboard's Devices page as pending.
  </Step>
  <Step title="Approve or reject">
    Approval generates a cryptographically random device token (returned once). Rejection discards the request.
  </Step>
  <Step title="Device connects">
    The token is used in the `Authorization` header for every subsequent call.
  </Step>
</Steps>

Tokens can be rotated (recommended every 90 days) or revoked instantly from the same page.

## The REST API

`https://alfred.black/api/v1/*`. Bearer auth with an `alf_...` key Sir creates from the dashboard. The full endpoint surface is the same one the agent uses through `self` — vault CRUD, streams, learning, workflows, schedules, chores, channels, admin, integrations.

Rate-limited at the SaaS proxy. SHA-256 hashes are stored, never plain keys; the original is shown once on creation.

## What every interface shares

Every interface — dashboard click, Slack DM, email, SMS, phone call, REST API call — eventually lands on `ctrl-api:3100`. Every read goes through the same vault routes; every write through the same validators; every model call through the same OpenClaw gateway; every external action through the same Composio dispatcher. There's one Alfred, with one memory and one set of capabilities, reachable many ways.
