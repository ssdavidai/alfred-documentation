<Note>
The semantic layer is everything that gives meaning to records. The data layer puts records into the vault; this layer makes the vault navigable, learnable, and self-reasoning.
</Note>

## Three loops on top of the vault

Once a record exists in the vault, three independent loops work on it: the Surveyor draws connections by meaning, the Distiller extracts the knowledge hiding between the lines, and the Learning + Judgment pair watches how Sir routes things and quietly takes over the routine cases.

| Loop | Specialist / workflow | What it produces |
|---|---|---|
| Embedding + clustering | Surveyor (`alfred` container daemon) | Cluster tags + suggested `related_*` links in record frontmatter |
| Latent extraction | Distiller (`alfred` container daemon) | `decision`, `assumption`, `constraint`, `contradiction`, `synthesis` records |
| Routing intuition | LearningWorkflow + JudgmentWorkflow (`alfred-learn`) | `observation` records, `instinct` records, autonomous routing of new inputs |

## The Surveyor — embedding and meaning

Source: `c/alfred/src/alfred/surveyor/`. Runs as a daemon inside the `alfred` container on a five-second tick (`daemon.py`, `LOOP_INTERVAL = 5.0`). Watches the vault for added/changed/removed files and keeps a vector index in step.

### Embedding

`embedder.py`. Each record's body is normalized, chunked at 3,000 characters with a 300-character overlap (safe across English, Hungarian, dense YAML, and code), and embedded with `nomic-embed-text` (768 dimensions) running locally in the `ollama` container. Multiple chunks per record are mean-pooled and L2-normalized into a single document-level vector.

The vector store is **Milvus Lite** — embedded directly in the surveyor process, not a separate container. The Milvus DB lives at `/mnt/encrypted/vault/.alfred/milvus_lite.db`.

<Tip>
Embeddings are local and free. Earlier versions used OpenRouter's hosted `nomic-embed-text`, which billed per call. The shift to local Ollama (`packages/ctrl/src/templates/docker-compose.yaml.njk`) cut the running cost of the surveyor to zero and removed a network dependency from a hot loop.
</Tip>

### Clustering

`clusterer.py`. Cosine-similarity clustering across the full vector space. Outputs cluster groups with member record paths. Re-clusters incrementally as the vault changes.

### Labelling + relationship suggestions

`labeler.py`. For each cluster, two LLM calls (against `xai/grok-4.1-fast` via OpenRouter):

1. **Cluster label.** Read the cluster's members, return 1–3 hierarchical tags. `["construction/residential", "project-management"]`.
2. **Relationship suggestions.** Read pairs of cluster members that don't already wikilink to each other; suggest a relationship type (`related-to`, `supports`, `depends-on`, `part-of`, `supersedes`, `contradicts`) plus a confidence and a 50-character context note. Only confidence ≥ 0.5 is returned.

When a cluster contains a first-class entity (`matter`, `person`, `org`, `project`), the entity's slug becomes a canonical cluster tag — every member of the cluster inherits it via `alfred_tags: [<slug>]` in their frontmatter, so downstream consumers can filter by it.

### Write-back

`writer.py`. The labeller's output is written into each member's frontmatter as `alfred_tags`, plus `related_to`, `supports_to`, `depends_on_to`, etc. fields pointing at cluster peers. This makes the vault a real graph: any record can answer "what else is related to me?" without re-running embeddings.

The Surveyor never edits a record's body, never creates a new record, and never deletes anything. It only patches frontmatter `related_*` fields.

## The Distiller — knowledge between the lines

Source: `c/alfred/src/alfred/distiller/`. Runs as a daemon in the `alfred` container with two passes (`pipeline.py`).

### Pass A — per-source extraction

For each candidate record (typically `conversation`, `session`, `note`), one Clerk call extracts:

- **Decisions** — phrases like "we agreed", "let's go with", "decided to", "approved", "going with", "settled on", "confirmed"
- **Assumptions** — "we're assuming", "should be fine", "expecting", "presumably", implicit beliefs about timelines or outcomes
- **Constraints** — "we can't", "regulation requires", "budget cap", "deadline is", "mandatory", "compliance"
- **Contradictions** — "but", "however", "conflicting", "disagrees", "inconsistent" — disagreements between participants

Candidate selection is itself a deterministic pass (`candidates.py`). Body length, presence of `## Outcome` sections, link density, and keyword counts produce a score; only above-threshold records get an LLM call. Already-distilled records are skipped via a content-hash check.

A pure-Python step deduplicates fuzzy near-matches (same decision worded differently across two meetings → one decision record with multiple `source_links`).

### Pass B — cross-learning meta-analysis

A second pass scans the existing learning records themselves, finds clusters of related decisions or contradictions across sources, and creates higher-order `synthesis` records that link the reasoning graph together.

The Distiller can only ever create five record types: `decision`, `assumption`, `constraint`, `contradiction`, `synthesis`. It never edits an existing record and never deletes one. The scope is enforced by the vault CLI, not by the prompt.

## How records connect

Every record can reference others through wikilinks (`[[person/Robert Clarke]]`, `[[matter/eagle-farm]]`) in its body and through `related_*` frontmatter fields written by the Surveyor. The result is a vault that grows richer the more Sir uses it.

A planning meeting:

1. Sir drops the meeting notes into the inbox.
2. The **Curator** writes a `conversation/` record with a `[[person/...]]` wikilink for each participant; creates new `person/` records for anyone unseen; emits cross-linked `event/` records for action items.
3. Within the hour, the **hourly enrichment** pass adds entity references and topic tags.
4. The **Janitor** verifies every wikilink resolves, every required field is set, every record sits in the right directory.
5. The **Distiller** later reads the conversation and surfaces an `assumption/` record (`"we're assuming the API will be ready by March"`) and a `constraint/` record (`"budget capped at $50k"`).
6. The **Surveyor** clusters this conversation with three earlier ones on the same project, tags the cluster `product/launch`, and adds `related_to` links across all four.

That's six independent passes producing a tightly cross-referenced piece of vault, each running on its own loop.

## Intuition — Alfred learns how Sir routes

When Sir routes something — moves an inbox item to a matter, files an email under a project, dismisses a triage as noise — Alfred takes notice. Over time, those notices become instincts; once instincts are confident enough, Alfred starts handling the routine cases himself.

Three workflows in `alfred-learn` make it work, each on its own cadence (`packages/learn/scripts/register_schedules.py`): Learning every fifteen minutes, Judgment every fifteen minutes, and Reflection once a day at 02:00 tenant-local.

<Note>
Distilled knowledge (decisions, assumptions, constraints) and intuition (instincts, observations) feed Alfred's downstream awareness. See [Signal layer](/architecture/signal-layer) for how those records become the per-matter perception loop.
</Note>

### Learning — observation capture

`packages/learn/src/workflows/learning.py`. Every fifteen minutes:

1. **Observation queue.** The `alfred-learn-observer` hook in OpenClaw watches assistant messages for routing language ("filed under", "moved to", "categorized as", "this is a"). Each detected event is written to `/mnt/encrypted/alfred/observation-queue.jsonl`. The workflow drains the queue and asks the Clerk to extract a structured observation.
2. **`alfred_instructions` watcher.** Vault records may carry an `alfred_instructions:` frontmatter field — Sir's explicit "do this whenever X happens". The workflow scans recently modified vault files, extracts those instructions, and creates observations for them (and executes them, if they're actionable).
3. **Chore-run seeding.** Cursor-tracked replay of `chore-run-history.jsonl`, so every chore execution becomes an observation without needing an LLM.

The output is `observation/` records — atomic, dated, with the input ref, the routing decision, the reasoning, and the matching signals (domain patterns, keyword patterns, input types, attachment patterns).

### Reflection — nightly synthesis

`packages/learn/src/workflows/reflection.py`. Runs at 02:00 tenant-local. Reviews the day's observations against existing instincts and asks the Clerk for a structured proposal:

- **Create** — a brand-new instinct (only when at least 3 observations match the same pattern)
- **Update** — strengthen an existing instinct (add signals, increment match count)
- **Merge** — two instincts now overlap enough to fuse
- **Deprecate** — new evidence contradicts an existing instinct

Validated proposals are applied; observations are marked processed; the intuition index is rebuilt; a `reflection/` record is written for Sir to review.

### Judgment — the routing decision

`packages/learn/src/workflows/judgment.py`. Runs every fifteen minutes, or inline when the event processor encounters a fresh input:

1. Load the intuition index. If empty (cold-start tenant), everything escalates to Sir.
2. Extract the input's metadata (domain, keywords, type, attachments).
3. Score each instinct against the metadata using a weighted match (domain 0.30, keywords 0.30, input type 0.15, attachment 0.15, tags 0.10).
4. Apply discretion. The threshold scales with how much evidence Alfred has:

| Observations behind the instinct | Discretion threshold | Butler equivalent |
|---|---|---|
| < 5 | 0.95 | "I've barely seen this before, sir." |
| 5–9 | 0.90 | "I believe I know, but I'd rather confirm." |
| 10–19 | 0.85 | "I'm fairly certain this goes here." |
| 20–49 | 0.80 | "I've seen this many times." |
| 50+ | 0.75 | "Routine. Already done." |

If the best match clears the threshold, Alfred routes autonomously and records a machine-attributed observation. If it doesn't, the input escalates: it goes to Triage in the dashboard, and Sir's routing decision becomes a fresh observation that feeds the next reflection.

When the matched instinct carries an `execution:` block (instructing what to actually do), judgment also creates an execution `task/` record and hands it to the Task Runner — see [Kinetic](/architecture/kinetic).

### Cold start, then accumulation

A brand-new tenant has zero instincts. Everything escalates. Sir routes; observations accumulate; the first reflection at 2 a.m. creates a few proposed instincts; the next morning judgment starts catching the obvious cases. Within a few weeks, the routine half of Sir's intake is handled before he sees it. The discretion gate guarantees Alfred always errs on the side of asking when the evidence is thin.

## Twenty-six record types, five families

The full schema lives in `alfred.vault.schema`. Five conceptual groupings:

<AccordionGroup>
<Accordion title="Standing entities — the stable elements">
`person`, `org`, `project`, `location`, `account`, `asset`, `process`, `matter`. These persist and accumulate connections.
</Accordion>

<Accordion title="Activity records — what happened">
`conversation`, `note`, `task`, `triage`, `event`, `session`, `input`, `run`, `ledger_entry`. Time-bound records of work, interactions, and execution outcomes.
</Accordion>

<Accordion title="Execution types — how work gets done">
`chore` (recurring scheduled workflow, Opus-generated), `ledger_entry` (closed-out record of an executed errand).
</Accordion>

<Accordion title="Learning types — what we know">
`assumption`, `decision`, `constraint`, `contradiction`, `synthesis`. Created exclusively by the Distiller.
</Accordion>

<Accordion title="Intuition types — how Alfred learns">
`observation`, `instinct`, `reflection`. Created by Learning, Reflection, and Judgment.
</Accordion>
</AccordionGroup>
