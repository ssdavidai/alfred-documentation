## Your Alfred lives on his own machine

When Sir signs up, the SaaS control plane provisions a fresh Hetzner Cloud VPS, attaches an encrypted volume, joins it to a private mesh, and stands up Sir's full Docker stack on it. Nothing is shared with another tenant — not the database, not the vault, not the AI gateway, not the workflow engine.

| Component | Detail |
|---|---|
| Server | Hetzner Cloud `cx53` for Premium and Black tiers; `cx33` for the older Starter tier (`packages/saas/app/src/payment/plans.ts`) |
| Storage | LUKS2 full-disk encryption, AES-256-XTS, key generated per-instance (`packages/ctrl/src/templates/cloud-init.yaml.njk`) |
| Network | Private Tailscale mesh on WireGuard; SSH on port 22, Tailscale on UDP 41641, ICMP — nothing else |
| Browser path | Cloudflare → SaaS host → Tailscale → tenant ctrl-api on `127.0.0.1:3100` |

There is no public route into your Alfred's services. The Hetzner firewall denies everything except SSH, ICMP, and Tailscale's WireGuard port. Every container binds to localhost.

## Provisioning, in order

The fleet TUI/CLI in `packages/ctrl/src/infra/provisioner.ts` walks the same numbered steps for every new instance:

<Steps>
  <Step title="Generate an Ed25519 keypair">
    Stored locally at `data/ssh_keys/<instance_id>/`. Never re-used between tenants.
  </Step>
  <Step title="Upload the public key + ensure shared firewall">
    Hetzner firewall labelled `managed-by: alfred-ctrl` allows only inbound SSH and ICMP.
  </Step>
  <Step title="Create the encrypted volume + render cloud-init">
    Cloud-init is rendered from `cloud-init.yaml.njk` and includes the LUKS keyfile generation, fail2ban + UFW setup, Docker install, Tailscale install, and the systemd timer that pulls fresh container images every 15 minutes.
  </Step>
  <Step title="Create the server + wait for cloud-init">
    Hetzner backups are enabled. The volume is attached. Cloud-init formats the disk with LUKS2 and creates `/mnt/encrypted/{vault,openclaw,alfred,temporal}`.
  </Step>
  <Step title="Upload secrets via SSH (not via cloud-init)">
    `.env` (LLM keys, AAS_API_KEY, Composio creds, AgentMail/AgentPhone tokens) is SFTP'd post-boot. Cloud-init `user_data` is readable through Hetzner's metadata API, so secrets never go there.
  </Step>
  <Step title="Render + upload docker-compose, start containers">
    The compose file is rendered from `docker-compose.yaml.njk`, the init container scaffolds the vault and writes config, and the rest of the stack comes up.
  </Step>
  <Step title="Bootstrap OpenClaw + Tailscale + Cloudflare Tunnel">
    Tailscale auth key is generated on the fly. The instance joins Sir's private mesh. Cloudflare DNS + Tunnel records are created if a custom subdomain is configured.
  </Step>
  <Step title="Back up the LUKS key + run health check">
    The keyfile is downloaded into `data/ssh_keys/<id>/luks.key` so the instance can be recovered if Hetzner loses the disk. A first health probe must come back green before the instance is marked `done`.
  </Step>
</Steps>

End-to-end: roughly three minutes when using the golden snapshot, eighteen to twenty when building from scratch.

## The encrypted volume

Everything that matters is on `/mnt/encrypted` — the LUKS2 volume — and nothing outside it.

| Path | What it holds |
|---|---|
| `/mnt/encrypted/vault` | Sir's Markdown vault. Bind-mounted into every container. |
| `/mnt/encrypted/openclaw` | Main agent's OpenClaw config + workspace. |
| `/mnt/encrypted/openclaw-workers` | Background workers' OpenClaw config + workspace. Separate from main. |
| `/mnt/encrypted/alfred` | Tenant secrets file (`.gateway-token`), authorized phone numbers, onboarding state, Composio user id, stream JSONLs. |
| `/mnt/encrypted/temporal` | Temporal SQLite database (workflow state). |
| `/mnt/encrypted/ollama` | Cached embedding model weights for the surveyor (`nomic-embed-text`). |

The LUKS keyfile lives at `/opt/alfred/luks.key` on the host — readable only by root, used by the `alfred-luks.service` systemd unit to auto-unlock the volume on boot.

## The Docker stack

Eight containers per tenant. All built from `docker-compose.yaml.njk`; image tags are `:latest` because the `alfred-update.timer` systemd unit pulls fresh images every fifteen minutes.

| Service | Image | Port | Role |
|---|---|---|---|
| `init` | `ssdavidai00/alfred-init` | — | One-shot. Scaffolds vault, copies skills, renders config, generates the gateway token, registers the MCP `self` server. Exits when done. |
| `temporal` | `temporalio/temporal` | `7233`, `8233` | Durable workflow engine. SQLite-backed, dev mode. |
| `ollama` | `ollama/ollama` | `11434` | Local embedding server. Auto-pulls `nomic-embed-text` on first boot. |
| `openclaw` | `ssdavidai00/alfred-openclaw` | `18789` | Main agent gateway. Where Sir's chat, voice, and channel messages land. |
| `openclaw-workers` | `ssdavidai00/alfred-openclaw` | `18790` | Background-worker gateway. Where the Clerk and ephemeral subagents run. Separate process so a long-running chore can't block Sir's chat. |
| `alfred` | `ssdavidai00/alfred-worker` | — | Vault specialists daemon — runs the Curator, Janitor, Distiller, and Surveyor processes. |
| `alfred-learn` | `ssdavidai00/alfred-learn` | — | Python + Temporal worker. Hosts every scheduled workflow (event processor, hourly enrichment, judgment, task runner, daily digest, reflection, and the dynamically loaded chores). |
| `ctrl-api` | `node:22-slim` running `api.mjs` | `3100` | Tenant API. Every read and write to the vault, every stream operation, every channel send goes through here. |

`init` must complete before `openclaw` and `openclaw-workers` start. Both must be healthy before `alfred` boots. `alfred-learn` waits on `temporal`, `openclaw-workers`, and `ctrl-api` all being healthy. The dependency graph is enforced in compose, not by hope.

<Tip>
The two OpenClaw processes (`openclaw` and `openclaw-workers`) share the same Docker image but read different config files and write to different state directories. This separation means a long batch job in `openclaw-workers` cannot stall Sir's interactive chat in `openclaw`.
</Tip>

## Network paths

There are exactly three ways into Sir's machine:

1. **Tailscale** — the SaaS host, the operator's laptop, and any paired devices reach `ctrl-api:3100` over the mesh. Encryption is WireGuard, end-to-end. The Hetzner firewall allows the WireGuard handshake on UDP 41641 and nothing else.
2. **Cloudflare Tunnel** — outbound-only. When Sir opens the dashboard at `alfred.black`, the SaaS host proxies the API call over Tailscale; no inbound HTTP hits the tenant directly.
3. **SSH** — port 22, key-based only, hardened in cloud-init (`PermitRootLogin no`, `PasswordAuthentication no`, `MaxAuthTries 3`, fail2ban on top).

The voice channel takes a different shape: Twilio's Media Stream WebSocket lands on `voice.alfred.black` (DNS-only at Cloudflare, because the WAF drops Twilio's WS upgrade with error 31920) and the Voice Bridge runs on the SaaS host, not on the tenant. See [Channels](/architecture/channels) for the full picture.

## Backups + recovery

A daily `restic` job runs at 03:00 host-local via the `alfred-backup.timer` systemd unit (`cloud-init.yaml.njk`). It stops `alfred`, `openclaw`, and `temporal` for a consistent snapshot, runs `restic backup /mnt/encrypted --exclude=/mnt/encrypted/temporal`, and restarts them. Retention: 7 daily, 4 weekly, 6 monthly. Destination: Hetzner Object Storage, encrypted with a Restic password that is itself stored only in `/opt/alfred/restic.env` and `data/ssh_keys/<id>/restic.env` on the operator's machine.

The LUKS keyfile is backed up at provision time and never touched again. Even if Hetzner loses the disk, the operator can re-mount the encrypted volume from its raw bytes plus the backed-up keyfile.

## Defence in depth

Every long-lived container runs with:

- `no-new-privileges: true`
- `cap_drop: ALL` (with narrow `cap_add` like `DAC_OVERRIDE` only where needed for `chown` on bind-mounts)
- `mem_limit` of 1–4 GB depending on workload (`alfred-learn` and `openclaw` get 4 GB; `ctrl-api` gets 1 GB)
- `pids_limit: 128–256`
- All ports bound to `127.0.0.1` only

`fail2ban` watches sshd. UFW denies inbound except 22/tcp and 41641/udp. Unattended-upgrades is enabled for security packages. SSH keys are Ed25519 and per-instance.
