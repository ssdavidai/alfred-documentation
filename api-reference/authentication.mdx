## Overview

Every API request must include a Bearer token in the `Authorization` header. API keys are created in the Alfred dashboard and scoped to your account.

## Getting your API key

1. Log in to the [Alfred dashboard](https://app.alfred.black)
2. Navigate to **Settings > API Keys**
3. Click **Create API Key** and give it a descriptive name
4. Copy the key immediately — it is only shown once

API keys use the format `alf_<32 hex characters>`. You can create up to 10 keys per account.

<Warning>
  API keys are shown only at creation time. If you lose a key, revoke it and create a new one.
</Warning>

## Making authenticated requests

Include the API key as a Bearer token in the `Authorization` header:

<CodeGroup>
```bash cURL
curl https://app.alfred.black/api/v1/vault/context \
  -H "Authorization: Bearer alf_your_api_key_here"
```

```javascript JavaScript (fetch)
const response = await fetch('https://app.alfred.black/api/v1/vault/context', {
  headers: {
    'Authorization': 'Bearer alf_your_api_key_here'
  }
});

const data = await response.json();
```

```python Python (requests)
import requests

response = requests.get(
    'https://app.alfred.black/api/v1/vault/context',
    headers={'Authorization': 'Bearer alf_your_api_key_here'}
)

data = response.json()
```

```typescript TypeScript (axios)
import axios from 'axios';

const response = await axios.get(
  'https://app.alfred.black/api/v1/vault/context',
  {
    headers: {
      'Authorization': 'Bearer alf_your_api_key_here'
    }
  }
);

const data = response.data;
```
</CodeGroup>

## Security

<Note>
  Token comparison uses `crypto.timingSafeEqual` for constant-time comparison, making the API resistant to timing attacks.
</Note>

### How keys are stored

- API keys are **hashed** server-side using SHA-256 before storage — the plaintext key is never persisted
- Only the hash is stored in the database — even a database breach does not reveal your key
- Each key is associated with your user account and instance

### Best practices

- Store API keys in environment variables or secret management systems — never hardcode them
- Use descriptive names when creating keys so you can identify and revoke them later
- Revoke unused keys promptly from **Settings > API Keys**
- All API traffic uses HTTPS (TLS) end-to-end

## Error responses

Missing or invalid authentication returns a `401` status:

<ResponseExample>
```json Missing or malformed header
{
  "error": "Missing or invalid Authorization header. Use: Bearer <api-key>"
}
```

```json Invalid key format
{
  "error": "Invalid API key format"
}
```

```json Key not found
{
  "error": "Invalid API key"
}
```
</ResponseExample>

| Status | Error | When |
|--------|-------|------|
| `401` | `Missing or invalid Authorization header...` | `Authorization` header is absent or doesn't start with `Bearer ` |
| `401` | `Invalid API key format` | Token is present but doesn't match the `alf_` format |
| `401` | `Invalid API key` | Token format is correct but the key doesn't match any valid key |

## Timeouts

API requests are subject to a **15-second timeout**. If the upstream operation takes longer, you will receive a `504` response:

```json
{
  "error": "Request to instance timed out after 15s"
}
```

Most endpoints respond well within this limit. Operations that invoke AI processing (e.g., `POST /api/v1/workers/process`, `POST /api/v1/workers/janitor/fix`, `POST /api/v1/workers/distiller/run`) run synchronously and may exceed this timeout for complex inputs. If you receive a `504`, the operation is likely still running on the instance — check status or history endpoints to confirm completion.

## Testing authentication

Use the `/api/v1/vault/schema` endpoint to verify your credentials:

```bash
curl https://app.alfred.black/api/v1/vault/schema \
  -H "Authorization: Bearer alf_your_api_key_here"
```

A successful response returns the vault schema with all record types, confirming authentication is working.
