## Overview

Every request to the Alfred API requires a Bearer token in the `Authorization` header. The token is Sir's tenant API key (the `AAS_API_KEY`), provisioned automatically when Sir's instance is created. **Each token authorises one tenant — Sir's. The same token cannot be used to access any other subscriber's instance.**

## Getting Sir's API key

<Steps>
  <Step title="Through the dashboard">
    Sign in at [alfred.black](https://alfred.black), go to **Settings → API Keys**, and create a key. The full key is shown once — copy it then. Sir can name keys (`my laptop`, `mobile script`, `automation`) to know which to revoke later.
  </Step>
  <Step title="Through the API itself">
    If Sir already has an authenticated session, `GET /api/v1/admin/credentials` returns the configured credentials with values masked. The `AAS_API_KEY` itself is protected and not modifiable through this endpoint — Sir can only rotate it via the dashboard.
  </Step>
</Steps>

## Token format

The token is provisioned at instance creation and stored in Sir's tenant `.env` as `AAS_API_KEY`. It is a long opaque string. Treat it as a password.

```
AAS_API_KEY=<long opaque string>
```

The dashboard may display newly created keys in a prefixed format (e.g. `alf_...`) — both forms work, the comparison is timing-safe and exact.

## Making authenticated requests

Include the key as `Authorization: Bearer ${AAS_API_KEY}`:

<CodeGroup>
```bash cURL
curl https://alfred.black/api/v1/vault/context \
  -H "Authorization: Bearer ${AAS_API_KEY}"
```

```javascript JavaScript
const response = await fetch('https://alfred.black/api/v1/vault/context', {
  headers: { 'Authorization': `Bearer ${process.env.AAS_API_KEY}` }
});
const data = await response.json();
```

```python Python
import os, requests

response = requests.get(
    'https://alfred.black/api/v1/vault/context',
    headers={'Authorization': f"Bearer {os.environ['AAS_API_KEY']}"}
)
data = response.json()
```

```typescript TypeScript
import axios from 'axios';

const response = await axios.get('https://alfred.black/api/v1/vault/context', {
  headers: { Authorization: `Bearer ${process.env.AAS_API_KEY}` }
});
const data = response.data;
```
</CodeGroup>

## How auth is checked

The SaaS edge at `alfred.black` looks up the key against Sir's `Instance` record, decrypts the per-tenant `apiKey` (AES-256-GCM at rest), and proxies the request over Sir's Tailscale mesh into the tenant's `ctrl-api` on port `:3100`. The tenant's auth check is a **timing-safe constant-time comparison** against the server-side `AAS_API_KEY` — wrong keys reveal no information about the right one.

## Per-tenant scoping

Each token is mapped at the SaaS layer to exactly one tenant `Instance`. There is no way to use Sir's token to reach another subscriber's data — the proxy chain only routes to the tenant the token is bound to.

Cross-tenant access exists only on **Alfred Prime** instances (a separate capability, gated by environment configuration on the peer side and explicit peer listing on the Prime side). It does not use the same bearer flow.

## Best practices

- **Environment variables only.** Never hardcode the key. Never commit it.
  ```bash
  export AAS_API_KEY="<key>"
  curl https://alfred.black/api/v1/vault/context \
    -H "Authorization: Bearer $AAS_API_KEY"
  ```
- **One key per use.** Name keys after their purpose (`laptop`, `n8n`, `home-assistant`) so Sir knows which one to revoke.
- **Rotate periodically.** Create a new key, switch over, revoke the old one. The dashboard supports this without downtime.
- **Revoke unused keys.** Cleaning up shrinks the blast radius if anything is exposed.

## Error responses

A failed auth check returns 401:

<ResponseExample>
```json Missing or malformed header
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Missing or malformed Authorization header"
  }
}
```

```json Invalid API key
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid API key"
  }
}
```
</ResponseExample>

The most common cause of `Invalid API key` is whitespace or quotes accidentally included around the token value when it was set in the environment. Strip both.

## Token rotation

To rotate without downtime:

<Steps>
  <Step title="Create a new key in the dashboard">
    **Settings → API Keys → Create**. Copy the value.
  </Step>
  <Step title="Update Sir's clients">
    Switch every script and integration to the new key.
  </Step>
  <Step title="Verify with the health endpoint">
    `curl https://alfred.black/api/v1/admin/health -H "Authorization: Bearer <new-key>"` should return 200.
  </Step>
  <Step title="Revoke the old key">
    Back in the dashboard, revoke the previous key. Any client still using it will start receiving 401.
  </Step>
</Steps>

## Testing authentication

The health endpoint is the lightest way to verify a token:

```bash
curl https://alfred.black/api/v1/admin/health \
  -H "Authorization: Bearer ${AAS_API_KEY}"
```

A 200 confirms both that the key is valid and that Sir's tenant is reachable.
