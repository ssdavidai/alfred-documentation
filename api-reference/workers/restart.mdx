---
title: "Restart Workers"
description: "Restart the alfred Docker container."
api: "POST /api/v1/workers/restart"
---

Issues `docker compose restart alfred` against the tenant's compose stack. The container is recreated, every daemon is started fresh, and any new code or config baked into the image takes effect.

This is heavier than [`down`](/api-reference/workers/down) + [`up`](/api-reference/workers/up): expect roughly 10–30 seconds of downtime while the container restarts.

## Authentication

Bearer token in the `Authorization` header.

<RequestExample>

```bash cURL
curl -X POST https://alfred.black/api/v1/workers/restart \
  -H "Authorization: Bearer alf_your_key_here" \
  -H "Content-Type: application/json"
```

</RequestExample>

<ResponseExample>

```json 200
{
  "message": "Alfred container restarted"
}
```

</ResponseExample>

<ResponseField name="message" type="string">
  Always `"Alfred container restarted"` on success. The endpoint waits for the compose command to exit before returning.
</ResponseField>

## When to use

- After pulling a new `alfred-worker` image.
- After editing files mounted from `/mnt/encrypted/openclaw` that the worker reads at boot only.
- To recover from a daemon that won't respond to `down` cleanly.

<Warning>
  Workflows that restart the `alfred` container from inside an activity (for example chore-template reloads) frequently appear to fail with a Temporal heartbeat timeout because the activity kills its own runtime. The restart still succeeds — verify with [`GET /api/v1/workers/status`](/api-reference/workers/status) and ignore the heartbeat error.
</Warning>
