title: "Upload to Inbox (Bulk)"
description: "Upload several files to the vault inbox in a single request. Same per-file rules as the single upload endpoint."
api: "POST /api/v1/vault/inbox/bulk"
---

## When to use

Send a batch of inbox uploads in one HTTP request. Behaves like `POST /api/v1/vault/inbox` per file: each filename is sanitised, base64 binaries are decoded, and media files emit individual `stream_type: "media"` events.

The whole request fails fast on the first invalid entry â€” there is no partial-success mode. If file `[2]` is missing `content`, files `[0]` and `[1]` are NOT rolled back (they were already written to disk by the time the validation error fires for `[2]`). Plan accordingly.

## Body Parameters

<ParamField body="files" type="array" required>
  Array of file entries. Empty array is allowed (returns `count: 0`).

  <Expandable title="file entry">
    <ParamField body="filename" type="string" required>
      Name to store the file under in the inbox. Same sanitisation rules as the single endpoint: `/`, `\`, `:`, `*`, `?`, `"`, `<`, `>`, `|`, `..` are replaced with `_`.
    </ParamField>
    <ParamField body="content" type="string" required>
      File contents. Plain text by default, base64 if `encoding: "base64"` is set.
    </ParamField>
    <ParamField body="encoding" type="string" default="utf-8">
      Set to `"base64"` for binary content.
    </ParamField>
    <ParamField body="mime_type" type="string">
      Optional MIME type hint. Used for media-category detection ahead of filename extension.
    </ParamField>
  </Expandable>
</ParamField>

## Response

<ResponseField name="message" type="string" required>
  Human-readable confirmation (e.g., `"Uploaded 3 files to inbox"`).
</ResponseField>

<ResponseField name="filenames" type="string[]" required>
  Sanitised filenames written to disk, in input order.
</ResponseField>

<ResponseField name="files" type="array" required>
  Per-file detail in input order.

  <Expandable title="file result">
    <ResponseField name="filename" type="string">
      Sanitised filename.
    </ResponseField>
    <ResponseField name="binary" type="boolean">
      `true` when the entry was base64.
    </ResponseField>
    <ResponseField name="media_event_id" type="string">
      Present only for media files â€” the id of the emitted media stream event.
    </ResponseField>
    <ResponseField name="media_type" type="string">
      Present only for media files â€” one of `audio`, `document`, `image`, `video`.
    </ResponseField>
  </Expandable>
</ResponseField>

## Request Example

<RequestExample>

```bash cURL (mixed text + media)
curl -X POST https://alfred.black/api/v1/vault/inbox/bulk \
  -H "Authorization: Bearer $AAS_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "files": [
      {
        "filename": "notes-1.md",
        "content": "# Note 1\n\nIdea about Q2 onboarding.\n"
      },
      {
        "filename": "notes-2.md",
        "content": "# Note 2\n\nFollow-up with Zsolt.\n"
      },
      {
        "filename": "screenshot.png",
        "content": "iVBORw0KGgoAAAANSUhEUgAA...",
        "encoding": "base64",
        "mime_type": "image/png"
      }
    ]
  }'
```

```javascript JavaScript
const files = await Promise.all(fileList.map(async f => ({
  filename: f.name,
  content: btoa(String.fromCharCode(...new Uint8Array(await f.arrayBuffer()))),
  encoding: 'base64',
  mime_type: f.type,
})));

const response = await fetch('https://alfred.black/api/v1/vault/inbox/bulk', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${process.env.AAS_API_KEY}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({ files }),
});
const { files: results } = await response.json();
const mediaEventIds = results.flatMap(r => r.media_event_id ?? []);
```

</RequestExample>

## Tips

<Warning>
Validation is per-file, but failures are NOT atomic. If file `[2]` is rejected, files `[0]` and `[1]` are already on disk. Either pre-validate client-side, or accept that partial uploads may remain in the inbox after a 400.
</Warning>

<Note>
The server reads every entry's `content` whole into memory before writing. For very large batches (hundreds of MB combined), prefer multiple `POST /api/v1/vault/inbox` calls over one mega-bulk request.
</Note>

<Note>
Each media file in the batch emits its own stream event â€” no coalescing. If you need batch-level progress, track the `media_event_id` values from the response.
</Note>
