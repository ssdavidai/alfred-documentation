---
title: "List Records by Type"
description: "Returns every record of a single type with full frontmatter and a body preview."
api: "GET /api/v1/vault/list/{type}"
---

## When to use

When you want a richer, type-scoped view than `GET /api/v1/vault/context` gives you. This endpoint includes the **complete frontmatter object** and a **truncated body preview** for each record, so list views can render rich cards (titles, statuses, descriptions, tags) without making a second request per row.

Records are sorted alphabetically by `name` (or `subject`, for `conversation` and `input` types).

## Path Parameters

<ParamField path="type" type="string" required>
  One of the types returned by `GET /api/v1/vault/schema` (`person`, `org`, `project`, `task`, `event`, `note`, `location`, `process`, `account`, `asset`, `conversation`, `input`, `run`, `session`, `decision`, `triage`, `assumption`, `constraint`, `contradiction`, `synthesis`, `observation`, `instinct`, `reflection`, `matter`, `ledger_entry`, `chore`). Anything else returns `400`.
</ParamField>

## Query Parameters

<ParamField query="preview" type="integer" default="500">
  Maximum length of the `body_preview` field, in characters. Bodies longer than this are truncated and an ellipsis (`…`) is appended. Capped at `2000`. Pass `0` to get an empty preview.
</ParamField>

## Response

<ResponseField name="results" type="array" required>
  Records of the requested type, sorted by display name.

  <Expandable title="record">
    <ResponseField name="path" type="string">
      Vault-relative path INCLUDING the `.md` extension (e.g., `matter/alfred-product-development-launch.md`).
    </ResponseField>
    <ResponseField name="name" type="string">
      The `name` frontmatter field, falling back to `subject`, then to the file stem if neither is set.
    </ResponseField>
    <ResponseField name="status" type="string">
      The `status` frontmatter field, or empty string if absent.
    </ResponseField>
    <ResponseField name="frontmatter" type="object">
      The full parsed YAML frontmatter. Every key the record declares, including arbitrary user-defined fields. Date and datetime values are serialized as ISO-8601 strings.
    </ResponseField>
    <ResponseField name="body_preview" type="string">
      First `preview` characters of the markdown body (everything after the closing `---`). Truncated with `…` when over the limit.
    </ResponseField>
    <ResponseField name="created" type="string">
      The `created` frontmatter field as a string. May be an ISO date (`2026-04-08`) or full datetime depending on how the record was authored.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="count" type="integer" required>
  Number of records returned. Equal to `results.length`.
</ResponseField>

## Request Example

<RequestExample>

```bash cURL
# Default 500-char body previews
curl -X GET "https://alfred.black/api/v1/vault/list/matter" \
  -H "Authorization: Bearer $AAS_API_KEY"

# Larger 2000-char previews for a richer list view
curl -X GET "https://alfred.black/api/v1/vault/list/matter?preview=2000" \
  -H "Authorization: Bearer $AAS_API_KEY"

# No previews (faster, smaller payload)
curl -X GET "https://alfred.black/api/v1/vault/list/task?preview=0" \
  -H "Authorization: Bearer $AAS_API_KEY"
```

```javascript JavaScript
const response = await fetch(
  'https://alfred.black/api/v1/vault/list/matter?preview=1000',
  { headers: { 'Authorization': `Bearer ${process.env.AAS_API_KEY}` } },
);
const { results, count } = await response.json();
console.log(`${count} matters`);
results.forEach(r => console.log(`- ${r.name} [${r.status}]`));
```

</RequestExample>

## Tips

<Note>
For the full body, follow up with `GET /api/v1/vault/records/{path}` using the `path` from the listing.
</Note>

<Warning>
Like `GET /api/v1/vault/context`, this endpoint walks every `.md` file in the vault on each call (it filters by type after parsing frontmatter). For very large vaults consider caching by type and refreshing on a sensible interval.
</Warning>
