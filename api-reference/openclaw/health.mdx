---
title: "Gateway health"
description: "Quick liveness probe for the OpenClaw gateway."
api: "GET /api/v1/openclaw/health"
---

Reports liveness for the OpenClaw gateway running inside your Alfred instance. OpenClaw is the AI gateway every agent talks through — Alfred himself, the vault workers, and any background subagents. If this endpoint isn't healthy, nothing else that needs an LLM will be either.

## How it works

ctrl-API shells into the OpenClaw container and runs `openclaw gateway health`. The output is parsed as JSON and returned. If the CLI emits non-JSON, the raw text is wrapped in a `raw` field.

## Response

The exact shape comes from the OpenClaw CLI. Expect a small object describing process status, configured providers, and basic checks. When the CLI is mid-restart and produces text, the response degrades to `{ "raw": "..." }`.

<RequestExample>

```bash cURL
curl -X GET https://alfred.black/api/v1/openclaw/health \
  -H "Authorization: Bearer alf_your_key_here"
```

</RequestExample>

<ResponseExample>

```json Structured
{
  "ok": true,
  "uptime_seconds": 3284,
  "providers_loaded": ["openrouter", "anthropic", "google", "xai"]
}
```

```json Raw fallback
{
  "raw": "Gateway is healthy"
}
```

</ResponseExample>

<Note>
  For a fast, dashboard-style readiness check that is aware of in-flight gateway restarts (e.g. after Sir connects a new Composio app), prefer `GET /api/v1/openclaw/ready` — it returns within ~1.5s and tells you when to expect the gateway back.
</Note>
