---
title: "Approve a device"
description: "Admit a pending pairing request and issue the device its token."
api: "POST /api/v1/devices/{requestId}/approve"
---

Approves a pending pairing request. After approval, the device receives its credentials and can begin talking to your Alfred. Use this only for requests you actually initiated — every approval grants access to the same vault Sir relies on.

## Path parameters

<ParamField path="requestId" type="string" required>
  The `request_id` from `GET /api/v1/devices` (under `pending`).
</ParamField>

## Body parameters

<ParamField body="latest" type="boolean" default="false">
  When `true`, approves the most recent pending request for the device's identity rather than the specific request ID. Useful when the user re-initiated pairing after a stale request and the request ID drifted.
</ParamField>

## Response

<ResponseField name="message" type="string">
  Confirmation string from the OpenClaw CLI (e.g. `"Device approved"`).
</ResponseField>

<RequestExample>

```bash cURL
curl -X POST https://alfred.black/api/v1/devices/req-91a3/approve \
  -H "Authorization: Bearer alf_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{}'
```

```bash With --latest
curl -X POST https://alfred.black/api/v1/devices/req-91a3/approve \
  -H "Authorization: Bearer alf_your_key_here" \
  -H "Content-Type: application/json" \
  -d '{"latest": true}'
```

</RequestExample>

<ResponseExample>

```json 200 OK
{
  "message": "Device approved"
}
```

</ResponseExample>

<Warning>
  Approval is irrevocable in the sense that the device receives a real token. If you later change your mind, use [`POST /api/v1/devices/{deviceId}/revoke`](/api-reference/devices/revoke) to invalidate the role.
</Warning>
