## Overview

Every operation Alfred performs is exposed as an HTTP endpoint on Sir's `ctrl-api`. The dashboard, the channels, the workflows, and the agent itself all use the same surface. This is the same API Sir's own scripts and integrations should use.

<Note>
  Base URL: `https://alfred.black/api/v1/...`. Authenticate every request with `Authorization: Bearer ${AAS_API_KEY}`.
</Note>

## The proxy chain

Sir's requests do not hit the tenant directly. The path is:

```
Sir's client
   │
   ▼
https://alfred.black            (SaaS edge — auth, rate limit, billing checks)
   │  authenticates request, decrypts tenant key
   ▼
Tailscale mesh                  (private WireGuard, no public route)
   │
   ▼
Cloudflare Tunnel               (no inbound ports on Sir's tenant)
   │
   ▼
ctrl-api :3100 on Sir's VPS    (the actual handler)
```

The SaaS layer is responsible for authentication and billing. The tenant `ctrl-api` is responsible for the real work — vault writes, worker control, workflow management, channel dispatch. Sir never opens a public port on the tenant.

## Versioning

The current version is **v1**. All endpoints are prefixed with `/api/v1/`. Breaking changes will move under `/api/v2/`; v1 will not silently change.

## Authentication

Every request requires a Bearer token. The token is Sir's tenant API key (the `AAS_API_KEY` issued at provisioning). Sir gets it from the dashboard under **Settings → API Keys** or via `GET /api/v1/admin/credentials`.

```bash
curl https://alfred.black/api/v1/vault/context \
  -H "Authorization: Bearer ${AAS_API_KEY}"
```

A 401 is returned for missing, malformed, or wrong tokens. See [Authentication](/api-reference/authentication) for the full flow.

<Note>
Each API key authorises **one tenant** — Sir's. The SaaS layer maps the bearer token to Sir's instance and proxies only into that instance. There is no cross-tenant access through this surface (cross-tenant tools exist on Alfred Prime but go through a separate, tenant-to-tenant route).
</Note>

## Request format

All requests sending data must use `Content-Type: application/json`:

```bash
curl -X POST https://alfred.black/api/v1/vault/inbox \
  -H "Authorization: Bearer ${AAS_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"filename": "meeting-note.md", "content": "..."}'
```

## Response format

Responses are JSON. Successful responses have an endpoint-specific shape. Errors follow:

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message",
    "details": { "additional": "context" }
  }
}
```

## Error codes

| Code | HTTP | Meaning |
|------|------|---------|
| `UNAUTHORIZED` | 401 | Missing, malformed, or invalid bearer token |
| `VALIDATION_ERROR` | 400 | Request body failed validation |
| `NOT_FOUND` | 404 | The resource does not exist |
| `CONFLICT` | 409 | Operation conflicts with current state |
| `EXEC_ERROR` | 500 | Operation execution failed |
| `INTERNAL_ERROR` | 500 | Unexpected server error |

## Rate limits

There are no fixed per-second rate limits today. Sir's instance is dedicated, so resource pressure is a function of Sir's own usage. The `tenant-proxy` enforces a 15-second timeout per call from the SaaS edge into the tenant — long-running operations should be triggered as workflows (`POST /api/v1/workflows/start`) rather than blocking on a single HTTP call.
