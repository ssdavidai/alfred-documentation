## Overview

The Alfred API provides programmatic access to all Alfred functionality. The REST API is secured by Bearer token authentication using `alf_` API keys created in the dashboard.

<Note>
  The base URL for all API requests is `https://app.alfred.black`. All endpoints are prefixed with `/api/v1/`.
</Note>

## Request format

All requests that send data must use `application/json` as the `Content-Type`.

```bash
curl https://app.alfred.black/api/v1/vault/context \
  -H "Authorization: Bearer alf_your_api_key_here" \
  -H "Content-Type: application/json"
```

## Authentication

<Warning>
  Every request (except OPTIONS for CORS preflight) requires a Bearer token in the `Authorization` header. See the [Authentication](/api-reference/authentication) page for details.
</Warning>

## Response format

All responses return JSON. Successful responses have a structure specific to each endpoint. Error responses follow a consistent format:

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message",
    "details": {
      "additional": "context"
    }
  }
}
```

## Error codes

| Code | HTTP Status | Description |
|------|-------------|-------------|
| `UNAUTHORIZED` | 401 | Missing, malformed, or invalid API key |
| `VALIDATION_ERROR` | 400 | Request body failed validation (missing required fields, invalid types) |
| `NOT_FOUND` | 404 | Requested resource does not exist |
| `CONFLICT` | 409 | Operation conflicts with current state (e.g., record already exists) |
| `EXEC_ERROR` | 500 | Command execution failed (stderr included in details) |
| `INTERNAL_ERROR` | 500 | Unexpected server error |

## CORS

All origins are allowed (`Access-Control-Allow-Origin: *`). The API responds to OPTIONS requests for CORS preflight without requiring authentication.

## Rate limits

There are no rate limits. The API is designed for single-user or small-team use.

## Versioning

All endpoints are prefixed with `/api/v1/`. Breaking changes will be introduced under a new version prefix.

## SDK & tooling

No official SDK is provided. The API follows REST conventions and can be consumed from any HTTP client.
