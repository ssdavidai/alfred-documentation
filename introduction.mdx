## Sir's life, attended to.

Alfred Black is a private butler service. Not an app Sir installs. Not a dashboard Sir is expected to maintain. A managed service — provisioned, configured, and tended to on Sir's behalf.

Sir shares the raw substance of life — emails, meetings, calls, notes, documents, ambient recordings — and Alfred's specialists turn it into a structured, interconnected vault. Every person, project, decision, commitment, and pattern, recorded once and linked everywhere it belongs.

Alfred attends to Sir on every channel Sir already uses — the dashboard, email, SMS, voice calls, Slack, Telegram. Sir does not learn Alfred's interface. Alfred meets Sir wherever Sir already is.

You don't use Alfred. Alfred serves you.

---

## A small staff of specialists

| Specialist | Role |
|------------|------|
| **The Curator** | Receives anything Sir hands in — meeting notes, documents, screenshots — and files it as connected vault records |
| **The Janitor** | Walks the vault on a schedule, mending broken links and tidying inconsistencies before Sir notices |
| **The Distiller** | Reads between the lines of existing records and surfaces the assumptions, decisions, constraints, and contradictions hiding there |
| **The Surveyor** | Embeds records, clusters them by meaning, and quietly maps how Sir's world fits together |
| **The Clerk** | A stateless analyst dispatched on demand for classification, extraction, and short-form reasoning |
| **Alfred himself** | The agent Sir actually speaks to — orchestrates the staff, answers questions, executes errands |

Behind them all, Alfred's **intuition** observes how Sir routes things. Over time, observations harden into **instincts** — learned patterns that let Alfred handle routine matters on Sir's behalf, asking only when judgment is genuinely required.

## The vault as Sir's life-OS

Every meeting, every promise, every preference is recorded as a Markdown file in Sir's vault and connected via wikilinks to everything else it touches. The vault is fully Obsidian-compatible — Sir can pair a device and browse it directly. There are 26 record types across four families: standing entities (people, organisations, projects), activity records (events, tasks, conversations), learning records (assumptions, decisions, contradictions), and intuition records (observations, instincts).

Hand Alfred a single standup note and it produces records for each participant, the organisation they represent, the decisions made, the action items with owners, the project being discussed — all linked to the conversation itself. Drop another note in tomorrow and the same people get richer, the same projects accumulate history.

The dashboard's homepage is the **Vault Nebula** — a living visualisation of those clusters as drifting clouds, giving Sir an at-a-glance map of everything Alfred is tracking.

## Connected Apps — 1,000+ integrations

Alfred connects to over a thousand applications via Composio. Each connection gives Alfred two things: **data in** (Sir's emails, calendar events, Notion pages, GitHub notifications stream into the vault automatically) and **actions out** (Alfred can send the email, create the calendar event, update the issue, post to Slack — on Sir's behalf, with Sir's credentials, never leaving Sir's namespace).

Common integrations:

- **Gmail** — 100-day backfill on connect, then incremental sync of only new mail
- **Google Calendar** — true delta sync via syncToken; only changes are fetched
- **Notion, GitHub, Slack, Linear, Stripe, Todoist, Asana** — and hundreds more
- **Omi** — ambient recordings transcribed via Groq Whisper
- **Custom webhooks** — any HTTPS source can push into a stream

Sir authorises once. Composio holds the credentials in its vault; nothing sensitive sits on Sir's tenant. Alfred picks up new mail every few minutes and folds it into the vault using zero-LLM Python templates — fast, predictable, free. An hourly enrichment pass batches recent records into a single LLM call to add entities and tags.

## Channels — wherever Sir already is

Alfred is reachable on every channel Sir uses for the rest of life:

- **Dashboard chat** at [alfred.black](https://alfred.black)
- **Email** at `alfred.<sir>@mail.alfred.black` — Sir writes; Alfred replies; PDFs and attachments work both ways
- **SMS** at Sir's dedicated phone number — full conversational replies, threaded context
- **Voice calls** at the same number — Alfred answers, primed with Sir's open matters and last week's summaries; transcript captured back into the vault on hangup
- **Slack** as a bot in Sir's workspace — DMs and mentions
- **Telegram** as a bot in Sir's chat

Sir can move freely between them. Something said over voice this morning informs the email response this afternoon — every channel writes to the same vault, every channel is primed from the same memory.

## Progressive autonomy

Alfred is not a chat window pretending to be useful. The Kinetic layer powers an active, growing surface:

- **Matters** — ongoing concerns that group related work (the kitchen renovation, the Q2 launch, the dispute with the landlord)
- **Errands** — discrete tasks Alfred either executes itself or surfaces for Sir
- **Chores** — recurring scheduled workflows generated by Opus and personalised to Sir (the weekly cashflow forecast, the morning brief, the quarterly contact warm-up). Each chore is a real Python workflow, validated and smoke-tested before going live
- **Instincts** — learned routing patterns from Sir's instructions ("from now on, treat receipts from Bolt as expense items"). Instincts execute via short-lived, scope-limited agents
- **Triage** — items Alfred could not confidently classify, surfaced for Sir's review

Trust grows in stages. New instincts require Sir's approval on every run. After ten observations of the same pattern with Sir's consent, the threshold relaxes. After fifty, Alfred acts autonomously and reports after the fact.

## Three planes, one tenant

Underneath the service voice, Alfred is a three-plane system:

- **SaaS plane** — `alfred.black` handles authentication, billing, the dashboard, and proxies API calls to Sir's tenant
- **Control plane** — `ctrl-api` runs on Sir's tenant at port `:3100`, exposing every operation Alfred performs as an HTTP endpoint
- **Data plane** — Sir's dedicated Hetzner VPS runs the full Docker stack: vault workers, OpenClaw gateway, Temporal workflow engine, the intelligence layer

Every subscriber gets their own server. Data is encrypted at rest with LUKS2. Services bind to localhost only — external access exclusively through Sir's private Tailscale mesh. No data ever shares a machine with another subscriber's data.
